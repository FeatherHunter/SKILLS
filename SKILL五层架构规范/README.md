|---|---|---|

## 🛠️ 实战 SOP:改一个 Skill 时的标准顺序

1. **先读指导手册**:确认这件事符合 6 大特性(可约束/可恢复/可约束/...)
2. **画清领域边界** → ⑤ 文档层先写 SKILL.md 第一段("管什么、不管什么")
3. **列触发词表** → ⑤ 文档层定触发词
4. **设计 schema** → ① 数据层先建表 + 索引 + .bak 策略
5. **写硬规则** → ③ 规则层先写 `validators.py`(无跳过通道)
6. **写 CLI** → ④ 接口层最后接(argparse + `{status, data, message}` JSON)
7. **写集成** → ② 操作层(对外部分)看需要再加(一切出站调用:外部服务/其他 skill/第三方 API,失败降级;≥2 个集成点抽 `integrations/` 子模块)
8. **跑自检清单**:用 `Skill_架构自检清单.md` 勾一遍 5 层 + 改动前 3 问

## 📏 规模判断

| Skill 代码量 | 架构选择 |
|---|---|
| ≤ 200 行 | **微型 3 层合并**(文档/接口/操作+数据) |
| 2-5 个 .py | **中型完整 5 层**(推荐,适合大多数 Skill) |
| 5+ .py 或多人协作 | **大型 5 层 + 对外操作独立子模块**(独立子模块) |

## ⚠️ 改动前必答 3 问(出自自检清单)

1. **影响哪些文件?** (代码/文档/数据库 schema/HTML/changelog)
2. **有没有数据迁移?** (改 schema → 写 migration 脚本,不直接 ALTER 跑生产)
3. **回滚方案?** (.bak 文件在哪、怎么还原)

答不上来 = 改动还没准备好。
