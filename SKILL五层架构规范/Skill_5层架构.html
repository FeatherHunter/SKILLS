一 · 为什么需要"层"的概念

手册讲了一个 Skill 该具备什么"品质"（可识别 / 可验证 / 可恢复 / 可约束 / 可联动 / 可演进），但没说这些品质该放在哪个文件里。

结果就是：

  • 硬规则散落在 CLI 文件里（居家管家的 item_ops.py L119-130 算例外）
  • 业务逻辑和契约解析混在同一个 .py（饼干记账的 record_bill.py）
  • SQL 查询藏进业务流程（卡路里早期版本）
  • 外部服务调用阻塞主流程（如果飞书挂了全卡死）

"层"这个概念的作用只有一个：让你写代码时知道往哪放。不分层的 Skill 短期能跑，分层的 Skill 长期能活。

二 · 5 层架构图

┌─────────────────────────────────────────────────────┐ │ ① 文档层 SKILL.md · HTML 镜像 · references/ │ │ ← N:1 同一 Skill 的不同切面 │ ├─────────────────────────────────────────────────────┤ │ ② 契约层 CLI: argparse · JSON 输出 · 错误码 │ │ ← 唯一对外入口 │ ├─────────────────────────────────────────────────────┤ │ ③ 业务层 领域模块 + 硬约束 │ │ ← 业务流程 + 心法 + 校验规则 │ ├─────────────────────────────────────────────────────┤ │ ④ 数据层 db.py + SQLite（schema + 索引 + 备份） │ │ ← 持久化 + 事务 + 迁移 │ ├─────────────────────────────────────────────────────┤ │ ⑤ 集成层 飞书 / 训记 / MCP / Webhook │ │ ← 对外接口（失败要降级，不能阻塞主流程） │ └─────────────────────────────────────────────────────┘ ↑ ↑ ↑ │ │ │ AI / 用户 业务流程 外部世界

关键约束

  • 箭头只能向下：上层可以 import 下层，下层绝不 import 上层
  • 下层不知道上层存在：db.py 不该知道有 CLI
  • 集成层是"软依赖"：挂了不能拖垮业务（失败降级）

三 · 每层的职责 + 不该做的事

① 文档层

职责不该做的事
SKILL.md 写触发词 + 操作流程 + 示例对话 把业务逻辑写成伪代码（应该在契约层）
HTML 镜像给人类读 HTML 里只放截图，不写文字（搜不到）
references/ 放深入文档（分类心法、术语表） references 写成 2000 行长文没人读
changelog 记录每次变更 改完代码忘了改 changelog

② 契约层（CLI）

职责不该做的事
argparse 定义参数 schema argparse 里直接调 sqlite3.connect（绕过业务层）
统一输出 {status, data, message} 输出格式每次不一样（print、json.dumps、str 混用）
兼容层（db_utils.py） 改 CLI 签名不通知下游
帮助文本 + 错误码 错误信息只有"失败"两字，不说怎么修

③ 业务层

职责不该做的事
领域模块：diet.py / water.py / weight.py 每个功能都写在 CLI 文件里
业务规则（软）：分类心法、推断逻辑 软规则写成硬规则，把 AI 锁死
硬约束：validators.py 集中校验 硬规则散落 N 个文件，难维护
编排：跨模块调用 + 事务包裹 业务模块直接 import 别的业务模块的私有函数

④ 数据层

职责不该做的事
db.py 封装 SQL + 事务 每个 .py 自己 connect 自己 cursor
schema 设计 + 索引 + 约束 外键不加索引（查询慢）、CHECK 约束缺失（脏数据）
migration 脚本（schema 升级用） 直接 ALTER TABLE 跑生产（无回滚点）
备份策略（.bak.YYYYMMDD_HHMMSS） 改 schema 前不备份

⑤ 集成层

职责不该做的事
外部服务调用：飞书 task / 训记 / MCP 把外部调用写进业务模块（应该独立 feishu_sync.py / xunji_bridge.py）
失败降级 + 重试 外部服务挂了，业务流程直接抛异常挂掉
幂等键（client_request_id） 重试时产生重复数据
反向对账（外部改了 → 本地跟上） 只单向写，不读

四 · 自检问题清单

每层 3-4 题。把它贴到 SKILL.md 末尾，改 Skill 前先答一遍。

① 文档层

  • 这层改了，SKILL.md / HTML / changelog 都同步了吗？
  • SKILL.md 第一段能在 30 秒内回答"管什么、不管什么"吗？
  • 所有触发词（动词+名词）都列进表了？AI 找不到的触发词会不会漏？
  • HTML 镜像路径规划好了吗？（同目录 Skill.html，不要散在各处）

② 契约层

  • argparse schema 文档化了？（每个参数含义 + 默认值 + 取值范围）
  • 所有 CLI 输出统一 {status, data, message} 三段式？
  • 错误信息包含字段名 + 当前值 + 期望值 + 怎么修？
  • 改 CLI 签名时，旧参数走兼容层（db_utils.py 模式）？

③ 业务层

  • 领域模块切得合理吗？（不是 1 个 2000 行的 .py，也不是 20 个 50 行的）
  • 软规则用心法表达，不靠关键词查表？
  • 硬规则集中在 validators.py，且无跳过通道？
  • 跨模块编排有独立层（或明确写在哪个文件）？

④ 数据层

  • 所有 SQL 走 db.py，没有散落的 connect/cursor？
  • 事务包裹了写入路径？（BEGIN → 操作 → COMMIT）
  • schema 改之前 .bak 一份？
  • 外键 + CHECK 约束下了吗？（把硬规则下沉到 DB 层）

⑤ 集成层

  • 外部服务调用有 try/except + 失败降级吗？
  • 重试带幂等键（client_request_id）？
  • 外部系统改了，本地有反向对账机制？
  • API 限频处理了？（训记 45s 间隔那种）

五 · 横向流动示意

以"记一笔山姆订单 ¥49.90"为例，看请求怎么穿过 5 层：

[用户说 "山姆买了午餐肉 ¥49.90"] │ ▼ ① 文档层: AI 读 SKILL.md 找到「记支出」触发词 → 饼干记账 skill │ ▼ ② 契约层: CLI 解析 → category=餐饮/食材, amount=-49.90, note=... │ ▼ ③ 业务层: ├─ 软规则: AI 心法推断 category（"食材" vs "日用品"） ├─ 硬规则: validators.py 校验 amount 必填为负数 └─ 编排: 调 record_bill.add() │ ▼ ④ 数据层: ├─ db.py: BEGIN TRANSACTION ├─ INSERT INTO bills (..., created_at) └─ COMMIT │ ▼ ⑤ 集成层: 路由检查 └─ 询问: "录了居家管家吗？"（可选，用户答是 → 居家管家 skill）

每一步都有错误恢复 + 审计。失败时事务回滚，外部服务失败不阻塞写入。

六 · 不同规模 Skill 的简化策略

不是所有 Skill 都需要完整 5 层。按规模伸缩，避免过度工程。

规模判定标准架构方案你的例子
微型 ≤1 个 .py，≤200 行 3 层合并：文档 + 契约 + 业务+数据 写一起 饼干记账（单 record_bill.py）
中型 2-5 个 .py 5 层完整：每个模块职责清晰 居家管家（db.py + item_ops.py + ...）
大型 5+ .py，含外部集成 5 层 + 集成层：每个子模块独立文件 卡路里 / 备忘录（业务模块切得最细）

判断标准：如果你一个人 1 周内能写完，用微型；如果你一个人 1 个月内写完，用中型；如果多人协作或长期维护，用大型。

给你的 3 个具体动作

① 立即 · 5 分钟

打开 Skill_架构自检清单.md，填一遍你目前 4 个 Skill 的状态。看到 ❌ 别慌，看到 ⚠️ 排优先级。

② 这周 · 半天

挑一个 Skill 做"饼干记账"→"中型"的升级：拆 validators.py + db.py，把硬规则集中。改完跑一遍回归。

③ 下次新建 · 从 0 开始

不要从 CLI 开始写！顺序：

  1. 画清领域边界（管什么、不管什么）→ ① 文档层先写 SKILL.md 第一段
  2. 列触发词表 → ① 文档层定触发词
  3. 设计 schema → ④ 数据层先建表
  4. 写硬规则 → ③ 业务层先 validators.py
  5. 写 CLI → ② 契约层最后接
  6. 写集成 → ⑤ 集成层看需要再加
