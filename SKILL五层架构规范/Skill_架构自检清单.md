# Skill 架构自检清单

> 配套 [《优秀 Skill 指导手册》](./优秀Skill指导手册.html) 第二部分"5 层结构"使用。
> 建议：每个 Skill 的 SKILL.md 末尾贴一份，**改 Skill 前先答一遍**。

---

## ① 文档层

- [ ] 这层改了，SKILL.md / HTML / changelog 都同步了吗？
- [ ] SKILL.md 第一段能在 30 秒内回答"管什么、不管什么"吗？
- [ ] 所有触发词（动词+名词）都列进表了？AI 找不到的触发词会不会漏？
- [ ] HTML 镜像路径规划好了吗？（同目录 `Skill.html`，不要散在各处）
- [ ] references/ 拆得合理吗？（不是 2000 行长文，也不是颗粒度过细）

---

## ② 契约层（CLI）

- [ ] argparse schema 文档化了？（每个参数含义 + 默认值 + 取值范围）
- [ ] 所有 CLI 输出统一 `{status, data, message}` 三段式？
- [ ] 错误信息包含字段名 + 当前值 + 期望值 + 怎么修？
- [ ] 改 CLI 签名时，旧参数走兼容层（`db_utils.py` 模式）？

---

## ③ 业务层

- [ ] 领域模块切得合理吗？（不是 1 个 2000 行的 .py，也不是 20 个 50 行的）
- [ ] 软规则用心法表达，不靠关键词查表？
- [ ] 硬规则集中在 `validators.py`，且无跳过通道？
- [ ] 跨模块编排有独立层（或明确写在哪个文件）？

---

## ④ 数据层

- [ ] 所有 SQL 走 `db.py`，没有散落的 `connect/cursor`？
- [ ] 事务包裹了写入路径？（BEGIN → 操作 → COMMIT）
- [ ] schema 改之前 .bak 一份？
- [ ] 外键 + CHECK 约束下了吗？（把硬规则下沉到 DB 层）
- [ ] 索引建在常用查询字段上了？

---

## ⑤ 集成层

- [ ] 外部服务调用有 try/except + 失败降级吗？
- [ ] 重试带幂等键（`client_request_id`）？
- [ ] 外部系统改了，本地有反向对账机制？
- [ ] API 限频处理了？（训记 45s 间隔那种）

---

## 规模自检（决定要不要拆层）

- [ ] Skill 总代码量 ≤ 200 行？→ 可以用**微型 3 层合并**（文档/契约/业务+数据）
- [ ] 2-5 个 .py？→ 用**中型完整 5 层**
- [ ] 5+ .py 或多人协作？→ **大型 5 层 + 集成层**

---

## 改动前必答 3 问

1. **影响哪些文件？**（代码 / 文档 / 数据库 schema / HTML / changelog）
2. **有没有数据迁移？**（改 schema → migration 脚本）
3. **回滚方案？**（.bak 文件在哪、怎么还原）

答不上来 = 改动还没准备好。
