第一部分 · 第一性原理：Skill 是什么

1.1 Agent 与 Skill 的关系

先定义两个东西：

  • Agent = 自主决策的演员，根据上下文调用工具完成任务
  • Skill = 给 Agent 用的领域工具包，包含数据、操作、规则、接口、文档

如果说 Agent 是厨师，那 Skill 就是厨房里的整套工具 + 食材 + 食谱。没有 Skill 的 Agent 是空有厨艺没有工具的厨师，什么都做不出来。

1.2 为什么 Skill 比"直接写脚本"更强？

你当然可以让 Agent 直接 python some_script.py 干完事，但 Skill 化之后多出 5 件事：

维度直接写脚本Skill 化
可发现AI 不知道它存在触发词表 + SKILL.md 让 AI 自动识别
可验证失败无人感知统一 JSON 输出 + ID 回执
可恢复半成品状态留在数据库事务 + 备份 + 反向对账
可约束AI 偷懒随便写硬规则在 CLI 层强制校验
可演进改一行全系统挂版本 + schema 迁移 + 文档同步

1.3 Skill 的本质 = "领域"

一个 Skill 不是"一个 Python 文件"，它是一整个领域。比如"居家管家"这个 Skill，定义的就是「家庭物品」这个领域——

  • 这个领域里有哪些实体（物品、分类、位置、标签）
  • 实体之间有什么关系（物品 → 分类 → 位置 → 标签）
  • 领域里允许什么操作（增删改查 + 盘点 + 推荐）
  • 操作要遵守什么规则（tag ≥ 10、备注必填）
  • 操作的输入输出长什么样（CLI 参数 + JSON 返回）
  • 怎么教别人用（SKILL.md + HTML 镜像 + 示例）

所以 Skill 不是"工具"，Skill 是"领域 + 工具 + 文档 + 规则"。这层认知错了，后面所有的设计都会跑偏。

第二部分 · Skill 的 5 层结构

第一部分讲了 Skill 该具备什么"品质"（可识别/可验证/可恢复/可约束/可联动/可演进）。但没说这些品质该放在哪个文件里。这一部分讲"骨架"——5 层结构 + 规模伸缩。

不分层的 Skill 短期能跑，分层的 Skill 长期能活。"层"这个概念的作用只有一个：让你写代码时知道往哪放。

2.1 五层是什么（按领域视角）

5 层从"领域由什么组成"的角度切：

① 数据层
state
管什么数据。数据库表结构、索引、约束、外键、事务、迁移、备份。例：居家管家的 items / item_locations / item_tags / categories 表
② 操作层
actions
能做什么动作。本地操作：原子化 CLI 子命令（add / update / delete / search / list / summary）。对外操作（集成）：调外部服务（飞书 / 训记 / MCP / 其他 skill）。例：record_bill.py add --amount -49.90 --category 餐饮 是本地操作；feishu_sync.py push-task 是对外操作
③ 规则层
rules
必须遵守什么。硬规则（系统强制，如 tag ≥ 10）+ 软规则（AI 自觉，如分类心法）。设计位置：硬规则集中在 validators.py，软规则放在 SKILL.md 的 references/ 章节
④ 接口层
contract
怎么说话。输入 schema（argparse 参数）+ 输出 schema（JSON）+ 错误码 + 退出码。例：所有 CLI 返回 {"status": "ok"|"error", "data": ..., "message": "..."}
⑤ 文档层
docs
怎么被理解。SKILL.md（Agent 读）+ HTML 镜像（人读）+ 示例对话 + 术语表 + references/ 子文档。例：居家管家.html 和 SKILL.md 内容必须保持一致

好的 Skill，5 层互相自洽；差的 Skill，往往某一层是空的或糊弄的。

2.2 关键约束（层间硬规则）

  • 箭头只能向下：上层可以 import 下层，下层绝不 import 上层。例：操作层可以 import 数据层（通过 db.py），但 db.py 不该知道有 CLI
  • 下层不知道上层存在：db.py 不该知道有 CLI；validators.py 不该知道被哪个 skill 调用
  • 操作层（对外部分）是"软依赖"：外部服务挂了，本地照常工作，只 stderr 记录。例：飞书同步失败 → 本地 due 仍生效
  • 对外操作 ≥ 2 个集成点 → 抽 integrations/ 子模块统一管理（避免代码散落业务模块）

2.3 每层落到什么文件

5 层是抽象概念，落到目录里是这样的：

层对应文件 / 目录职责
① 数据层 db.py + SQLite SQL 封装、事务、迁移、备份
② 操作层（本地） *_ops.py（item_ops / inventory_ops / ...） 领域模块、编排、跨模块事务
② 操作层（对外/集成） integrations/feishu_sync.py / xunji_bridge.py 外部服务调用、失败降级、幂等键
③ 规则层 validators.py（硬）+ references/（软） 硬规则系统校验、软规则 AI 自觉
④ 接口层 CLI 入口文件（如 home_manager.py） argparse 定义、JSON 输出、错误码、兼容层
⑤ 文档层 SKILL.md + <skill>.html + features/ + references/ 触发词、操作流程、HTML 镜像、深入文档

2.4 规模伸缩

不是所有 Skill 都需要完整 5 层。按规模伸缩，避免过度工程。

规模判定标准架构方案例子
微型 ≤1 个 .py，≤200 行 3 层合并：文档 + 接口 + 操作+数据 写一起 饼干记账（单 record_bill.py）
中型 2-5 个 .py 5 层完整：每层职责清晰 居家管家（db.py + item_ops.py + ...）
大型 5+ .py，含外部集成 5 层 + 对外操作独立子模块：每个子模块独立文件 卡路里 / 备忘录（业务模块切得最细）

判断标准：如果你一个人 1 周内能写完，用微型；如果你一个人 1 个月内写完，用中型；如果多人协作或长期维护，用大型。

第三部分 · 优秀 Skill 的 6 大特性

按重要性排序，每条配"为什么这条要放在这里"的实战理由。

① 可识别 Discoverable ⭐⭐⭐⭐

AI 不用提示就知道用哪个 Skill、传什么参数、什么时候停。

具体设计模式

  • 触发词表（动词+名词结构）：记吃了 / 查今天吃 / 记喝水 / 查今天喝水 / 查营养配比。一个动作 + 一个对象，不含糊。
  • 领域边界声明：SKILL.md 第一段必须能回答"这个 skill 管什么、不管什么"。
  • 自然语言变体覆盖：「花了 35」「付了」「买了」「消费」全部映射到 记支出。AI 解析时同时匹配。
  • 反例区分：「查热量」（搜食品库）vs「查热量趋势」（分析模块）—— 易混淆的词要在文档里区分清楚。
  • 歧义消解规则：记运动 vs 查运动记录 → "记"=新增，"查"=查询。让 AI 有明确二选一标准。

反面教材

❌ 触发词全是抽象动词

用"管理 / 处理 / 操作"这种抽象词当触发词 —— AI 完全没法判断你要干嘛。

② 可验证 Verifiable ⭐⭐⭐⭐⭐

写入前能查、写入后能审、重复调用不出问题。

具体设计模式

  • 幂等性（idempotency）：重复请求不产生重复数据。
    实现方式：先 search → 0 命中才 add（不是 add 命令自己支持）。
    例：备忘录加心愿「八段锦 20分钟」前先 search 八段锦 -c 心愿，命中 0 才 add。
  • 写入前回执：每个 write 操作前 AI 应该能 query 当前状态。例：add 之前先 list --category-id 157 看看有没有同名。
  • 写入后回执：返回 ID + 影响行数 + 时间戳。例：add 返回 {id: 10320, created_at: "2026-07-15 10:06:44"}。
  • 稳定主键：用自增整数 ID 当主键，不用自然语言当 key。例：item_id=810 而不是 name="Tulip..."（名字会改、ID 不会）。
  • 审计字段：每张表必须有 created_at + updated_at，可查询"谁什么时候改了什么"。
  • 查询命令齐全：每个 add/update/delete 都对应一个 query / list / search，没有"写完查不到"的情况。

幂等性深度解释

「幂等」 = 喊一次到位，重发 N 次结果一样。通俗替代词：「重发不重数」「最多只算 1 次」。
为什么需要？网络会抖、AI agent 会重复调、定时任务会重跑 —— 没幂等必出 bug。

③ 可恢复 Recoverable ⭐⭐⭐⭐

错误能 rollback、不一致能对账、误操作能补救。

具体设计模式

  • 原子操作：要么全成要么全败（事务包裹）。例：备忘录的 complete-wish = 删旧心愿 + 新建打卡，用 BEGIN TRANSACTION 包起来。
  • 失败降级：依赖的外部服务挂了，本地照常工作。例：飞书同步失败 → 本地 due 仍生效，只 stderr 记录。
  • 反向同步（双向对账）：本地和外部服务互相同步。例：备忘录 sync-from-feishu 3 步流程：本地补建 → 反向同步 done → 反向同步 due。
  • 软删除 > 硬删除：标记 is_deprecated=1 比 DELETE 好。居家管家思想：物品只增不删，"已废弃"是状态不是删除。
  • 备份机制：改 schema 前 .bak 一份。例：home.db.bak.20260703_104428 这种命名让恢复点清晰。
  • 幂等键：跨系统写操作带 client_request_id，避免重试产生重复。例：训记同步用 {date}_{session_label}_{uuid8} 当幂等键。

④ 可约束 Constrainable ⭐⭐⭐⭐⭐

硬规则系统强制、软规则 AI 自觉；两者各司其职。

硬规则 vs 软规则的分界

硬规则（Hard）软规则（Soft）
执行者CLI / DB 层AI（理解 + 自律）
违反后果直接报错拒收AI 自己改 / 询问用户
典型例子tag ≥ 10、备注非空、金额必填分类心法（"主体 / 目的 / 对象" 维度）
设计位置item_ops.py 等核心逻辑文件SKILL.md 的 references/ 章节
绕过难度无跳过通道AI 可以违反（但不应该）

硬规则的设计原则

  • 早失败优于晚失败：在校验处直接拒，不要等数据库报错。
  • 失败信息要具体："tag 数量 3 (要求 ≥ 10)" 优于 "tag 不够"。
  • 无跳过通道：不要留 --force / --skip-validation 这种 flag —— AI 偷懒会立刻用上。
  • 错误要可恢复：报错信息要告诉 AI 怎么修。例：返回"当前 tag: A, B, C，建议补充：X, Y, Z"

软规则的设计原则

  • 心法而非清单：给思考框架，不给"看到 X 就归到 Y"这种关键词表。例：饼干的分类心法 5 维度（主体/目的/对象/工具vs体验/谁付费），优于关键词查表。
  • 边界示例：举"灰色地带"案例让 AI 学到边界判断。例：理发店的洗发水 → 居家/日用品；家用的洗发水 → 居家/日用品；但如果是"做发型用的"→ 穿着/洗护？

⑤ 可联动 Composable ⭐⭐⭐

Skill 与外部系统（其他 skill / 飞书 / 训记 / MCP）通过操作层（对外部分）对话，失败不阻塞主流程。

5.1 本地操作 vs 对外操作

"操作层"包含两种：

  • 本地操作：对本地数据库的增删改查（add / update / delete / search / list / summary）
  • 对外操作（集成）：调外部服务（飞书 task / 训记训练 / MCP / 其他 skill）

两者职责不同，对外操作 ≥ 2 个集成点 → 抽 integrations/ 子模块，避免代码散落在本地操作模块里。

5.2 具体设计模式

  • 统一 JSON 输出：{status, data, message} 三段式。所有 CLI 一致。
  • 稳定字段名：返回 id, content, created_at, category_id, ... 不要今天叫 note_id 明天叫 noteId。
  • 显式外键：返回 feishu_task_guid / category_id 让别的 Skill 能引用。
  • 联动路由：做完主操作后，AI 自动检查要不要联动。例：记完一笔"山姆买沐浴露" → 询问"要不要录居家管家？"
  • Skill 间不直连：Skill A 不 import Skill B 的内部代码。通过字段值 / 外部 ID 通信。
  • 路由器集中：联动规则放在 图片路由/SKILL.md 等中央路由，不在每个 Skill 里重复维护。
  • 失败降级（对外操作铁律）：外部服务挂了 → 本地照常工作，只 stderr 记录。例：飞书同步失败 → 本地 due 仍生效。
  • 幂等键（对外写操作必带）：client_request_id 避免重试产生重复。例：训记同步用 {date}_{session_label}_{uuid8} 当幂等键。

⑥ 可演进 Evolve-able ⭐⭐⭐

改一行代码不会让整套系统挂掉；改一处文档不会让另一处过期。

具体设计模式

  • HTML 镜像：SKILL.md 改了，HTML 必须同步。这是备忘录 / 居家管家 / 饼干记账都写进 SKILL.md 第一条的硬规定。
  • 版本号 / changelog：CLI 改动要带版本或 changelog。
  • schema 迁移：新增字段/表要 migration 脚本（不要直接 ALTER 跑生产）。
  • 向后兼容：老 API 还能用。例：卡路里的 db_utils.py 给旧脚本转发到 db.py。
  • 兼容层：内部重构时保留旧 import 路径一段时间，避免雪崩。

第四部分 · 反模式与陷阱

知道什么是好之后，更要清楚什么是坑。下面每条反模式都来自真实事故。

❌ 黑盒 CLI

症状：报错信息只有"内部转码失败"但不说哪个字段错了。
后果：AI 无法 debug，用户干瞪眼。
正解：错误信息要带字段名 + 当前值 + 期望值 + 怎么修。

❌ 隐式状态

症状：缓存里改了但数据库没改；session 里有数据但重启就没了。
后果：数据脏了找不到原因，幽灵 bug。
正解：所有状态都写到 SQLite 或文件，不在内存里藏东西。

❌ 魔法字符串

症状：硬编码 category == "心愿" 而不是从枚举读。
后果：某天分类改名，全系统挂。
正解：分类、状态、类型都用枚举常量，定义在共享文件里。

❌ 静默失败

症状：except: pass 把异常吞了。
后果：用户以为成了其实没成，数据不一致。
正解：所有异常至少 stderr 打印 + 写日志；CLI 返回非 0 退出码。

❌ 紧耦合

症状：Skill A 直接 import Skill B 的私有函数。
后果：改 B，A 莫名挂；升级 B，A 也要改。
正解：Skill 之间通过 JSON 字段通信，不互相 import。

❌ 慢反馈

症状：批处理 5 分钟没回应，AI 不知道是死是活。
后果：AI 重复调，幂等性又没做，结果翻倍。
正解：长操作每 5-10 秒输出进度，至少告诉 AI "我还在干"。

❌ 中文 CLI 参数乱码（PowerShell 5.1 专属）

症状：从 PowerShell 直接调 python script.py "中文"，数据库里存的是乱码 ????????。
根因：PowerShell 5.1 把中文按系统代码页（GBK）转字节，Python 收到后用错编码解码。
正解：用 Python subprocess.run([...], list of args) 列表传参，不要从 PowerShell 直接传中文。

第五部分 · Self-Audit 自检清单

三个时点：新建 Skill、修改 Skill、日常维护。每条都要打勾才能发布。

5.1 新建 Skill 前

  • 领域边界清晰（不和其他 Skill 撞）
  • 数据 schema 画好 ER 图（含外键 / 索引 / 约束）
  • 触发词列表（动词+名词结构，覆盖自然语言变体）
  • 区分了硬规则和软规则（哪些必须系统校验，哪些靠 AI 自觉）
  • 输出格式定为 {status, data, message} 三段式 JSON
  • SKILL.md 第一段能回答"管什么、不管什么"
  • HTML 镜像路径规划好

5.2 修改 Skill 时

  • 改动前 .bak 一份数据库
  • 如果改 schema：写 migration 脚本，不直接 ALTER
  • 如果改 CLI 参数：保留旧参数做兼容层
  • 如果改硬规则：先评估对存量数据的影响
  • 改完代码 → 改 SKILL.md → 改 HTML → 改 changelog（顺序不能乱）
  • 跑回归：之前积累的测试 case 全过一遍

5.3 日常维护

  • 定期 查异常（数据健康检查）：标签完整性、表利用率、状态时效性、位置规范性
  • 定期跑 Lint：标签 < 2 个的、单级位置的、"快递中"超过 N 天的
  • 跨系统对账：飞书 task ↔ 本地 notes 不一致项排查
  • 用户报告 bug 后：先复现 → 找根因 → 修复 → 加测试 → 改文档

第六部分 · 实战案例：从今天踩的坑

本手册不是纸上谈兵，所有原则都来自 2026-07-15 这一天的实战。以下 4 个案例对应不同特性的真实教训。

案例 1：山姆订单录入（5 笔 + 4 物品）— 验证「可验证」

场景：一张山姆订单截图，4 SKU 共 6 件 ¥323.30。

流程：

  1. AI 识别金额 → 发现三文鱼那行 ¥49.90 可能是单价或小计
  2. 用户发第二张截图 → 确认 ¥49.90 是单价，3 条 = ¥149.70
  3. AI 算总和：49.90 + 42.80 + 149.70 + 79.90 = 322.30 ✓ 对上
  4. 拆 5 笔记账 + 4 件物品，每笔带订单号 tag 订单26071508562 便于追溯

体现的特性：可识别（触发词明确）、可验证（订单号贯通两套库）、可联动（记账 + 居家管家 + 备注 tag 互通）。

案例 2：居家管家 add 硬规则 — 验证「可约束」

场景：录 Tulip 午餐肉罐头，AI 只传了 3 个 tag + 空备注。

报错：

案例 4：PowerShell 中文 CLI 乱码 — 验证「可验证」的边界

场景：录备忘 ID 10324 时，PowerShell 直接传中文 → 数据库存乱码。

根因：PowerShell 5.1 把中文按 GBK 转字节 → Python 收到后用错编码 → sqlite3 存错。

修复：

# 错的写法
python memo_cli.py add "中文内容" -c 备忘
# ↓ 数据库里是乱码

# 对的写法（Python subprocess 列表传参，绕过 shell 编码）
import subprocess
subprocess.run(['python', 'memo_cli.py', 'add', '中文内容', '-c', '备忘'])

教训：即使 Skill 设计得再好，运行环境的字符编码也会偷走你的数据。验证机制要包含「写入后立刻回读检查」。

第七部分 · 术语对照表（专业词 ↔ 大白话）

专业词通俗替代词生活类比
幂等性
(idempotency)		 喊一次到位 / 重发不重数 锁门这件事 —— 不管锁 1 次还是锁 5 次，最终都是「锁好」状态
原子操作
(atomic operation)		 要么全成要么全败 / 包起来的事 转账：A 扣 100 和 B 加 100 必须同时完成，不然就回滚
事务
(transaction)		 包起来的事 网购下单：扣款 + 减库存 + 创建订单，要么全成要么都不动
软删除
(soft delete)		 假装删除 / 加个「已废弃」标签 图书馆下架旧书 —— 不烧掉，贴个「停借」标签还能查到
硬规则 / 软规则
(hard/soft rule)		 系统强制 / AI 自觉 硬规则 = 考场带监考老师；软规则 = 家里自习靠自觉
changelog 改动日记 装修记录 —— 每改一面墙就写一笔，以后回忆时一目了然
回执
(receipt / ack)		 办事后的凭条 银行 ATM 取钱后吐出来的小票 —— 有金额、ID、时间
外键
(foreign key)		 引用别人的钩子 朋友圈 @好友 —— 你的动态指向另一个用户
schema 迁移 表格升级 老房子改造 —— 加个房间、改个朝向，但不能拆承重墙
路由
(routing)		 分诊台 医院挂号 —— 「发烧咳嗽」分到呼吸科，「胃痛」分到消化科
联动
(cross-skill flow)		 跨部门协作 点外卖后顺丰自动推送物流 —— 一个动作触发多个系统
幂等键
(idempotency key)		 唯一小票号 网购订单号 —— 重复点击支付不会扣两次
