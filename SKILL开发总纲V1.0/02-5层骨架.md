# 02 · 5 层骨架

> Skill 设计的**唯一权威**。新写 / 改任何 Skill,先读这章。

## 5 层结构(按领域视角)

| 层 | 职责 | 落点 |
|---|---|---|
| ① 数据层 `state` | 表 / 索引 / 约束 / 事务 / 迁移 / 备份 | `db.py` |
| ② 操作层 `actions` | 本地:原子化 CLI 子命令;对外:外部服务调用 | `*_ops.py` + `integrations/` |
| ③ 规则层 `rules` | 硬规则(系统强制)+ 软规则(AI 自觉) | `validators.py` + `references/` |
| ④ 接口层 `contract` | argparse + `{status, data, message}` JSON | CLI 入口 |
| ⑤ 文档层 `docs` | SKILL.md + HTML + references/ | `.md` + `.html` |

### 层间硬规则

- **箭头只能向下**:上层 import 下层,下层绝不 import 上层
- **下层不知道上层存在**:`db.py` 不该知道有 CLI;`validators.py` 不该知道被哪个 skill 调用
- **对外操作是"软依赖"**:外部挂了,本地照常工作,只 stderr 记录
- **对外操作 ≥ 2 个集成点 → 抽 `integrations/` 子模块**

## 6 大特性

按重要性排序,每条配"为什么这条要放在这里"的实战理由。

### ① 可识别 Discoverable ⭐⭐⭐⭐

**AI 不用提示就知道用哪个 Skill、传什么参数、什么时候停。**

| 设计模式 | 说明 |
|---|---|
| 触发词表(4 元组) | 动作 + 对象 + 维度(可选) + 类型(可选) |
| 领域边界声明 | SKILL.md 第一段能 30 秒内回答"管什么、不管什么" |
| 自然语言变体 | 每个核心触发词配 2-3 个变体 |
| 跨 Skill 路由声明 | 重叠时显式声明"路由到 X" |

→ 详细设计见 [03-触发词设计v2.md](./03-触发词设计v2.md)

### ② 可验证 Verifiable ⭐⭐⭐⭐⭐

**写入前能查、写入后能审、重复调用不出问题。**

| 设计模式 | 说明 |
|---|---|
| 幂等性(流程层面) | 先 search → 0 命中才 add(不是 add 命令自己支持) |
| 写入前回执 | 每个 write 前能 query 当前状态 |
| 写入后回执 | 返回 ID + 时间戳 + 影响行数 |
| 稳定主键 | 用自增整数 ID,不用自然语言 |
| 审计字段 | 每张表必须有 `created_at` + `updated_at` |
| 查询命令齐全 | 每个 add/update/delete 都对应一个 query/list/search |

**幂等性深度解释**:重发 N 次结果一样。**通俗替代词**:喊一次到位 / 重发不重数。

### ③ 可恢复 Recoverable ⭐⭐⭐⭐

**错误能 rollback、不一致能对账、误操作能补救。**

| 设计模式 | 说明 |
|---|---|
| 原子操作 | BEGIN → 操作 → COMMIT(要么全成要么全败) |
| 失败降级 | 外部依赖挂了,本地照常工作,只 stderr 记录 |
| 反向同步 | 本地和外部互相同步(双向对账) |
| 软删除 > 硬删除 | 标记 `is_deprecated=1`,不 DELETE |
| 备份机制 | 改 schema 前 `.bak` + git commit 额外备份 |
| 幂等键 | 跨系统写操作带 `client_request_id` |

### ④ 可约束 Constrainable ⭐⭐⭐⭐⭐

**硬规则系统强制、软规则 AI 自觉;两者各司其职。**

| | 硬规则 Hard | 软规则 Soft |
|---|---|---|
| 执行者 | CLI / DB 层 | AI(理解 + 自律) |
| 违反后果 | 直接报错拒收 | AI 自己改 / 询问用户 |
| 典型例子 | tag ≥ 10、备注非空、金额必填 | 分类心法(主体 / 目的 / 对象) |
| 设计位置 | `validators.py` 等核心逻辑文件 | SKILL.md 的 references/ 章节 |
| 绕过难度 | 无跳过通道 | AI 可以违反(但不应该) |

**硬规则设计原则**:
- **早失败优于晚失败**:在校验处直接拒
- **失败信息要具体**:含字段名 + 当前值 + 期望值 + 怎么修
- **无跳过通道**:不留 `--force` / `--skip-validation`
- **错误要可恢复**:报错信息告诉 AI 怎么修

**软规则设计原则**:
- **心法而非清单**:给思考框架,不给关键词表
- **边界示例**:举"灰色地带"案例让 AI 学到边界判断

### ⑤ 可联动 Composable ⭐⭐⭐

**Skill 与外部系统(其他 skill / 飞书 / 训记 / MCP)通过操作层(对外部分)对话,失败不阻塞主流程。**

| 设计模式 | 说明 |
|---|---|
| 统一 JSON 输出 | `{status, data, message}` 三段式 |
| 稳定字段名 | `id, content, created_at`(不要今天 noteId 明天 note_id) |
| 显式外键 | `feishu_task_guid` / `category_id` 让别的 Skill 引用 |
| 联动路由 | 做完主操作 → AI 自动检查要不要联动 |
| Skill 间不直连 | 通过 JSON 字段通信,不互相 import |
| 路由器集中 | 联动规则放在中央路由(如图片路由/SKILL.md) |
| 失败降级 | 外部挂了 → 本地照常工作 |
| 幂等键(对外写操作必带) | 避免重试产生重复 |

### ⑥ 可演进 Evolve-able ⭐⭐⭐

**改一行代码不会让整套系统挂掉;改一处文档不会让另一处过期。**

| 设计模式 | 说明 |
|---|---|
| **HTML 镜像** | SKILL.md 改 → HTML 必同步(**硬规则**,见 [04 §4.2](./04-可视化与注入v2.md)) |
| 版本号 / changelog | CLI 改动要带版本或 changelog |
| schema 迁移 | 新增字段 / 表要 migration 脚本(不直接 ALTER 跑生产) |
| 向后兼容 | 老 API 还能用(老 import 路径走兼容层) |

## 8 反模式(不要做)

| # | 反模式 | 正解 |
|---|---|---|
| 1 | **黑盒 CLI** — 报错不说哪个字段 | 错误信息含字段名 + 当前值 + 期望值 + 怎么修 |
| 2 | **隐式状态** — 缓存 / 内存藏数据 | 所有状态写 SQLite 或文件 |
| 3 | **魔法字符串** — 硬编码枚举 | 用共享枚举常量 |
| 4 | **静默失败** — `except: pass` | stderr + 非 0 退出码 |
| 5 | **紧耦合** — Skill A import B 私有函数 | 通过 JSON 字段通信 |
| 6 | **慢反馈** — 批处理 5 分钟无响应 | 每 5-10 秒输出进度 |
| 7 | **中文 CLI 乱码**(PowerShell 5.1) | Python `subprocess.run([...], list)` 列表传参 |
| 8 | **CLI 不返回结构化 JSON** | 加 `as_dict=True` 参数,向后兼容默认 print 行为 |

## 规模自检

| 规模 | 标准 | 方案 |
|---|---|---|
| 微型 | ≤ 1 .py,≤ 200 行 | 3 层合并(文档 / 接口 / 操作+数据) |
| 中型 | 2-5 个 .py | **5 层完整(推荐)** |
| 大型 | 5+ .py 或多人协作 | 5 层 + 对外操作独立子模块 |

## 5 层自检清单

### ① 数据层

- [ ] 所有 SQL 走 `db.py`,没有散落的 `connect/cursor`?
- [ ] 事务包裹了写入路径?(BEGIN → 操作 → COMMIT)
- [ ] schema 改之前 .bak 一份?
- [ ] 外键 + CHECK 约束下了?(把硬规则下沉到 DB 层)
- [ ] 索引建在常用查询字段上了?

### ② 操作层(本地)

- [ ] 领域模块切得合理?(不 1 个 2000 行,不 20 个 50 行)
- [ ] 软规则用心法表达,不靠关键词查表?
- [ ] 跨模块编排有独立层?

### ② 操作层(对外 / 集成)

- [ ] 外部服务调用有 try/except + 失败降级?
- [ ] 重试带幂等键(`client_request_id`)?
- [ ] 外部系统改了,本地有反向对账机制?
- [ ] API 限频处理了?
- [ ] ≥ 2 个集成点 → 抽 `integrations/` 子模块?

### ③ 规则层

- [ ] 硬规则集中在 `validators.py`,且无跳过通道?
- [ ] 软规则写在 references/ 章节?
- [ ] 错误信息含字段名 + 当前值 + 期望值 + 怎么修?
- [ ] 软规则用心法表达,不靠关键词查表?

### ④ 接口层(CLI)

- [ ] argparse schema 文档化了?(每个参数含义 + 默认值 + 取值范围)
- [ ] 所有 CLI 输出统一 `{status, data, message}` 三段式?
- [ ] 错误信息含字段名 + 当前值 + 期望值 + 怎么修?
- [ ] 改 CLI 签名时,旧参数走兼容层?

### ⑤ 文档层

- [ ] 这层改了,SKILL.md / HTML / changelog 都同步了吗?
- [ ] SKILL.md 第一段能在 30 秒内回答"管什么、不管什么"?
- [ ] 所有触发词(4 元组)都列进表了?AI 找不到的触发词会不会漏?
- [ ] HTML 镜像路径规划好了吗?(同目录 `<skill>.html`,不散在各处)
- [ ] references/ 拆得合理?(不 2000 行长文,也不颗粒度过细)

## 改动前必答 3 问

1. **影响哪些文件?**(代码 / 文档 / DB schema / HTML / changelog)
2. **有没有数据迁移?**(改 schema → 写 migration 脚本,不直接 ALTER 跑生产)
3. **回滚方案?**(`.bak` 文件在哪、怎么还原;如果当前目录是 git 仓库,是否已用 git commit 额外备份本次修改)

**答不上来 = 改动还没准备好。**
