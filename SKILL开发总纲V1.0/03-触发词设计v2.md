# 03 · 触发词设计 v2

> **v2 = 矩阵方案**。v1(老 prompt)= 单维"动作+对象",只服务 1 个 skill。v2 服务 50+ skill。

## v1 局限(为什么大刀阔斧改)

老 prompt(2026-05-28)的问题:

| # | 问题 | 后果 |
|---|---|---|
| 1 | **单维结构** — 只规定"动词+名词" | 无法表达复杂意图(维度 / 类型) |
| 2 | **单 skill 经验** — 从 1 个学习规划师推不出通用规则 | 50+ skill 没法套用 |
| 3 | **无量化** — 没给上下限 | AI 写 3 个或 50 个都没标准 |
| 4 | **无去重规则** — 多 skill 重叠时不知道让谁负责 | 路由混乱 |
| 5 | **无变体管理** — 每个核心词配多少变体没说 | AI 匹配率不稳定 |

## v2 矩阵方案

把"动词+名词"升维为 **4 元组**:

```
触发词 = 动作 + 对象 + 维度(可选) + 类型(可选)
```

### 4 元组说明

| 元素 | 必需 | 说明 | 例子 |
|---|---|---|---|
| **动作** | ✓ | 动词,原子化(≤ 2 字) | 记 / 查 / 改 / 删 / 算 / 导出 |
| **对象** | ✓ | 名词,领域实体 | 餐 / 水 / 运动 / 体重 |
| **维度** | ✗ | 限定范围 / 状态 | 今天 / 本周 / 趋势 / 排行 |
| **类型** | ✗ | 进一步细分 | 高热量 / 低热量 / 频繁 |

### 完整 vs 简写

```
完整 4 元: 查饮食-高热量排行   = 查(动作) 饮食(对象) -高热量(类型) 排行(维度)
简写 2 元: 记餐               = 记(动作) 餐(对象)
```

**规则**:2 元能表达就不写 4 元。**复杂才用 4 元**。

## 数量规则

- **下限**:**8 个**。8 是质量底线,覆盖核心 CRUD + 统计 + 导出
- **上限**:**不设上限**。复杂领域(医药 / 法律)可能 50+,治理优先于限制

> 老 prompt 没给硬区间,AI 写 3 个就交差;或 50+ 词但去重混乱。**8 是质量底线,无上限**。

## 3 条去重铁律

### 铁律 1 · 精度优先

```
skill_a_trigger = "查餐"                  ← 范围广
skill_b_trigger = "查饮食-高热量排行"     ← 精度高

# 同一输入 "查高热量"
→ 优先路由到 skill_b(精度更高)
```

### 铁律 2 · 同义合并

```
# ❌ 不要有
"记餐" / "记录饮食" / "吃了啥" / "记吃了"

# ✅ 合并为 1 个 + 3 个变体
"记餐" + 变体("记录饮食" / "吃了啥" / "记吃了")
```

### 铁律 3 · 跨 Skill 路由声明

当 2 个 skill 都可能响应时,在 SKILL.md 显式声明"路由到 X"(示例):

```markdown
## 与其他 Skill 的边界

- "[触发词 X]" → 走 [Skill A](**当前 skill**)
- "[触发词 Y]" → 走 [Skill B]
```

## 变体管理

每个核心触发词配 **2-3 个自然语言变体**。变体写在 SKILL.md 的 references/ 章节,或 `references/wake_words.md` 子文件。

```markdown
## 触发词变体(references)

- 记餐 ← "记录饮食" / "吃了啥" / "吃了什么"
- 查今天吃 ← "查今日餐" / "今天吃了什么" / "今天饮食"
- 查营养配比 ← "查营养" / "查配比" / "今天营养够吗"
```

**为什么 2-3 个**:少于 2 → AI 匹配不鲁棒;多于 3 → 维护成本飙升且无收益(用户表达千变万化但收敛于 2-3 高频变体)。

## 变体到 CLI 参数的相对时间换算(铁律 4 · 2026-07-24 补)

**第一性**:用户说自然语言相对时间(今天/本周/上月等),AI 必须能精确换算为绝对日期才能调 CLI。**每个 Skill 自己实现**自己的换算 helper,不抽到 `_shared/`(各 Skill 时间语义可能不同 — 例:卡路里的"本周"按健身周期,作息的"本周"按周一~周日)。

### 必实现的 12 个相对表达式

| 类型 | 表达式 | 换算目标 |
|---|---|---|
| **单日** | 今天 / 昨天 / 前天 / 大前天 / 明天 / 后天 / 大后天 | 单个 `YYYY-MM-DD` |
| **周范围** | 本周(或"这周") / 上周 / 上上周 | `(start, end)` 元组(周一~周日) |
| **月范围** | 本月(或"这个月") / 上个月(或"上月") | `(start, end)` 元组(1 号~月末) |
| **窗口** | 最近 N 天 | `(start, end)` 元组(N ≥ 1) |

### 边界 case 必须测试覆盖

- **跨月**:今天=8/1,问"昨天"→ 7/31
- **跨年**:今天=2026/1/1,问"上月"→ 2025-12-01~2025-12-31
- **闰年**:今天=2024/2/15,问"本月"→ 2024-02-01~2024-02-29
- **12 月**:今天=2026/12/15,问"本月"→ 2026-12-01~2026-12-31
- **跨年 +1**:今天=2026/12/31,问"明天"→ 2027-01-01

### 设计原则

1. **纯函数,无副作用** — 不读 DB / 不写文件 / 不调外部 API
2. **today 参数支持注入** — 测试时传 `today=date(2026, 7, 24)` 固定时间,避免依赖 `date.today()`
3. **未知表达式抛 ValueError** — 不静默返回今天
4. **同义词必须等价** — "本月" == "这个月"、"上月" == "上个月"(有测试覆盖)
5. **跨 Skill 不共享代码,只共享原则** — 抽到 `_shared/` 是反模式(各 Skill 时间语义可能不同)

### 在 SKILL.md 必有的章节

```markdown
## 路由规则 · 用户表达 → 唤醒词 → CLI

[总路由表:用户表达示例 → 唤醒词 → CLI 命令 → 输出形式]

### 相对时间映射
[12 个表达式的换算规则 + 示例(今天=YYYY-MM-DD)]

### 组合场景(典型多步查询)
[典型用户场景 → 串行步骤]

### 歧义消解原则
[5 个常见模糊表达 → 反问策略]
```

### 参考实现(作息管家 · 2026-07-24)

- `scripts/routing.py` (120 行):`today_str` + `relative_to_date` + `relative_to_range` + `recent_n_days`
- `tests/test_routing.py` (29 用例):覆盖 12 表达式 + 跨月/跨年/闰年边界
- `SKILL.md` 新增"路由规则"章节

**为什么是原则不是代码**:每个 Skill 的"周"定义可能不同(中国周一 vs 美国周日),Skill 边界由自己决定。但**实现相对时间换算 + 测试覆盖**是通用要求。

## 矩阵设计练习(模板)

| | 动作=记 | 动作=查 | 动作=导出 |
|---|---|---|---|
| 对象=餐 | 记餐 | 查今天吃 | 导出饮食日志 |
| 对象=水 | 记喝水 | 查今天喝 | 导出饮水记录 |
| 对象=运动 | 记运动 | 查运动记录 | 导出训练日志 |
| 对象=体重 | 记体重 | 查体重趋势 | — |

→ 11 个核心触发词 + 2-3 变体 = ~30 个自然语言匹配点。足够覆盖日常对话。

## 与 5 层架构对接

触发词属于 **⑤ 文档层**的入口词。设计完后必须:

1. 写入 SKILL.md 第一段("管什么")的触发词表
2. 与 HTML 镜像同步(见 [04-可视化与注入v2.md](./04-可视化与注入v2.md))
3. 写 changelog 记录新增 / 删除 / 合并
4. 在 `references/wake_words.md`(如有)写变体
