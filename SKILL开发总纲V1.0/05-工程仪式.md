# 05 · 工程仪式

> 动手前的最后一道防线。这章是**自动检查清单**,不是原则解释。

## 改动前必答 3 问

1. **影响哪些文件?**(代码 / 文档 / DB schema / HTML / changelog)
2. **有没有数据迁移?**(改 schema → migration 脚本,不直接 ALTER)
3. **回滚方案?**(`.bak` 文件在哪、怎么还原;如果当前目录是 git 仓库,是否已用 git commit 额外备份本次修改)

**答不上来 = 改动还没准备好。**

## 工程实施 SOP

### 改一个 Skill 时的标准顺序

1. **先读指导手册** — 确认这件事符合 6 大特性
2. **画清领域边界** → ⑤ 文档层先写 SKILL.md 第一段("管什么、不管什么")
3. **列触发词表** → ⑤ 文档层定触发词(用 v2 矩阵,见 [03](./03-触发词设计v2.md))
4. **设计 schema** → ① 数据层先建表 + 索引 + .bak 策略
5. **写硬规则** → ③ 规则层先写 `validators.py`(无跳过通道)
6. **写 CLI** → ④ 接口层最后接(argparse + `{status, data, message}`)
7. **写集成** → ② 操作层(对外部分)按需要加
8. **跑自检清单** — 5 层 + 改动前 3 问

### 写一个 HTML 模板时的标准顺序

1. **判定是否值得做** → 原则 0(多字段 + 多条目 + 视觉密度 ≥ 5)
2. **检查 CLI** → 原则 1(必须支持 `as_dict=True`)
3. **选模板模式** → 决策矩阵(单卡 / 趋势 / 排行 / 状态 / 综合 / 对比)
4. **复制 [_assets/template_skeleton.html](./_assets/template_skeleton.html)** → 改具体字段
5. **引用 [_assets/style.css](./_assets/style.css)** → 调色板和字体自动统一
6. **写渲染器** → 4 步模式(subprocess + assert + inject + write)
7. **测试** → Chrome 打开 + 5 状态都触发
8. **同步** → SKILL.md + HTML + changelog 同 commit

## Commit 规范

| 规则 | 说明 |
|---|---|
| 每 Phase 一个 commit | 失败可独立 `git revert` |
| commit 前缀 `[技能名]` | `git log --grep` 精确搜 |
| 信息含:文件清单 + 行为变化 + 向后兼容 | 便于 reviewer |
| 不进 git:测试 HTML / 临时方案 / 备份目录 | `.gitignore` 排除 |

## Git Hooks(pre-commit)

`.githooks/pre-commit` 自动跑对应 skill 的 pytest:

```bash
case "$path" in
    居家管家/*) NEED_TEST[居家管家]=1 ;;
    卡路里/*)    NEED_TEST[卡路里]=1    ;;
esac
```

**新增 skill 时**:在 case 列表中加新条目。

## 测试模板(最小可行)

```
<skill>/tests/
├── conftest.py        # 添加 scripts/ 到 sys.path + 共享 fixture
├── test_validators.py # 硬规则单元测试(不依赖 DB)
├── test_render.py     # 模板生成测试(如有 HTML 模板)
└── test_payloads.py   # payload 函数结构测试(需 DB)
```

`pytest.ini`(最小):
```ini
[pytest]
testpaths = tests
python_files = test_*.py
addopts = -v --tb=short
```

## 跨 Agent 知识传播清单

**新 agent 第一次处理 SKILLS/ 时**:

1. 读 `SKILLS/SKILL开发总纲V1.0/SKILL.md`(主入口)
2. 读 `SKILL开发总纲V1.0/02-5层骨架.md`(核心权威)
3. 读目标 skill 的 `SKILL.md`(第一段定义管什么)
4. 跑 `cd <skill> && python -m pytest tests/ -v`(验证基线)
5. 检查目标 skill 的 `.html` 镜像是否与 SKILL.md 一致

## 已知边界(用户决策)

- ✗ **不做 web 端 CI**(GitHub Actions / GitLab CI)
- ✗ **不做 web merge 防护**(branch protection / required check)
- ✗ **不做 SDK 防护**(依赖 SKILL.md 硬规则 + AI 自约束)
- ✓ **本地 hook + 总纲硬规则**是当前完整防护

接受的风险:
- Web 端绕过(用户决策忽略)
- `--no-verify`(依赖 AI 自约束 + commit message HOTFIX 约定)
- Git 库直调(依赖 AI 自约束)

## 已知陷阱(自检)

- ❌ `cmd.exe /c start chrome "..."` — 引号陷阱,URL 会被 URL-encode 进乱码
- ❌ 把含中文的路径直接传给 shell — PowerShell 中文参数 → Python subprocess 列表传参
- ❌ `--no-sandbox` — 除非调试环境,否则有安全风险
- ✅ 用 `file:///` 协议而非路径直传
- ✅ 路径 URL-encode — 中文 / 空格必须编码

## 失败处理

- 测试失败 → 排查后修复,**不** `--no-verify` 跳过
- Hook 失败 → 检查 `core.hooksPath` 配置
- 数据不一致 → 先回滚到上次 commit,再排查根因
