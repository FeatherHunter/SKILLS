# 06 · 附录 · 演化规则(RULE Forms)

> 本附录是<strong>从历史案例中抽出的 RULE 形式</strong>,非硬编码案例。所有 skill 可复用。
> <b>不含具体 skill 名 / 具体 BUG 列表</b>(尊重"不喜欢硬编码案例"原则)。

## 附录 A · Skill 改造优先级判据(3 维度评估)

任何 skill 改造前先打分,再决定 S/A/B/不做。

| 维度 | 权重 | 评分标准 |
|---|---|---|
| **复用频率** | ⭐⭐⭐⭐⭐ | 用户每周 / 每天调用次数 |
| **视觉密度** | ⭐⭐⭐⭐ | 多字段 + 多条目 + 需要扫读 |
| **改造阻力** | ⭐⭐⭐ | CLI 现状是否容易支持 JSON 化 |

### 判定矩阵

| 维度组合 | 优先级 | 含义 |
|---|---|---|
| 高频 + 高密度 + 低阻力 | **S**(1-2 周动手) | 立即做 |
| 高频 + 低密度 | **A**(2-4 周动手) | 排第二批 |
| 高密度 + 低频 | **B**(按需) | 第三批 |
| 低频 + 低密度 + 高阻力 | **不做** | 投入产出比不够 |

→ 任何 skill 改造前**先做这 3 维度评估**,再开 Phase。**跳过这步直接动手** = 浪费时间。

## 附录 B · Skill 改造 4 Phase 模式

任何 skill 改造按"**先低阻力后高阻力**"顺序分 4 个 Phase。每个 Phase 一个 commit + `[技能名]` 前缀,失败可独立 `git revert`。

| Phase | 适合场景 | 动作 | 输出 |
|---|---|---|---|
| **A · 最小阻力先做** | CLI 已合规(返回结构化 JSON) | 补字段 → 写 1 个简单模板 → 渲染器 → commit | 1 commit,1 模板 |
| **B · 数据已合规** | 数据完整,但缺可视化层 | 选模板模式 → 复用 `_assets/template_skeleton.html` → 渲染器 → commit | 1 commit,1 模板 |
| **C · 规范化已有 HTML** | 有 HTML 但不符合规范 | 拆分为模板 + 注入模式 → 数据契约 `{status, data, message}` → commit | 1 commit,文件拆分 |
| **D · CLI 重构 + 解锁阻塞** | 多个 P0 阻塞于 CLI 不返回 JSON | 加 `as_dict=True` 参数(**保持默认 print 行为**)→ 多模板 → commit | 1 commit,N 模板 |

### 为什么按这个顺序

- A 阻力最小,先吃**便宜柿子**积累信心
- B 数据已有,只缺可视化,投入小收益大
- C 是"已有但不规范",改造而非新增
- D 阻力最大,放最后 — 因为它会**影响前面所有 Phase 的成果**

→ **违反顺序的代价**:D 提前做 → 之前写的模板全部失效 → 返工。

## 附录 C · 通用改造顺序(综合 §02 + §05)

改 Skill 时的标准步骤,每步都要"**前一步完成 + 单测通过**"才进下一步:

1. 画领域边界 → ⑤ 文档层先写 SKILL.md 第一段
2. 列触发词表 → 用 v2 矩阵(见 §03)
3. 设计 schema → ① 数据层 db.py + 索引 + .bak 策略
4. 写硬规则 → ③ 规则层 `validators.py`(无跳过通道)
5. 写 CLI → ④ 接口层 argparse + `{status, data, message}`
6. 写集成(如需要)→ ② 操作层(对外部分)`integrations/`
7. 跑自检 → 5 层自检 + 改动前 3 问(见 §02)

→ **步骤 1-2 合并可行**(都在文档层);**3-7 不能合并**(每步必须前一步稳)。

## 附录 D · 何时**不**该做 Skill 化(避免过度工程)

| 类型 | 决策 | 理由 |
|---|---|---|
| 单条 CRUD(add / update / delete) | ❌ | 不需要 Skill 框架 |
| 高频但无视觉(快速回执) | ❌ | print 即可 |
| 后台管理(需权限 / 事务) | ❌ | HTML 不适合 |
| 数据采集(数据即产物) | ❌ | HTML 没价值 |
| 一次性脚本 | ❌ | 不值得维护 |
| 2 步以内的查询 | ❌ | 直接命令行更快 |

→ **判据**(对应 §04 原则 0):唤醒词里 **≥ 5 个**含"多字段 + 多条目 + 视觉密度",才值得做。

## 附录 E · BUG 修复优先级判据

发现 BUG 时,按**严重度**修复,而不是按发现顺序:

| 严重度 | 含义 | 反应时间 |
|---|---|---|
| 🔴 **致命** | 主流程阻塞 / 数据丢失 / 安全 | 立即 |
| 🟡 **中等** | 功能部分损坏 / 体验降级 | 当前 Phase |
| 🟢 **低** | 边角 / 文案 / 美化 | 下个 Phase |

→ 🔴 不修不 commit;🟡 跟着主功能走;🟢 攒一批再修。
