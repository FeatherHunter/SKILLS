第一部分 · 第一性原理

Skill 是什么?Skill = 领域 + 工具 + 文档 + 规则。一个 Skill 定义一整个领域(如某家庭物品管理 Skill),包含该领域的所有:

  • 实体(物品 / 分类 / 位置 / 标签)
  • 操作(增删改查 / 统计 / 联动)
  • 规则(硬规则系统校验 + 软规则 AI 自觉)
  • 接口(CLI / JSON / 错误码)
  • 文档(SKILL.md / HTML 镜像 / references)

没有 Skill 的 Agent = 空有厨艺没有工具的厨师。

为什么分层?5 层让写代码时知道往哪放。不分层的 Skill 短期能跑,分层的 Skill 长期能活。

→ 详见 01-第一性原理.md

第二部分 · 5 层骨架

Skill 设计的唯一权威。新写 / 改任何 Skill,先读这章。

第三部分 · 触发词设计 v2

v2 重写 v1(老 prompt)= 单维"动作+对象",只服务 1 个 skill。v2 服务 50+ skill。

核心:4 元组结构

触发词 = 动作 + 对象 + 维度(可选) + 类型(可选)

例: 查饮食-高热量排行
   = 查(动作) 饮食(对象) -高热量(类型) 排行(维度)

3 个硬规则

  • 数量:8 ≤ N 无上限(覆盖核心 CRUD + 统计 + 导出)
  • 变体:每个核心触发词配 2-3 个自然语言变体,写在 references/
  • 去重:3 条铁律 — 精度优先 / 同义合并 / 跨 Skill 路由声明

→ 详见 03-触发词设计v2.md

第四部分 · 可视化与注入 v2

v2 资产化 v1 = 原则 + 案例,AI 每次重新设计 → 反复返工。v2 = 可加载资产 + 决策树。

10 原则(速览)

#原则要点
0何时不做单条 CRUD / 后台管理 / 数据采集 — 不做
1CLI 阻塞CLI 不支持 as_dict=True → 模板做不出来
21 模板多参数N 个数据 schema 相同的唤醒词 → 1 模板 + N 参数
3模板四段式首屏 / 主体 / 交互 / 尾部
4占位符 + 注入(5 个必须)占位符唯一 + </ 转义 + 5 状态 + XSS 防护
5渲染器 4 步subprocess → 读模板 → inject → 写副本
6改动前 3 问影响文件 / 数据迁移 / 回滚方案
7Commit 与检索每 Phase 一个 commit + [技能名] 前缀
88 反模式--force / except:pass / 魔法字符串 / 紧耦合…
9命名不带版本review_report.html ✅ review_v2.html ❌
10⭐ HTML 单工铁律(最高)HTML 无反通 LLM 路径,过程型必须设计"复制 prompt"

→ 详见 04-可视化与注入v2.md

设计模式全景(头脑风暴触发器)

总纲不提供模板库,而是给 AI 一份"模式词典"——让 AI 知道有哪些 HTML 设计模式存在,自己头脑风暴挑最适合的。

3 大类:布局模式(单卡列表 / 网格 / 时间线 / 树状 / 看板 / 卡片栈) + 可视化模式(折线 / 柱状 / 热力 / 进度 / 仪表盘 / 散点) + 交互模式(筛选 / 排序 / 搜索 / 下钻 / 对比 / 复制 prompt)。

动手前 3 步:① 列出 3-5 个候选模式 → ② 剔除数据形状不匹配的 → ③ 挑 1 个最贴用户心智的。

第五部分 · 工程仪式

动手前的最后一道防线。这章是自动检查清单,不是原则解释。

3 个核心 SOP

  • 改 Skill SOP:画领域边界 → 列触发词 → 设计 schema → 写硬规则 → 写 CLI → 写集成 → 跑自检
  • 写 HTML 模板 SOP:判定值得做 → 检查 CLI → 选模式 → 复制骨架 → 引用 CSS → 渲染器 → 测试 → 同步
  • Commit 规范:每 Phase 一个 commit + [技能名] 前缀 + 信息含 文件清单 + 行为变化 + 向后兼容

→ 详见 05-工程仪式.md

第六部分 · 5 个不可违背的钩子

这 5 条是总纲的硬规则。违反即视为改动未准备好。

  1. HTML 同步硬规则 — SKILL.md 任何改动,<skill>.html 必须同 commit 同步
  2. 改动前 3 问 — 影响哪些文件 / 数据迁移 / 回滚方案
  3. 触发词设计 v2 — 8 ≤ N,无上限;4 元组(动作+对象+维度+类型)
  4. CLI 阻塞判定 — CLI 不支持 JSON → 模板做不出来
  5. HTML 单工铁律(⭐ 最高) — HTML 是单工设备,过程型 HTML 必须设计"复制 prompt"

第七部分 · 3 个可加载资产

总纲 v2 的关键:从"原则 + 案例" → "可加载资产 + 决策树"。所有 HTML 模板复用这套资产,无需重设计。

资产作用何时用
_assets/style.css 设计令牌(色板 / 字体 / 圆角 / 阴影 / 5 状态) 任何 HTML 模板的 <link rel="stylesheet">
_assets/injector.py 占位符注入(inject() + render_4step()) 渲染器复用,2 行代码完成 4 步流程
_assets/template_skeleton.html 4 段式骨架(首屏 / 主体 / 交互 / 尾部)+ 5 状态 fallback 复制 → 改具体字段 → 上线

样式来源声明:style.css 的设计令牌源自 HTML 唤醒词开发 prompt 描述的 Apple 风格 + 5 层架构手册的 CSS 调色板 + 4 段式样式需求。

第八部分 · 自我检查(动手前)

新写 / 改造一个 Skill 前,逐条勾选:

  • 触发词 ≥ 8 个,用 4 元组结构?
  • SKILL.md 第一段能 30 秒回答"管什么、不管什么"?
  • 5 层划分清晰,规模匹配(微 / 中 / 大)?
  • 硬规则集中在 validators.py,无跳过通道?
  • CLI 都返回 {status, data, message} 三段式 JSON?
  • HTML 模板(如有)有"复制 prompt"按钮?
  • 改动前 3 问答得上来?
  • 每 Phase 一个 commit + [技能名] 前缀?

6/8 通过才能动手。低于 6 个 = 改动未准备好,回去补。

第九部分 · 附录 · 演化规则(RULE Forms)

本附录是从历史案例中抽出的 RULE 形式,非硬编码案例。所有 skill 可复用。不含具体 skill 名 / 具体 BUG 列表。

A · Skill 改造优先级判据(3 维度评估)

维度权重评分标准
复用频率⭐⭐⭐⭐⭐用户每周 / 每天调用次数
视觉密度⭐⭐⭐⭐多字段 + 多条目 + 需要扫读
改造阻力⭐⭐⭐CLI 现状是否容易支持 JSON 化

判定矩阵:

  • S(1-2 周动手):高频 + 高密度 + 低阻力
  • A(2-4 周动手):高频 + 低密度 / 高密度 + 低频
  • B(按需):高密度 + 低频
  • 不做:低频 + 低密度 + 高阻力

B · Skill 改造 4 Phase 模式

任何 skill 改造按"先低阻力后高阻力"顺序。违反顺序 = 返工。

Phase适合场景动作
A · 最小阻力先做CLI 已合规(返回结构化 JSON)补字段 → 1 模板 → 渲染器 → commit
B · 数据已合规数据完整,但缺可视化层选模板模式 → 复用骨架 → 渲染器 → commit
C · 规范化已有 HTML有 HTML 但不符合规范拆分为模板 + 注入 → 数据契约 → commit
D · CLI 重构 + 解锁阻塞多个 P0 阻塞于 CLI 不返回 JSON加 as_dict=True(保默认 print)→ 多模板 → commit

C · 通用改造顺序(综合 §02 + §05)

  1. 画领域边界 → SKILL.md 第一段
  2. 列触发词表 → v2 矩阵(见 §03)
  3. 设计 schema → db.py + 索引 + .bak
  4. 写硬规则 → validators.py(无跳过通道)
  5. 写 CLI → argparse + {status, data, message}
  6. 写集成(如需要)→ integrations/
  7. 跑自检 → 5 层 + 改动前 3 问

D · 何时不该做 Skill 化

单条 CRUD / 高频无视觉 / 后台管理 / 数据采集 / 一次性脚本 / 2 步内查询 → 都不做 Skill 化。判据:唤醒词 ≥ 5 个含"多字段 + 多条目 + 视觉密度"才值得做(对应 §04 原则 0)。

E · BUG 修复优先级

严重度含义反应时间
🔴 致命主流程阻塞 / 数据丢失 / 安全立即
🟡 中等功能部分损坏 / 体验降级当前 Phase
🟢 低边角 / 文案 / 美化下个 Phase
