Q1 · 必须答 影响 AI 加载行为

主入口 SKILL.md 的 description 怎么写？

这是 AI 决定"何时加载总纲"的唯一依据。写得太宽 → 过度参考；太窄 → 找不到。description 应该显式说明边界。

A 推荐 显式排除 cookbook
"SKILL 设计/改造总纲。开发新 Skill 或改造现有 Skill 之前必读。覆盖 5 层架构、触发词、可视化、注入机制、工程仪式。不提供完整代码示例,只给原则 + 可加载资产。"
B 简洁宽泛
"SKILL 开发手册。任何对 Skill 的改动从这里开始。"
→ 易被 AI 过度参考;缺乏边界
C 强约束字眼
"在用户要求'新建/改造/重构 Skill'时强制加载,作为唯一权威。先读后做。"
→ "强制"字眼对 AI 无实际约束力,但显得权威

Q5 · 必须答 影响书写工作量

HTML 镜像（SKILL.md ⇔ HTML）策略？

HTML prompt 要求"SKILL.md 改 → HTML 必同步"。这个铁律怎么执行，决定写总纲的工作量。

A 推荐 主入口 + §02 必配
SKILL.md 和 02-5层骨架.md 各配 1 个 .html(最常被读的两份)
→ 工作量可控,覆盖最高频用例
B 8 .md 8 .html 全配
严格执行"改文档必改 HTML" - 每个 .md 一份镜像
→ 工作量 ×2,维护成本高,但最完整
C 仅主入口
只有 SKILL.md 配 .html,其余 .md 用 markdown 直接读
→ 工作量最小,但违反 HTML prompt 硬规则
Q6 · 必须答 影响触发词 v2 设计

触发词数量上下限？

老 prompt 只说"动词+名词 + 广度×深度",没有量化。v2 必须给硬区间。

A 推荐 8 ≤ N ≤ 30
基于现有 60+ skill 分布:8 是质量底线(覆盖核心 CRUD + 统计 + 导出),30 是认知上限(人脑可记)
→ 中位数 ~16,大多数 skill 的健康范围
B 5 ≤ N ≤ 20 · 严格聚焦
强迫 skill 收缩 - 每个触发词必须高频且不可去重
→ 适合"小型 skill"标准;老 skill 可能需要大删
C 10 ≤ N ≤ 50 · 宁滥勿缺
覆盖优先 - 触发词越多 AI 命中率越高
→ 适合复杂领域(如医药/法律);治理成本高
D 取消上下限
完全靠"广度×深度矩阵"自约束,不给硬区间
→ 最自由,但 AI 可能写 50+ 触发词或 3 个就放弃
Q7 · 必须答 影响全文语气

总纲调性（口吻）？

调性决定总纲怎么说话 — 决定了 AI 模仿的语气。错配会很怪。

A 推荐 指南 · 原则 + 落地
每条原则 → "为什么" + "怎么用" + "什么时候不适用"
→ 中性,最通用,AI 模仿最稳
B 宪法 · 不可违背
每条都是"硬规则",违反即 fail
→ 权威感强,但容易僵化;开发者抵触情绪大
C 课程 · 从零到熟练
循序渐进 + 大量类比 + 反复举例
→ 教学友好,但难以维护
