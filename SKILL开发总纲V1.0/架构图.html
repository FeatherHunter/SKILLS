AI 阅读路径 · 3 种场景

AI 接到 Skill 任务时,根据场景读不同文件。不一定要读完全部,按需加载:

场景 A · 写新 Skill(从 0 到 1)

新建 用户说"帮我写个 XXX skill"
SKILL.md → 01 概念 → ★ 02 5 层 → 03 触发词 → 04 模式词典 → 05 SOP → 06 附录

关键:先画领域边界(§02)→ 列触发词(§03)→ 设计 schema → 写硬规则 → 写 CLI。§05 SOP 给完整 8 步顺序。

场景 B · 改造现有 Skill(1 → 1.1)

改造 用户说"改一下 XXX skill" / "重构"
SKILL.md → 06 附录 A 3维度 → 06 附录 B 4 Phase → ★ 02 5 层 → 05 改动 3 问

关键:先做 3 维度评估(S/A/B/不做),再选 Phase 顺序(A→B→C→D 不跳)。每 Phase 一个 commit。

场景 C · 评审他人改动

评审 用户说"看看这个改动 OK 吗" / Code Review
SKILL.md 5 钩子 → ★ 02 5 层自检 → 04 模式对照 → 05 Commit 规范

关键:用 5 钩子逐条勾;用 §02 5 层自检清单;用 §04 模式对照看 HTML 是否错配。

设计令牌流动 · 从哪来到哪去

总纲 v2 的关键是把"原则"变成"可加载资产"。设计令牌流向如下:

HTML 唤醒词开发 prompt (2026-05-31)
  ├─ Apple 风格描述(白底/留白/克制/SF Pro)
  ├─ Notion 风格描述(米白/圆角卡片/胶囊)
  └─ 4 段式 + 5 状态样式需求(从模板实践抽出)
        ↓
  _assets/style.css (6KB · 总纲提炼的设计令牌)
        ↓ 被以下引用
  ├─ 主 HTML (SKILL开发总纲V1.0.html)
  ├─ 每个 skill 的 HTML 模板
  └─ _assets/template_skeleton.html (同目录 link)

关键事实:

  • 总纲不强制每 skill 复用 style.css——只作为"已验证的设计模板"参考
  • 风格令牌(色板/字体/圆角)应保持一致——避免 50+ skill 各搞一套
  • style.css 与 _shared/injector.py 解耦——前者管视觉,后者管数据

设计哲学 · 总纲遵守的 5 条

总纲本身也是按某些原则设计的。这些原则**也适用于其他文档/工具的设计**。

① 不写硬编码案例

总纲不写"居家管家怎么改"、"卡路里 Phase D 怎么写"。写"原则"(怎么改)和"模式"(可能怎么写),让 AI 自己实例化。用户 Q8 偏好。

② 模式词典 > 模板库

给 18 个 HTML 设计模式(布局 6 + 可视化 6 + 交互 6),不给 18 个完整模板。让 AI 头脑风暴挑 + 从零设计,而不是抄骨架。

③ 可加载资产 > 描述性规则

10 条原则 → 1 个 CSS + 1 个 Python + 1 个 HTML 骨架。AI 复用而非重设计,稳定输出而非每次重新综合。

④ 单一权威 + 显式出处

§02 是 5 层权威,§04 是 HTML 权威。其他章节引用它们,避免规则散落重复。智剪工坊/作息管家 broken refs 已统一指向总纲。

⑤ 软约束为主 · 钩子 5 个硬约束

大部分规则靠 AI 自约束(soft);只有 5 个钩子是硬规则,且其中 1 个(HTML 单工铁律)是最高优先级。其他地方用"软建议"+ checklist 形式。
