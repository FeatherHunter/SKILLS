🍎 SKILL HTML 类型分类

5 类型 × 2 维网格 · 基于"用户意图"的最全面分类

讨论稿 v0.1 · 2026-07-24 · 准备加入《预置HTML+注入数据指导手册》§5.1

核心观点:HTML 类型分类应基于用户打开页面时的意图(看 / 做 / 确认),而不是技术实现(Jinja2 / window / JSON Script)。后者是注入模式(手册 §5),前者是业务类型(本草案)。两者正交。

② 5 类型详解 + 判定公式

① 报告型 Report 最常见

典型场景:用户说"查 X / 看 X" → 打开 → 看 → 关

判定公式:IF 数据已存在 AND 用户无动作 THEN 报告型

设计要点:

  • 首屏 KPI 卡 + 数据可视化
  • 无"复制"按钮(用户看完就走)
  • 空态/错误态/缺数据态必须明确

卡路里映射:home_dashboard · food_ranking · health_dashboard · review_template · workout_plan_view · exercise_review

② 诊断型 Diagnose 易被混为报告

典型场景:用户刚做 X → 系统发现潜在问题 → 用户看到警告 + 安全替代

判定公式:IF 数据已存在 AND 但有"立即行动"建议 THEN 诊断型

关键差异 vs 报告型:诊断型虽然也是"展示",但有"立即行动"的暗示——用户看完应能"复制修复指令"。

设计要点:

  • 顶部 KPI 必含警告数量
  • 每条警告必须有安全替代动作(让用户知道"该怎么办")
  • 推荐提供"复制修复指令"按钮(从报告型升级为过程型特性)

卡路里映射:contraindication_report(腰/膝/肩禁忌 + 安全替代动作)

③ 预览型 Preview 易被混为报告/过程

典型场景:用户准备批量导入 / 提交大批修改 → 先看预演 → 确认才真做

判定公式:IF 数据未提交 AND 用户看完需决定 THEN 预览型

关键差异 vs 报告型:预览型有"即将发生"的张力——用户看完会决定要不要继续。

关键差异 vs 配置型:预览型是"看 + 可选确认",配置型是"调整 + 必须复制 prompt"。

设计要点:

  • 预演 N 条数据(用户能看到每条)
  • 推荐提供"确认执行"复制按钮(从纯报告升级)
  • 失败行高亮 + 跳过策略提示

卡路里映射:batch_import_preview · process_progress

④ 配置/决策型 Configure/Decide 必须有复制按钮

典型场景:用户要做某事 → 在 HTML 里勾选/调整 → 复制 prompt 给 AI → AI 写入 DB

判定公式:IF 必须有"复制 prompt 按钮" + 4 部分 prompt 模板 THEN 配置型

设计要点(来自 04_架构师原则 §原则 10):

  • 必须有"复制 prompt"按钮——HTML 是单工设备,复制是"用户→AI"的唯一通路
  • 必须有"采纳"按钮——一致性
  • 复制内容 4 部分:① 场景 ② 数据 ③ 期望(CLI 命令) ④ 来源
  • 禁止:"一键保存"、HTML fetch AI、隐藏复制按钮

卡路里映射:goal_config(5 slider) · plan_builder_wizard(4 轮对话) · nutrition_label_wizard(识别后校验)

作息管家映射:plan_review(每条事件 5 状态选项)

⑤ 回执型 Receipt 写后漂亮确认

典型场景:用户刚记/改/删 X → CLI 已写 → 漂亮确认页

判定公式:IF 紧跟 CRUD 之后 AND 无需用户决策 THEN 回执型

设计要点:

  • 无"用户选择"(强迫 A/B/C 决定是反模式)
  • 上下文展示:刚记的 ID + 当日累计 + 7 天累计 + 同类排名
  • 可选"一键复制今日进度"(给 AI 自主决定下一步的输入)
  • AI 收到上下文自主回复:继续记 / 看全貌 / 复盘 / 补漏

卡路里映射:weight_log_receipt(记体重 → 新 ID + BMI + 趋势)

作息管家映射:record_receipt(记作息 → 新 ID + 当日累计) · plan_receipt(改/删计划 → 旧→新对比) · plan_receipt_add(补计划,绿色调) · plan_receipt_write(写摘要,紫色调)

④ 与现有 3 类型分类对比

作息管家/卡路里 现有注释里常用"结果型/过程型/回执型"3 类型。这套 5 类型是 3 类型的细化:

3 类型(已有)

  • 结果型 = 报告型 + 诊断型 + 部分预览型
  • 过程型 = 配置型 + 部分预览型
  • 回执型 = 回执型 ✓

问题:边界案例(contraindication_report / batch_import_preview)归类不清

5 类型(本草案)

  • 报告型:纯展示,无操作
  • 诊断型:警告 + 行动建议
  • 预览型:写前预演 + 可选确认
  • 配置型:必须复制 prompt 给 AI
  • 回执型:写后漂亮确认

优势:每类型边界清晰,可逐个判定

⑤ 判定速查表

新模板归属判定流程(点击展开) 
Step 1: 数据是否已存在?
  ├─ YES → 用户有操作吗?
  │         ├─ NO  → 报告型
  │         └─ YES → 诊断型(有"行动"建议)
  └─ NO → 用户看完需决定吗?
            ├─ NO  → 预览型(可选"确认执行"复制)
            └─ YES → 配置型(必须有"复制 prompt"按钮)

Step 2: 这是写后确认吗?
  ├─ YES → 回执型
  └─ NO  → 回到 Step 1
反模式清单(每类型 3 条)

报告型 反模式

  • ❌ 加"复制"按钮但没复制内容
  • ❌ 报告页里塞交互表单
  • ❌ 报告页加"编辑"入口(应跳转到配置型)

诊断型 反模式

  • ❌ 只警告不给出替代动作
  • ❌ 把所有警告塞进 1 个段落(应分组)
  • ❌ 没有"复制修复指令"按钮

预览型 反模式

  • ❌ 预览页没有"确认"或"取消"出口
  • ❌ 预演数据超过 100 行不分页
  • ❌ 用户点了"确认"但页面无响应(没有 prompt 复制)

配置型 反模式

  • ❌ 缺"复制 prompt"按钮
  • ❌ 缺"采纳"按钮(不一致)
  • ❌ 复制内容缺 4 部分之一

回执型 反模式

  • ❌ 强迫用户做 A/B/C 决定
  • ❌ 没有上下文展示(只有 ID,无累计)
  • ❌ 没有"一键复制今日进度"快捷
