C-01 实现路径缺失 / 静默失败 schedule_html_render.py:337-340

问题:render_and_write() 函数里的 template_map 是硬编码字典,只包含 2 个 mode:

template_map = {
    "list-events": "schedule_list_events.html",
    "query-plans": "schedule_list_events.html",
}

方案新增的 3 个 mode(record-list / record-summary / record-range)不在映射内 → 直接进 "未知 mode" 错误分支返回 {status:"error"}。CLI 端不会写盘,只输出一行 JSON 错误。AI 拿到这个错误也只会重试 3 次然后放弃。也就是方案"设计"出来了但首跑 0 文件落盘。

C-03 数据丢失风险 · 映射表过期 _render_report_*.py:44-49 → 新模板 emoji_map

问题:方案说"emoji 用旧词映射,因为历史 category 是旧词"。但旧的 emoji_map 是 flat 字符串("睡眠" / "工作" / "学习" / "运动" / ...),不支持 "工作.AI调优" 这类 8.70 字符串。

C-04 功能覆盖盲点 · 7 选 3 没解释 方案 §3 vs schedule_cli.py:294-358

问题:作息记录域现存的文本 CLI 命令有 7 个:list / detail / summary / timeline / report / range / status。方案新增 HTML 版只覆盖 3 个(list/summary/range),遗漏了 detail / timeline / report / status,且未解释 "选 3 不选 4" 的依据。

具体被遗漏的:

  • detail → 包含 source_contents / analysis_reasoning 完整原文 + AI 推理链。HTML 化后是"溯源"利器,缺了它用户只能回到 CLI 看原始 JSON。
  • timeline → 24h 时间轴可视化已成型(cmd_timeline:316-348)。新模板如果做"list + 时间轴",其实就包含了 timeline。
  • report → 完整报告(records + summary + timeline 综合)。当前 23 个历史 reports/record/*.html 全是这个角色。方案新增的 3 个命令 + 这个 role 没一个对得上,意味着"完整报告"这个最常见诉求在 HTML 化后 没出口(落到 23 个旧 AI 渲染上,但新版用户不知道有这个)。
  • status → 元信息(总记录数/天数/日期范围)。可作为页脚/边栏塞进任意模板,不是必须独立命令。

最致命的是 report:用户最常说"今天总结" → 应该对应 HTML 综合报告。方案只给 list/summary/range,意味着"今天总结"在 HTML 路径里要么拼合 2 个文件(list + summary)、要么召唤 23 个旧 AI 报告——无一致路径。

修复:重新设计映射矩阵。
文本 CLI建议 HTML 命令理由
listrender-records-list单日全字段记录表
summaryrender-records-summary按分类总时长汇总
rangerender-records-range日期范围聚合
detailrender-records-detail含 source_contents 折叠区
timeline(并入 list)list 内嵌 24h 块即可
reportrender-records-report替换 23 个旧 AI 报告,提供统一入口
status(塞 footer)任意模板的 footer 区域
7 → 5(其中 2 个并入,2 个新加),内部一致 + 覆盖所有用户场景。

H-02 payload 注入字段未指定 schedule_records schema

问题:schedule_records 有 9 个业务字段(id / date / time_start / time_end / duration_minutes / activity / category / source_contents / source_timestamps / analysis_reasoning / created_at 实际 11 个)。方案没指明哪几个注入 payload。

盲注的风险:

  • source_contents:每条记录 100~500 字的原文消息。30 条/天 → 30*300 = ~9KB/天,一年 HTML 累积 3.3MB。
  • analysis_reasoning:AI 推理链,可能含用户隐私/情绪原文。直接注入 = 永久磁盘可见,可被截图/分享泄露。

且 escapeHtml 在 schedule_list_events.html:388-392 只对 renderTimelineEvent 用了,但 compPill / escapeAttr 对 JSON.stringify 后的 attribute 是 ' 转义,不是 JSON.parse 时的 \\\" 转义。模板里 data-ev=' + escapeAttr(JSON.stringify(...)) 这条路径里 JSON 内引号转义在 JSON.parse 时可能炸 — — 看 applyFilter 走的是 card.getAttribute("data-ev") 用单引号包裹,OK。但 source_contents 含换行符时 data-ev 属性会断,需要 JSON.stringify(...).replace(...) 二次处理。

修复:payload 字段白名单 ——
  1. 默认注入 id, date, time_start, time_end, duration_minutes, activity, category 共 7 个
  2. source_contents / analysis_reasoning / source_timestamps 三者默认 不注入,只在 render-records-detail 命令下注入
  3. created_at 仅在 list/detail 注入,其余命令省略
  4. 在 HTML 中如需溯源,提供 <details> 默认折叠 + "查看 AI 推理原文" 按钮,二次 fetch(可用 data URL 内嵌,避免新增 API 端点)
  5. 隐私约定:文档建议用户在分享 HTML 前 --redact 参数脱敏

H-06 "Mode B" 术语错误 方案 §3, vs schedule_list_events.html:136-143

问题:方案说"沿用模式 B = 静态 HTML + window.__DATA__ JSON 注入"。但 schedule_list_events.html 实际注入方式是:

<script id="payload" type="application/json">{...JSON...}</script>
...
var raw = document.getElementById("payload");
var payload = JSON.parse(raw.textContent || "{}");

即 从 script 元素 textContent 读,不是 window 全局变量。window.__DATA__ 是另一种模式(把 JSON 赋值给 window 全局),与现有 plan 模板不一致。新模板如果照"window.__DATA__"写,会出现两个 SKU 不同入口——以后模板维护分裂。

修复:统一命名 ——
  1. 叫它 "script#payload" 注入模式 或类似更精确的术语,而不是 "window.__DATA__"
  2. 新模板的 JS 入口代码 必须复用 document.getElementById("payload").textContent 这条路径,与 plan 模板 byte-for-byte 一致
  3. 如果想把 JSON 暴露到 window.__DATA__ 给 devtools 调试,在原模板末尾追加 try{window.__DATA__=payload}catch(e){}(不破坏主路径)
