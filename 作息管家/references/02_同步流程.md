# AI 同步流程

> 本文件定义如何将语录数据库中的消息同步为作息记录。

---

## 同步流程（5步）

```
1. 调用 prepare-messages 
   → 获取游标到现在的所有消息（JSON格式）

2. AI 分析消息 
   → 智能识别活动切换点，生成细粒度记录

3. AI 补全空白 
   → 无消息时段生成 category='未知' 的记录

4. AI 调用 add_record_full() 
   → 逐条写入，保证首尾相接

5. 循环直至记录延伸到"现在"
```

---

## 连续性规则

### 时间必须首尾相接

| 规则 | 说明 |
|------|------|
| 首条记录 | time_start = 游标时间（或 00:00） |
| 后续记录 | time_start = 上一条记录的 time_end |
| 最后记录 | time_end = 当前时间（或 23:59） |
| 空白补全 | 无消息的时间段必须填充 `category=未知` |

### 示例

```
✅ 正确：
  07:38-08:47 社交
  08:47-09:05 通勤
  09:05-10:49 工作
  10:49-10:50 未知（空白1分钟）

❌ 错误：
  07:38-08:47 社交
  09:05-10:49 工作  （中间有空白未填充）
```

---

## 粒度规则

### 记录要细，不要合并

| 规则 | 说明 |
|------|------|
| 最小单元 | 每条记录反映一个最小单元的活动 |
| 活动切换 | 用户切换活动时生成新记录 |
| 细粒度 | 宁可多条记录，不要合并笼统 |

### 示例

```
✅ 正确：1小时对话干了10件事 → 10条记录
  08:00-08:06 工作
  08:06-08:12 学习
  08:12-08:18 运动
  ...

❌ 错误：合并成1条
  08:00-09:00 工作+学习+运动（太笼统）
```

## 游标机制

### 游标定义

游标 = `schedule_records` 表中最后一条记录

```sql
SELECT date, time_start, time_end 
FROM schedule_records 
ORDER BY date DESC, time_end DESC 
LIMIT 1;
```

### 游标作用

- 下次同步从游标位置继续处理新消息
- 不需要额外的 checkpoint 表
- 重启/换环境不会丢失同步位置
