# 同步流程（严格版）

> 本文件定义如何将语录数据库中的消息同步为作息记录。**所有 AI 必须严格遵守本规则，不得自由发挥。**

---

## 核心原则

1. **消息原文**：所有 `source_contents` 必须是一字不差的原始消息内容，不得总结、不得臆造、不得删减
2. **逐条处理**：严格按时间顺序逐条读取消息，不跳号
3. **上下文窗口**：单条消息无法判断时，用前后各5条消息辅助判断

---

## 同步流程（5步）

```
1. 获取该日全部消息（按时间排序）
2. 从第一条消息开始，用滑动窗口判断活动边界
3. 每次判断出一个时间段，立即写入 add_record_full()
4. 该 block 的 source_contents 包含窗口内所有消息原文
5. 继续从下一条消息开始，循环处理
```

---

## 滑动窗口判断规则

### 规则1：明确的结束/开始信号出现时，切割 block

出现以下关键词时，前一个 block 结束，新 block 开始：
- 「晚安」「准备睡觉」「困了」「起来了」「到公司了」「到家了」
- 活动时间发生切换（如从通勤变为会议、从休息变为工作）

### 规则2：消息按时间顺序逐条读取

- 每条消息都看时间戳
- 不跳号、不合并消息
- 按顺序逐条处理

### 规则3：消息原文完整写入

- 该 block 内所有消息的**原文**都写入 `source_contents`
- 不是摘要，是原始消息文本
- 每个 block 的消息数量 = 能判断出活动时间所需的最少消息数

### 规则4：模糊时用上下文窗口

**单条消息无法判断** → 查看「**前5条 + 后5条**」消息

窗口内的消息都作为 `source_contents` 写入该 block。

### 规则5：必须给出推断

- 结合上下文窗口仍无法判断具体活动类型 → AI 必须基于上下文给出最合理的推断
- 不得以任何理由填「未知」或其他模糊类别

---

## 时间连续性规则

- 第一个 block 的 `time_start` = 第一条消息的时间（或 00:00）
- 每个 block 的 `time_start` = 上一个 block 的 `time_end`
- 最后一个 block 的 `time_end` = 最后一条消息的时间
  - **若该日期为过去日期**：可以补到 `23:59`
  - **若该日期为当天**：`time_end` 不得超过当前时刻（如现在是 `11:17`，则 `time_end` 不能超过 `11:17`）
- **前5后5窗口推断**：
  - 以block「开始时间点」为中心，取前5条消息推断该时间段起点前的活动
  - 以block「结束时间点」为中心，取后5条消息推断该时间段终点后的活动
  - 日期边界：首条消息无法说明00:00~首条时间点；末条消息无法说明末条时间点~23:59
    → 用「前一天末尾5条+当天开头5条」推断开头时段
    → 用「当天末尾5条+下一天开头5条」推断结尾时段
- **禁止将「无消息的将来时段」补齐**，最后一条消息之后到当前时刻之间的空白不写入任何记录

## Block 数量校验（必须执行）

### 前置：获取最少 block 数量
```python
from block_count import get_required_block_count

# 获取该时间区间最少需要的 block 数量
# messages_per_block: 每个 block 最多承载的消息数（默认5）
required_min = get_required_block_count(start_ts, end_ts, messages_per_block=5)
```

### 后置：验证 block 数量是否达标
```python
from block_count import validate_record_count

result = validate_record_count(start_ts, end_ts, messages_per_block=5)
# result == True: 验证通过
# result is str: 验证失败，提示词会告诉 AI 哪里不足、如何修正
if result is not True:
    # AI 必须根据提示词重新拆分，直到验证通过
    ...
```

### 提示词示例（验证失败时）
```
Block 数量不足：当前 5 条消息/1个block 规则，区间共 748 条消息，最少需要 150 个 block，实际只有 29 个。请将活动拆分得更细致，每5条消息一个block。
```

## 字段写入规范

| 字段 | 要求 |
|------|------|
| `source_contents` | 消息原文，一字不差，包含窗口内所有消息 |
| `source_timestamps` | 消息原始时间戳，不得修改 |
| `analysis_reasoning` | 说明如何用上下文窗口判断出该 block |
| `activity` | 描述该时间段的活动 |
| `category` | 分类（AI自由填写，参考建议列表） |

## 示例

```
消息序列：
[00:00] hello
[00:21] 晚安
[00:23] 还没睡 再看会儿直播
[00:38] 看好了 晚安
[06:26] 醒了 继续睡🥱

处理过程：
block1 (00:00~00:38):
  - [00:00] hello → 无法判断
  - [00:21] 晚安 → 前5后5窗口：[00:00],[00:21],[00:23] → 仍在聊天/看直播
  - [00:38] 看好了 晚安 → 明确结束信号
  source_contents: "[00:00] hello\n[00:21] 晚安\n[00:23] 还没睡 再看会儿直播\n[00:38] 看好了 晚安"

block2 (00:38~06:26):
  - [00:38] 后无消息 → 用前5后5判断：前5条[00:38]看好了晚安→确认睡觉
  source_contents: "[00:38] 看好了 晚安（中间无消息，用前5后5判断为睡眠）"
```

## 日程计划同步流程（2026-06-29 新增）

> 上述 `schedule_records` 同步是"消息 → 作息记录"。
> 下面说明"AI + 用户讨论 → 日程计划"，与飞书日历联动。

### 日程录入流程

```
1. 用户提出"规划明天/某天"
2. AI 与用户多轮对话，逐步细化：
   - 大块节奏（睡觉/工作/吃饭/学习/...）
   - 时间分段细节（"8点起床洗漱，8:30出门，骑车通勤到公司"）
   - 落到具体分钟的事件列表
3. AI 生成结构化 JSON（按 upsert-plan-events 的 schema）：
   [
     {"time_start":"00:00","time_end":"07:00","title":"睡觉","notes":"深度睡眠","category":"休息"},
     {"time_start":"07:00","time_end":"08:30","title":"起床+通勤","notes":"洗漱+骑车","category":"起居"},
     ...
     {"time_start":"22:00","time_end":"24:00","title":"休息","category":"休息"}
   ]
4. AI 调用 CLI：
   schedule_cli.py upsert-plan-events <日期> --json @plan.json
5. CLI 自动：
   a) 校验 24h 联合覆盖（无空隙无重叠）
   b) 写入 schedule_plans（缺失小时自动 is_active=0）
   c) 询问飞书同步：[Y/n]
      - yes → diff_and_sync 批量 create + 回写 feishu_event_id
      - no  → 仅本地写入
```

### 日程 CRUD + 飞书同步规则

| 操作 | CLI 命令 | 飞书同步 |
|---|---|---|
| **C**reate | `upsert-plan-events <date> --json ...` | 写库后询问"是否同步飞书？" [Y/n] |
| **R**ead | `list-events <date>` | 不触发同步 |
| **U**pdate（改 title/notes/category） | `update-event <id> --title ...` | 若该事件已绑飞书 → 询问"也同步飞书？" |
| **U**pdate（改 time_start/time_end） | `update-event <id> --time-start ...` | 飞书**删旧 + 建新**（不可改时段），询问 |
| **D**elete | `deactivate-event <id>` | 若该事件已绑飞书 → 询问"也删飞书？" |
| Resync | `feishu-resync <date>` | 全量 diff，按事件逐个询问 |

### 决策记录

- **触发时机**：写库成功后 AI 询问用户（不静默同步）
- **删除语义**：软删 (`is_active=0`) + 询问同步飞书
- **改时段**：飞书端点不支持"改时间"，拆为"删旧 + 建新"
- **关联主键**：`schedule_plans.feishu_event_id` 内联维护
- **历史事件处理**：用户已说"飞书视为空白起点"，不主动同步旧孤儿事件

### 飞书能力探测

每次飞书 CLI 自动跑 `feishu_available()`：
- **full** (cli + auth + writable) → 询问同步
- **partial** (cli 已装但 auth/权限不全) → 提示授权
- **missing** (cli 未装) → 提示安装价值（不解锁=①计划同步 ②分钟级拆分 ③双向 CRUD）
