# 数据库结构（2026-06-29 重大重构版）

> 本文件定义作息管家技能的三个核心数据表的结构。
> 2026-06-29 重构：`schedule_plans` 从「24 个 hour_N_planned 字段」改造为「事件型 schema（多行）」，与飞书日历事件模型对齐。

---

## schedule_records（作息记录主表 — 未变）

存储每一条作息时间块记录。

| 字段 | 类型 | 说明 | 必填 |
|------|------|------|------|
| id | INTEGER | 主键（自增） | - |
| date | TEXT | 日期（YYYY-MM-DD） | ✅ |
| time_start | TEXT | 开始时间（HH:MM） | ✅ |
| time_end | TEXT | 结束时间（HH:MM） | ✅ |
| duration_minutes | INTEGER | 持续时长（分钟） | ✅ |
| activity | TEXT | 活动描述 | ✅ |
| category | TEXT | 分类 | ✅ |
| source_contents | TEXT | 消息原文（一字不差，保留原始内容，不得总结、臆造、删减） | ✅ |
| source_timestamps | TEXT | 消息时间戳（原始时间，不得修改） | ✅ |
| analysis_reasoning | TEXT | AI分析推理过程（基于source_contents判断） | ✅ |
| created_at | TEXT | 创建时间 | - |

### 索引规则

- `date + time_start` 为主要查询索引
- 游标定位使用 `ORDER BY date DESC, time_end DESC`

---

## daily_summary（每日摘要 — 未变）

存储每日各类活动的时长汇总。采用 KV 结构，适配任意分类名。

| 字段 | 类型 | 说明 | 必填 |
|------|------|------|------|
| date | TEXT | 日期（YYYY-MM-DD） | ✅ |
| category | TEXT | 分类名（AI 自由填写） | ✅ |
| total_minutes | INTEGER | 该分类总分钟数 | ✅ |
| generated_at | TEXT | 生成时间 | - |

**主键**: `(date, category)` 联合主键

### 写入规则

- **必须先有完整的 24 小时作息记录**，才能生成 summary
- summary 由 AI 分析 schedule_records，按 category 分组计算 duration_minutes 之和
- 分类名由 AI 自由判定，参考 `分类清单.md` 中的建议列表
- 同一 date 下每个 category 只有一行（upsert 语义）

---

## schedule_plans（计划作息表 — **2026-06-29 重构**：从"按小时格子"改为"事件型"）

> **2026-06-29 之前的版本**：24 个 `hour_0_planned ~ hour_23_planned` 字段，每字段存储 1 小时描述（用 `Xmin子段+...` 聚合多条子时间）。
> **本次重构**：1 条记录 = 1 个时间事件（与飞书日历事件模型对齐）。
> **老表**重命名为 `schedule_plans_legacy_2026_06_29`，数据通过 `scripts/migrate_plan_to_events.py` 一次性迁移。

**新表结构**（13 个字段）：

| 字段 | 类型 | 说明 | 必填 |
|------|------|------|------|
| id | INTEGER | 主键（自增） | - |
| date | TEXT | 日期（YYYY-MM-DD） | ✅ |
| time_start | TEXT | 开始时间（HH:MM） | ✅ |
| time_end | TEXT | 结束时间（HH:MM，**末段可为 24:00**） | ✅ |
| title | TEXT | 事件标题（飞书 summary） | ✅ |
| notes | TEXT | 备注/细节（飞书 description） | - |
| category | TEXT | 分类（飞书 emoji / 兜底） | - |
| feishu_event_id | TEXT | 飞书日历事件 ID（用于双向 CRUD 联动） | - |
| last_synced_at | TEXT | 最近一次同步飞书的时间 | - |
| is_active | INTEGER | 1=活跃 / 0=软删（**不真删**，留痕） | ✅ 默认 1 |
| completion | TEXT | 完成情况：已完成 / 已完成（超时） / 部分完成 / 未完成 / 未完成（不可抗力），NULL 表示未复盘 | - |
| completion_note | TEXT | 完成备注/反思（自由文本，如"下雨没跑步，改成在家练"） | - |
| created_at | TEXT | 创建时间 | - |
| updated_at | TEXT | 更新时间 | - |

### 索引

```sql
CREATE INDEX idx_plans_date       ON schedule_plans(date);
CREATE INDEX idx_plans_date_time  ON schedule_plans(date, time_start);
```

不加 UNIQUE 约束：用户会迭代讨论，期间反复 upsert 同一时间段不报错。

### 时间粒度

- `time_start` / `time_end` **不强制整点**：8:13-8:47 也合法
- **24h 录满是 CLI 层硬约束**：所有事件的联合区间必须 ⊇ [00:00, 24:00]
  - 首段 time_start == "00:00"
  - 末段 time_end == "24:00"
  - 中间衔接连续无重叠
- `24:00` 在写飞书时自动转换为次日 `00:00`（飞书 ISO 8601 不接受 24:00）

### 软删语义

- 删除 = `UPDATE is_active=0`
- 不提供硬删（物理 DELETE 仍被操作规范禁止）
- 飞书联动：deactivate 后由 CLI 层询问"是否同步删除飞书事件？"

### 飞书联动模型

- `feishu_event_id` 是 DB 与飞书事件的主外键
- 同步后由 CLI 写回 `feishu_event_id` 和 `last_synced_at`
- 改时段 = 飞书「删旧 + 建新」（飞书日历无"改时间"按钮）
- 改 title/notes = 飞书 `+update`
- diff_and_sync 算法每次 upsert 后可选触发，对比本地与飞书事件差异做 create/update/delete

### 写入规则

- **整日覆盖式 upsert**：传完整 24h 事件列表，缺失的小时自动 is_active=0
- 单条 UPDATE：通过 `id` 定位（仅 title/notes/category/time_start/time_end）
- 单条软删：通过 `id`，UPDATE is_active=0
- 回写飞书事件 ID：通过 `id`，UPDATE feishu_event_id + last_synced_at
- 任意时间可重新激活：UPDATE is_active=1（手动）

### 一致性保证

- 操作规范.md 规定：`schedule_records` / `daily_summary` 严禁 DELETE
- `schedule_plans` 的逻辑删除（is_active=0）允许使用，是计划数据的合理修订语义
- 任何 sync 操作不破坏 `schedule_plans` 的 is_active 标记

### completion 完成情况（2026-07-12 新增）

用于复盘"计划 vs 执行"的闭环。两个字段配合：

| completion 值 | 含义 |
|---|---|
| `已完成` | 按时完成 |
| `已完成（超时）` | 做了但拖延了 |
| `部分完成` | 做了一部分 |
| `未完成` | 没做，责任在己 |
| `未完成（不可抗力）` | 没做，外部原因 |
| `NULL` | 还没复盘（默认） |

- `completion` 为 NULL 时 `completion_note` 无意义
- 筛选靠 `completion`（可枚举），反思靠 `completion_note`（自由文本）
- AI 复盘时可调 `list-events` 看哪些 NULL → 逐条标注（详见 SKILL.md「复盘」章节）
