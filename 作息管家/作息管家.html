作息管家

作息记录与日程计划管理工具。AI Agent 通过 24 个唤醒词驱动 CLI 完成记录、查询、计划、回执。底层数据存 SQLite,可视化通过 13 个预置 HTML 模板输出。

0
记作息 / 补一条作息 / 录作息
单条作息记录规范化写入(9 字段强校验,所有写入路径都走这一入口)
add <9 字段> 或 --json @file ▸ 展开
HTML 输出: record_receipt.html(回执) 必填字段: 9 个

用法示例

# 方式 A: 命令行参数(适合 1 条)
python scripts/schedule_cli.py add \
  --date 2026-07-22 --time-start 10:00 --time-end 11:00 \
  --duration-minutes 60 --activity "写小帅相关代码" \
  --category "工作.AI调优" \
  --source-contents "我在优化 AI" --source-timestamps "10:00" \
  --analysis-reasoning "AI 主动优化"

# 方式 B: JSON 文件(适合批量)
python scripts/schedule_cli.py add --json @record.json

底层工作流程

  1. CLI 接收 9 字段,进入 add_record_full()
  2. 字段完整性校验:9 个字段全填,空字符串视为缺失
  3. category 白名单校验(8 一级 + 70 二级,纯一级给 warning)
  4. INSERT INTO schedule_records
  5. 返回 JSON 三段式 {status, data, message}
  6. AI 用返回的 id 调 render-record-receipt <id> 生成回执 HTML

失败处理

  • 缺字段 → {"status": "error", "missing": [...]}
  • 非法 category → 返回"字段名+当前值+期望值+怎么修"
  • 字段类型错 → 返回具体错误信息

强约束:任何 AI/星火录入流程都应走 add CLI,禁止直调函数绕过校验。

1
准备消息(游标分页)
从 daily_recorder 拉消息供 AI 分析(默认 200 条/页)
prepare-messages [<开始> [<结束>]] ▸ 展开
HTML 输出: 无(数据准备阶段) 必填字段: 无

用法示例

python scripts/schedule_cli.py prepare-messages
python scripts/schedule_cli.py prepare-messages --page 2
python scripts/schedule_cli.py prepare-messages --page-size 100

底层工作流程

  1. CLI 接收分页参数(默认 page=1, page_size=200)
  2. 从 daily_recorder 库拉消息(按 cursor 续读)
  3. 返回 JSON 含 pagination.has_next 标识是否还有下一页
  4. AI 处理完当前页后用 --page N 续读

注意

消息量少时只有 1 页,无需额外操作。这是 同步作息(#2) 的数据准备阶段。

2
同步作息
完整同步流程(准备消息→AI 分析→逐条 add),AI 内部循环
prepare-messages → AI → add ▸ 展开
HTML 输出: 每条记录 1 个 record_receipt.html 粒度: 按活动切换点细分

完整执行流程

  1. 调 prepare-messages 获取消息(始终分页)
  2. 前置: get_required_block_count(start, end, messages_per_block=5) 拿最少 block 数
  3. AI 用滑动窗口(前 5 后 5)判断活动边界
  4. 逐条调 add CLI 写入(9 字段全填)
  5. 日期边界:首条无法说明 00:00~首条,末条无法说明末条~23:59 → 用「前一天末尾 5 条+当天开头 5 条」推断
  6. 后置: validate_record_count(start, end) 校验 block 数量达标
  7. 保证首尾相接(24h 完整覆盖)

粒度原则

按活动切换点细分,宁可多条不要合并。详见 references/同步流程.md。

3
增量同步
从游标位置继续同步(自动读取 get_last_record_full 拿游标)
prepare-messages(自动续游标) ▸ 展开
HTML 输出: 每条记录 1 个 record_receipt.html

与"同步作息"区别

增量同步 = 同步作息的子集,只处理上次游标之后的新消息。流程同 #2,只是 prepare-messages 自动读取游标无需指定时间区间。

4
今天总结 / 生成摘要 / 查作息报告
按分类汇总当日时间(满 24h 出综合报告,不满出摘要)
report <日期> 或 summary <日期> ▸ 展开
HTML 输出: record_day.html(24h 完整)或 record_range.html(不满)

AI 判断逻辑

  • records 已满 24h → 调 report <日期>(list + summary + timeline 综合)
  • records 未满 24h → 调 summary <日期>(仅按分类汇总,提示补全记录后再看完整报告)

输出格式

📊 2026-05-22 作息摘要
  😴 睡眠: 7h0m
  💼 工作: 4h0m
  🚴 通勤: 0h36m
  总计: 24h ✓

17
商量计划
AI + 用户多轮讨论 → 结构化事件 → 24h 录满写入 + 询问飞书同步
upsert-plan-events <日期> --json @plan.json ▸ 展开
HTML 输出: plan_preview.html(预览)→ plan_receipt.html(写入回执)

完整流程

  1. AI 与用户多轮对话,收集每时段活动
  2. AI 整理为 plan.json(events 数组,每条含 time_start/time_end/title/category)
  3. 渲染 plan_preview.html 让用户确认
  4. 用户确认后,调 upsert-plan-events 写入(24h 严格衔接,自动校验)
  5. 写完后渲染 plan_receipt.html 漂亮回执
  6. 询问是否同步到飞书日历(本机装 lark-cli 时)

plan.json 示例

{
  "date": "2026-07-25",
  "events": [
    {"time_start": "07:00", "time_end": "08:00", "title": "晨间冥想", "category": "健康.修行"},
    {"time_start": "08:00", "time_end": "12:00", "title": "写代码", "category": "工作.AI调优"},
    ...
  ]
}

详见 templates/plan_preview.html 预览样式。

19
删计划
单条软删(is_active=0),询问飞书同步删除
deactivate-event <id> ▸ 展开
HTML 输出: plan_receipt.html(共用,action=deactivate)

软删机制

13
补计划
单条追加日程事件。幂等(2026-07-15 修复):按 (date+time_start+time_end) 三元组查重
ensure-plan-event <日期> --time-start HH:MM --time-end HH:MM --title X ▸ 展开
HTML 输出: plan_receipt_add.html(回执,绿色调)

与"商量计划"区别

  • 补 = 增量 1 条(本命令)
  • 商量 = 完整 24h 规划(#17)

幂等性(2026-07-15 修复)

按 (date+time_start+time_end) 三元组查重,title 不参与(title 是展示标签不是身份)。同 (date, time) 多次"落地健身计划"只 INSERT 1 条。

14
复盘
对当天日程逐条回顾:哪些做了、哪些没做、为什么 → 写入 completion 字段
list-events → 逐条 update-event --completion ▸ 展开
HTML 输出: plan_review.html(当日复盘视图)

完整流程

  1. 调 list-events <今天> 拉当日所有事件
  2. AI 逐条问用户:这条做了吗?(已完成/已完成(超时)/部分完成/未完成/未完成(不可抗力))
  3. 逐条调 update-event <id> --completion X [--completion-note Y]
  4. 渲染 plan_review.html 当日复盘视图(完成度进度条 + 分类统计)
20
日程管家同步
Phase 0 反向对账 + diff 询问 create/update/delete
feishu-resync <日期> ▸ 展开
HTML 输出: plan_receipt.html(共用)

完整流程

  1. Phase 0: 拉飞书日历当日事件 vs 本地 DB 事件,做 diff
  2. 对每条差异询问用户:本地创建 / 本地更新 / 本地软删 / 飞书同步 / 跳过
  3. 批量执行用户确认的操作
  4. 渲染 plan_receipt.html 同步结果回执

前提

本机装了 lark-cli 且已授权(#21 飞书探测可验证)。
