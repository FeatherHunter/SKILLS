强制性规定（优先级最高）

以下规定优先级高于本文件中的任何其他规则。

  1. 同步更新 HTML：该技能的所有优化和变动、脚本的所有变动都必须体现在 勋章.html 上。HTML 是技能的唯一可视化参考文档，任何功能变更如果未同步到 HTML，视为未完成。

  2. 逐行确认：对该技能的所有文件（SKILL.md、scripts/*.py、勋章.html）的任何一行修改，都需要明确得到用户的 1 次确认。未经确认不得执行写入操作。

  3. 查询类严禁生成：当用户触发查询类唤醒词（查勋章记录/统计/某类/最近）时，严禁进入"颁发"或"设计生成"路径。查询是只读操作，只读取 medal.py list/stats 的输出 + 发送已有 file_path 文件。

禁用词汇清单（路径C - 绝对禁止）

以下词汇会让AI生成"通用模板感"，必须替换为具体视觉描述：

❌ 禁用词汇 替换方向 示例
"蓝色" 精确到色相 "深靛蓝"、"普鲁士蓝"、"钴蓝"
"金属质感" 具体材质+状态 "水银液态金属"、"氧化古铜"
"清新禅意风格" 视觉现象描述 "涟漪扩散形成年轮"
"金属浮雕质感" 多层结构描述 "阴影层→主体层→高光层"
"好看"、"漂亮" 废话，禁止使用 -
"圆形边框" 具体形态描述 "水银液态金属，表面有微型山脊状波纹"
"发光主体" 光学现象 "宝珠散发白金光晕，向外扩散形成涟漪"
"黑色背景" 精确环境 "深海海底光学效果"、"绝对零度蓝黑虚空"
"金色光效" 具体光效 "日冕物质抛射成金色丝状光流"
"抽象概念词" 具体视觉符号 禁止："神圣感" → 允许："不可战胜的压迫感"

原则：每当你写"XX风格"、"XX感"、"XX效果"时，立刻停下来——这意味着你还在用抽象词汇描述，AI无法理解。

Prompt质量分级表（路径C）

等级 特征 示例 评级
❌ 禁用 颜色泛泛、色调模糊 "浅蓝与银白配色" 不合格
✅ 最低 有具体颜色名 "深靛蓝+紫罗兰+月白" 勉强可用
✅✅ 优秀 有色相+明度+材质描述 "铂金色水银液态金属边框" 合格
✅✅✅ 顶级 有物理现象+光学效果 "干涉衍射、七彩折射、宝珠光晕涟漪扩散" 完美

7个必须维度（路径C - 每少一个效果打折扣）

生成Prompt时必须覆盖以下7个维度，缺任何一个都是不完整的Prompt：

# 维度 检查问题 示例
1 颜色 有没有精确到色相？（不是"蓝色"而是"深靛蓝"） "深靛蓝、铂金色、珍珠白"
2 主体 有没有具体物件+状态？（不是"莲花"而是"未开的莲花蓓蕾"） "墨蓝湖水下沉睡的莲花蓓蕾"
3 光效 有没有光学现象？（折射/衍射/散射/干涉/丁达尔） "露珠表面折射出七彩微光"
4 质感 有没有材质对比？（液态vs凝固、透明vs固态） "水银液态金属 vs 凝固岩浆"
5 背景 背景是否明确？（不是"深色背景"而是具体环境） "深海海底光学效果"、"绝对零度蓝黑虚空"
6 尺度感 有没有微观/宏观的尺度描述？ "像显微镜下的金属晶体结构"
7 氛围 有没有通过现象传达情感？（不是"神圣感"而是"静谧与压迫感下的绝对平和"） "完全的静谧与压迫感下的绝对平和"

生成前自检流程（必须通过才能生成）

路径C Prompt自检清单

生成图片前，逐项检查，每项都必须合格：

❏ 我写的prompt里有没有❌禁用词汇？（有→替换）
❏ 颜色有没有精确到色相？（没有→补全）
❏ 有没有具体物件+状态描述？（模糊→具体化）
❏ 有没有光学现象？（没有→加入折射/衍射/干涉等）
❏ 有没有材质对比？（没有→加入液态/凝固/透明等对比）
❏ 背景有没有明确？（泛泛→具体环境）
❏ 有没有尺度感？（没有→加入微观/宏观描述）
❏ 有没有通过现象传达氛围？（抽象词→视觉现象）
❏ 7个维度是否全部覆盖？（缺→补全）

只有全部通过，才能调用image_generate。

输出=验收标准

生成的图片如果"不够好"，问题不在AI的审美，而在Prompt本身就违反了规范。

正确的做法是回溯Prompt检查哪个维度缺失，而不是让AI去"提高审美"。

症状 原因 解决方案
图片平庸没特色 缺少光学现象+尺度感 加入干涉衍射+微观描述
颜色模糊像模板 颜色没精确到色相 替换为具体色相词
没有神圣感 用抽象词而非视觉现象 改为具体光学现象描述
像普通插画 缺少身份标识或材质对比 加入边框/丝带/底座+质感对比

核心原则（必须遵守）

  1. 每次必须全新设计：不允许复用任何之前的HTML、动画、元素，100件事100种设计
  2. 勋章/奖杯/证书各有灵魂：勋章如独特印章让人眼前一亮；奖杯光芒四射有胜利重量感；证书优雅庄重值得珍藏

视觉设计体系

设计分两层：质量层保证不丑，创意层保证不重复。 质量层是铁律不可妥协；创意层因事件而异，自由发挥。

第一层：创意从事件中来

设计前必须经过三个阶段：类型判断 → 事件解构 → 设计概念。跳过任何一步都会导致设计缺乏灵魂。

阶段一：类型判断

根据事件的性质，先确定颁发的是勋章、证书还是奖杯。这决定了设计的气质和分量：

事件性质 颁发类型 设计气质
完成了某件事 勋章 轻巧、独特、像一枚值得收藏的印章
达到了某个里程碑 证书 庄重、优雅、像一份值得珍藏的文书
赢得了某种竞争 奖杯 辉煌、厚重、像一座值得展示的雕塑

阶段二：事件解构

每个勋章的灵魂来自用户具体做了什么。不要从配色表里挑颜色，从事件本身提取颜色。

关键原则：100件事应该有100种设计，而不是100个不同颜色的同一个模板。

事件解构四步法（适用于任何事件）

遇到任何事件，按以下步骤提取视觉元素：

第一步：提取核心符号 - 这件事的标志性物件是什么？（书、锤子、跑道、锅铲、键盘、画笔……） - 这件事的标志性动作是什么？（翻开、敲击、奔跑、搅拌、点亮……） - 选一个最能代表这件事的，作为视觉焦点

第二步：感受情感色调 - 做完这件事的感受：温暖？力量？平静？兴奋？满足？ - 情感决定画面的冷暖和明暗： - 温暖/满足 → 暖色调（黄、橙、棕） - 力量/突破 → 高对比（深底+亮色冲击） - 平静/自由 → 冷色调（蓝、绿、白） - 兴奋/胜利 → 鲜艳+光效（金、红、粒子）

第三步：想象发生场景 - 这件事发生在哪里？（书房、健身房、厨房、田野、深夜书桌……） - 场景决定背景质感和光线： - 室内暖光 → 柔和的暖色渐变背景 - 户外阳光 → 明亮的自然色调+光斑 - 深夜 → 深邃背景+星光/灯光点缀 - 赛场 → 高饱和+动态模糊

第四步：确定动效语言 - 这件事的节奏是什么？（缓慢而坚定？瞬间爆发？轻柔流动？精密运转？） - 动效应匹配事件本身的节奏： - 阅读/冥想 → 缓慢渐显、柔和呼吸 - 冲刺/举重 → 爆发入场、冲击感 - 创作/烹饪 → 流动感、有机曲线 - 编程/工程 → 精确节奏、几何运动

示例：用户说"我刚跑完5公里" 1. 核心符号：跑鞋、终点线、跑道弧线 → 选"跑鞋踏过终点线"作焦点 2. 情感色调：成就感+畅快 → 暖色高对比，金色光效 3. 发生场景：户外跑道 → 天空蓝渐变背景+阳光光斑 4. 动效语言：奔跑的动感 → 从左到右的速度线+冲线瞬间的爆发光效

示例：用户说"我读完了《百年孤独》" 1. 核心符号：书页、文字、马孔多 → 选"缓缓合上的书"作焦点 2. 情感色调：旅程结束的怅然+满足 → 暖棕+纸白色调 3. 发生场景：安静的阅读角 → 柔和暖光背景 4. 动效语言：文学的优雅 → 书页缓缓闭合、文字渐隐如墨迹消散

阶段三：设计概念（最关键，最容易跳过）

事件解构产出的是视觉素材（符号、配色、动效），但素材本身不是设计。在画HTML之前，必须先想清楚：这些素材怎么组织成一个勋章/证书/奖杯？

这一步决定设计的成败。常见方向：

经典形态——传统的边框+中心图案。边框定义"这是勋章"，中心图案承载事件内容。适合需要明确仪式感的场景。

例：圆形金属边框 + 中间浮雕着合上的书 → 一看就知道是读书勋章

形态融合——边框本身来自事件。边框不是通用的圆形，而是事件中的某个元素变成了框。

例：炸篮网格纹组成圆形边框 + 中间是炸鸡 → 勋章形态和事件融为一体

突破形态——没有传统边框，但通过材质和构图传达"这是勋章"。完全用事件的视觉语言表达奖励感。

例：一块金属质感的炸鸡浮雕，底部有绶带，背后有光晕 → 虽然没有圆框，但材质+绶带+光晕=勋章

微缩场景——整个勋章是一个小世界。事件的场景被微缩进一个奖章里。

例：俯视视角的一盘炸鸡，但全部用金属浮雕质感呈现，周围环绕橄榄枝 → 场景即勋章

选择方向时问自己： - 这个事件是偏仪式感的还是偏生活感的？仪式感用经典形态，生活感可以突破形态 - 事件有没有天然的"框"可以利用？有就用形态融合 - 事件的核心物件有没有足够的视觉冲击力？有就可以尝试突破形态

最重要的原则：先想形态，再填内容。形态和内容是两层，分别设计再组合。不要直接把内容当成形态。

材质和背景从事件的世界观中提取，禁止使用默认值：

设计时最容易犯的错误是：金属框 + 黑色背景 + 发光主体。这是"通用勋章模板"，不是设计。

  • 如果事件来自某个游戏 → 用那个游戏的视觉语言（配色、材质、UI风格）
  • 如果事件来自某个App → 用那个App的设计语言
  • 如果事件来自真实生活 → 用那个场景的自然环境色和材质
  • 只有当事件没有明确归属时 → 才用通用的勋章视觉语言

示例： - 炉石传说排位 → 酒馆木纹+羊皮纸+暖金色，不是金属+黑背景 - 原神探索提瓦特 → 游戏内的元素色调+璃月/蒙德风格 - 跑完5公里 → 跑道塑胶色+天空蓝+自然光 - 做了一顿饭 → 厨房暖光+木桌面+陶瓷质感

第二层：质量铁律（不可妥协）

无论创意如何发挥，以下规则必须遵守：

SVG绘制

  • 勋章主体必须用SVG：勋章的边框、主体图案、挂环、绶带必须用 <svg> 绘制。CSS 只负责页面布局和背景，不要用 CSS 画勋章元素（CSS 圆角矩形、CSS 三角形 hack、CSS conic-gradient 旋转等都不行——质感差、动效受限、渲染不一致）
  • 渐变创造质感：每个主要形状都用 <linearGradient> 或 <radialGradient>，禁止纯色 fill
  • 背景是环境，不是纯色：背景本身需要材质。木纹桌面用 feTurbulence，纸张用纤维纹理，石板用噪点，夜空用星点。一个纯色渐变做背景 = 偷懒
  • 复杂物体用多面叠加：宝石、水晶、金属雕件不是单个形状+单个渐变。用多个形状叠加，每个面用不同渐变模拟不同角度受光。一个六边形+一个紫色渐变 ≠ 宝石
  • 三层结构：阴影层（深色偏移）→ 主体层 → 高光层（亮色叠加）
  • 柔和阴影：用 <filter> 的 feGaussianBlur + feOffset 做投影，不要硬边
  • 滤镜泄漏警告：feTurbulence（噪点纹理）会在整个滤镜区域（矩形）内生成噪点，不仅限于应用它的形状。圆形+feTurbulence = 圆外出现灰色矩形纹理。解决方案：要么把滤镜包在 <clipPath> 内，要么对非矩形形状（圆形、不规则轮廓）避免使用 feTurbulence，改用 feImage 或直接用渐变模拟纹理
  • 圆角与曲线：用 rx/ry 和贝塞尔曲线，避免尖锐直角
  • 最多3个核心元素：一个焦点 + 一到两个辅助，不要堆砌
  • 配色必须来自事件：所有颜色（包括背景色、装饰色、绶带色）必须从事件的世界观中提取，禁止使用默认配色。冥想→深蓝/靛青/月白（宁静），不是红+蓝的随意搭配。运动→活力色系，读书→暖纸色系。如果发现自己在用"通用的勋章配色"，停下来重新从事件提取
  • 身份标识不可省略：每个设计必须一眼看出是"勋章/证书/奖杯"，而不是一张插画。最低要求：
  • 勋章：必须包含边框（圆形/盾形/六角形等）+ 挂环或绶带。挂环和绶带是两种不同的连接方式，选一种即可。没有边框也没有连接件的设计看起来像插画或图标，不是勋章
  • 证书：必须有文书边框（矩形装饰框）
  • 奖杯：必须有底座或台面
  • 没有身份标识的设计 = 失败，无论画面多好看

动效节奏

  • 入场必须有层次：背景先动 → 主体入场 → 细节点缀，有先后
  • 缓动函数：入场用 cubic-bezier(0.34, 1.56, 0.64, 1)，循环用 ease-in-out，禁止 linear
  • 循环要微妙：呼吸（±10%亮度）、微移（±3px）、高光漂移——不要旋转/弹跳/同步运动

文字

文字不是必选项。勋章可以完全没有文字——如果视觉本身已经传达了事件含义，文字就是多余的。只在以下情况加文字： - 视觉符号本身不够明确（如抽象图案需要点明主题） - 事件有标志性数字值得铭刻（如"4000""5km"） - 文字本身作为装饰元素融入构图

字数限制：勋章最多 4 个字；证书可以更长，但文字是装饰的一部分，不是信息载体。

对比度铁律（防止看不清）： - 文字颜色必须与背景有足够对比度。深底用浅字，浅底用深字 - 禁止用低 opacity 让文字"融入背景"作为设计手法——这会导致不可读。如果希望文字低调，用更小的字号或更细的字重，而不是降低 opacity - SVG <text> 的 fill 不低于 rgba(x,x,x,0.7)；CSS 的 color 不低于 opacity: 0.65 - 文字下方如有复杂图案，必须加半透明底色块或文字阴影确保可读性 - 浅色背景特别注意：春日、纸张、晨曦等浅色调勋章，文字容易被"柔和化"导致不可读。浅底上的文字颜色至少 rgba(100,70,50,0.75)，不要用"淡雅"作为牺牲对比度的理由

字体粗细： - font-weight: bold（700）— 主标题、数字，强调存在感 - font-weight: 600 — 副标题，稍弱但仍清晰 - font-weight: normal（400）— 注释性文字、英文副标 - 禁止用 lighter 或 100-300，在小字号下几乎不可见

字号参考： - 勋章主文字：12-16px（CSS）或 14-20px（SVG） - 勋章副文字/英文：8-10px - 数字铭刻（如分数、里程）：28-40px，是视觉焦点的一部分 - 证书文字：按层级递减

生图Prompt构建指南

适用于路径B和路径C。Prompt用中文写。

核心思路

Prompt不是填表，是描述你脑海中看到的那幅画。

事件解构已经给了你素材（符号、色彩、场景、节奏），现在要做的是把这些素材"画"出来——用文字描述给AI听。问自己：如果这个勋章/奖杯是一件实物，它长什么样？摸起来什么质感？放在什么光线下？

写Prompt的思路

  1. 先想清楚主体是什么：不是抽象概念，是具体的物件。"读书"的主体不是"知识"，是"一本合上的书"
  2. 再想它长什么样：什么材质？金属的、纸的、木头的？表面有雕刻还是光滑的？反光还是哑光？
  3. 然后想它在什么环境里：深色背景？暖光？自然光？星空下？
  4. 最后想整体气质：是庄严的、温暖的、有力量感的、还是轻盈的？

铁律：Prompt中禁止出现任何文字内容。 不要写"刻着XX"、"标题XX"、"写着XX"、"铭文XX"。AI生图模型的文字渲染能力极差，生成的文字必然模糊变形。勋章/证书的含义完全通过图案、材质、配色来传达。

路径B的额外要求

路径B的图要嵌入HTML，所以prompt里要加： - 主体居中，周围留够空白（方便后续加边框） - 背景干净（纯深色或透明，方便抠图或直接用） - 不要把画面撑太满，给SVG边框留空间

路径C的额外要求

路径C直接出最终图，所以prompt里要包含： - 勋章形态：边框、挂环或绶带 - 身份感：一看就知道是勋章而不是插画 - 材质感：金属浮雕、丝带光泽、浮雕纹理

路径C绝对禁止在图片中生成文字（这是硬性约束，不可违反）： - Prompt中不得包含任何文字描述：不要写"刻着XX字"、"标题XX"、"写着XX"、"铭文XX" - 不要依赖AI生图的文字能力：当前所有AI生图模型（包括mmx）的文字渲染质量极差，生成的文字必然模糊、变形、错乱 - 用纯视觉符号传达含义：勋章的含义完全通过图案、材质、配色来表达，不靠文字 - 如果确实需要文字（如证书标题、数字铭刻），必须在生成图片后通过HTML/SVG叠加层添加——但这会让路径C变成路径B，所以除非必要，路径C应避免需要文字的场景 - 证书场景特别注意：证书容易让人想加标题，但路径C的证书只能用装饰性边框+浮雕图案表达"证书感"，不能依赖文字

怎么让prompt不简单

一个简单的prompt和一个好prompt的区别在于细节密度。简单prompt只说了"是什么"，好prompt还说了"怎么拍的、什么光、什么触感、什么氛围"。

补充细节的方向： - 光线：从哪个方向照过来？暖光还是冷光？有没有光斑、光晕、丁达尔效应？ - 材质触感：表面是光滑的还是粗糙的？有没有磨损、氧化、使用痕迹？ - 构图视角：俯视？平视？微距特写？广角？ - 色彩细节：不要只说"金色"，是哑光古金、亮面玫瑰金、还是暗铜色？ - 环境氛围：有没有烟雾、尘埃、水汽、光粒子？

奖励分类

颁发原则

宁可多发，不可漏发。奖励的意义是鼓励，不是考核。只要用户做了一件值得肯定的事，就该有回应。

判断标准：用户是否为此付出了努力或克服了阻力？ 是→颁发。不需要"完美完成"才算，开始做了、坚持了、尝试了，都算。

勋章 (Badge)

触发条件：做了任何值得肯定的事

包括但不限于： - 完成任务、习惯打卡 - 开始做某件事（开始了就是胜利） - 坚持了一段时间（哪怕只有10分钟） - 尝试了新事物（第一次做饭、第一次跑步） - 帮助了别人 - 学到了一点东西 - 克服了一点小阻力（不想做但还是做了）

气质：轻巧、高频、像朋友说"不错哦"

数据库记录：medal_type = "badge"

证书 (Certificate)

触发条件：完成了有份量的事或达到了某个节点

包括但不限于： - 完成课程、项目、一本书 - 达成里程碑（连续打卡7天、跑了100公里） - 通过考试、提交了作品 - 完成了一件拖了很久的事 - 在某个领域有了明显进步

气质：庄重、有仪式感、像一份正式认可

数据库记录：medal_type = "certificate"

奖杯 (Trophy)

触发条件：赢了——不管是赢别人还是赢自己

包括但不限于： - 比赛获胜、排名靠前 - 打破了个人记录（PB、最快、最多） - 完成了一个挑战（30天挑战、马拉松） - 克服了一个重大困难 - 在某件事上做到了自己没想过能做到的程度

气质：辉煌、有重量感、像一座值得展示的雕塑

数据库记录：medal_type = "trophy"

渲染路径规范

三条路径共享前三步（类型判断 → 事件解构 → 设计概念），从第四步开始分流。

路径A：SVG路径（产出GIF）

技术栈：Playwright + ffmpeg + MiniMax视觉审查

流程：

第四步：设计HTML
    AI生成全新HTML（含CSS动画+SVG图形）
    body尺寸根据类型决定（脚本会自动读取body尺寸并按输出分辨率等比缩放）：
    - 勋章：400x450（接近正方形，适合圆形/方形徽章）
    - 证书：600x400（横向，适合文书/卷轴）
    - 奖杯：350x500（竖长，适合奖杯轮廓）
    ↓
第五步：保存中间文件
    保存HTML到 MEDAL_RESOURCE_PATH/<名称>.html
    ↓
第六步：视觉审查（核心迭代步骤）
    使用 Playwright 渲染HTML并截图单帧（非GIF）
    注意：截图等待时间必须覆盖所有动画延迟。如果入场动画延迟2秒+持续1秒，截图至少等3.5秒，否则会漏掉晚入场的元素
    ↓
    通过 subagent 调 mmx vision describe（ui-review 模式）分析截图
    ↓
    AI根据审查结果修改HTML → 重新渲染截图 → 再次审查
    ↓
    迭代直到审查通过（最多3轮）
    ↓
第七步：生成最终GIF
    python3 scripts/generate_medal_gif.py --html-file <HTML文件路径> --output <文件名>
    ↓
    ffmpeg合成GIF → 保存到 MEDAL_RESOURCE_PATH/<名称>.gif

中间文件要求：HTML源文件必须与GIF保存在同一目录（MEDAL_RESOURCE_PATH），文件名一致仅后缀不同，便于回溯和调试。

路径B：混合路径（产出GIF）

技术栈：mmx image generate + Playwright + ffmpeg + MiniMax视觉审查

流程：

第四步：生成主体图
    事件解构结果 → 构建prompt（见"生图Prompt构建指南"路径B约束）
    ↓
    mmx image generate --prompt "<prompt>" --out $MEDAL_RESOURCE_PATH/<名称>_subject.png --width 512 --height 512
    ↓
第五步：包装成动画HTML
    主体图作为 <img> 嵌入HTML（src指向本地文件路径）
    外围用SVG画边框、挂环、绶带（几何形状SVG擅长）
    CSS做动画：光晕呼吸、入场层次、微动效
    body尺寸：同路径A规范
    ↓
第六步：保存HTML到 MEDAL_RESOURCE_PATH/<名称>.html
    ↓
第七步：视觉审查（同路径A）
    Playwright渲染截图 → 通过 subagent 调 mmx vision describe 审查 → 迭代修改
    ↓
第八步：生成GIF（同路径A）
    python3 scripts/generate_medal_gif.py --html-file <HTML文件路径> --output <文件名>

关键：路径B的HTML中，<img> 标签的 src 使用绝对路径引用生成的主体图PNG。Playwright渲染时能直接读取本地文件。

路径C：纯生图路径（产出PNG）

技术栈：mmx image generate + MiniMax视觉审查

流程：

第四步：生成完整勋章图
    事件解构结果 → 构建prompt（见"生图Prompt构建指南"路径C约束）
    ↓
    mmx image generate --prompt "<prompt>" --out $MEDAL_RESOURCE_PATH/<名称>.png --width 1024 --height 1024
    ↓
第五步：视觉审查
    通过 subagent 调 mmx vision describe 直接审查生成的图片：
    - 是否符合勋章/证书/奖杯的形态？
    - 材质质感如何？是否有金属/玻璃/纸张的感觉？
    - 配色是否与事件匹配？
    - 身份标识是否清晰（边框、挂环/绶带）？
    - 图中是否出现了文字？如果出现了模糊/变形的文字，必须判定为不合格并重新生成
    ↓
    不满意 → 调整prompt重新生成（最多3轮）
    ↓
第六步：调用 medal.py add --file 写入数据库

路径C不需要Playwright和ffmpeg，直接保存图片即可。

视觉审查通用提示词

三条路径共用同一套审查标准，通过 subagent 调 mmx vision describe（ui-review 模式）：

这是一个勋章/证书/奖杯的设计稿。请从以下角度评审：
1. 视觉焦点是否明确？第一眼看到什么？
2. 材质质感如何？是否有金属/玻璃/纸张的感觉，还是纯色扁平？
3. 配色是否和谐？是否与[具体事件]的氛围匹配？
4. 构图是否克制？有没有元素过载？
5. 身份标识是否清晰？（边框、挂环/绶带、底座）
6. 图中是否出现了文字？如果有，文字是否清晰可读、没有变形？（AI生图的文字质量通常很差，出现模糊/错乱文字必须判定为不合格）
7. 如果要改进，具体改什么？

AI执行规范

  1. 主动鼓励：当用户分享了任何完成的事情、进步、努力，AI应主动提议颁发奖励。不需要用户开口要——"你刚做完了XX，要不要颁个勋章？"
  2. 禁止直接操作数据库：所有读写必须通过 scripts/medal.py CLI
  3. 环境变量必检：执行前检查 SKILLS_DB_PATH 和 MEDAL_RESOURCE_PATH
  4. 生成前确认：展示将颁发的奖励类型、名称和选择的渲染路径，用户确认后再生成
  5. 奖励文件必须发送：生成后必须通过 qqbot 发送给用户（GIF或PNG）
  6. 每次全新设计：不允许复用任何之前的HTML或prompt，必须完全重新设计
  7. 结合具体事件：设计必须与用户完成的具体事情紧密相关
  8. 路径A/B保留中间文件：HTML源文件必须先保存到 MEDAL_RESOURCE_PATH，再用 --html-file 调用脚本，禁止用 --html 直接传入
  9. 路径B保留主体图：生成的主体PNG必须保存到 MEDAL_RESOURCE_PATH，便于回溯
  10. 视觉审查必做：三条路径都必须进行视觉审查，不可跳过。迭代直到审查通过，最多3轮
  11. 路径选择需说明理由：AI选择渲染路径时，应简要说明为什么选这条路径
  12. 成就链主动检查：颁发勋章时，应主动检查用户是否解锁了对应的进阶/里程碑成就
  13. 查询结果格式化：查询类唤醒词输出结果时，使用清晰的表格或列表格式
  14. 成就链进度报告：检查成就链时，输出当前等级、进度百分比、下一解锁条件
  15. 视觉审查必须 subagent：审查设计稿/截图时，必须通过 subagent 子代理调用 mmx vision describe。主 agent 不得直接使用 image 工具或任何视觉模型 API；任何视觉识别都需经 subagent 中转。违者视为严重违规，需向用户说明原因并立即纠正。

成就系统

角色定位

当用户完成某件事时，AI需要： 1. 颁发本次奖励（基础勋章） 2. 检查是否解锁成就（进阶/里程碑勋章） 3. 写清楚 remark（让未来的自己能读懂上下文）

进度追踪完全依赖现有勋章记录，不新增任何表或存储结构。

进度查询方法

每次颁发勋章前，先查数据库：

# 查询某类勋章的所有历史记录
python3 scripts/medal.py list --type badge --limit 100

# 查看统计
python3 scripts/medal.py stats

AI自行分析查询结果，推算进度： - medal_name → 判断事件类别（如"早睡党"属于早睡链） - awarded_at → 判断时间连续性（按日期排序，检查是否连续中断） - remark → 获取上下文详情（入睡时间、睡眠时长等）

成就链匹配规则

每个成就链由三级组成：

等级 解锁条件 触发时机
基础 单次完成某事 用户做了一件值得鼓励的事，立刻颁发 badge
进阶 累计N次同类型行为 颁发基础勋章时，顺便查历史记录，达到条件则追加颁发 certificate
里程碑 连续N天同类型行为 颁发基础勋章时，查历史记录按日期排序，连续天数达标则颁发 trophy

连续判定规则：连续N天中若中断1天即视为断链，连续天数从最近一次开始重新累计。

remark 写作规范

remark 是AI自己的"记忆备份"，未来判断进度时全靠它。

必须包含的要素： - 日期（05-25） - 时间（22:30入睡） - 关键上下文（睡了6.5小时/学习2小时/骑行了45分钟）

格式示例：

早睡链：05-25 22:30入睡，睡了6.5小时
学习链：05-25 学习Java并发2小时
运动链：05-25 骑行上班45分钟
体重链：05-25 称重95.2kg

中断恢复时remark格式：

早睡链：05-28 中断恢复，前两天加班到凌晨
学习链：05-20 中断，连续3天未学习
