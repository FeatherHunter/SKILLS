一、第一性原理 · 4 问判据

来自《预置HTML+注入数据指导手册》第 3 节 + 《优秀 Skill 指导手册》第④层接口层。

4 问判据模型

Q1 · 数据复杂度
返回值是否含多组数据/字段（≥10 个）？
Q2 · 视觉密度
用户要扫读几次以上？反复对比？
Q3 · 交互需求
需要折叠/勾选/计时/筛选/份数调整？
Q4 · 复用频率
用户会不会 1 周内多次调用？

4 个"谁负责"

谁负责卡路里当前做法HTML 模板可改进点
数据✅ CLI/JSON（已经做对）不变
业务规则✅ validators/constraints不变
体验⚠️ 依赖 print 文本输出← HTML 模板的用武之地
变化 vs 稳定❌ 每次回答都临时拼✅ 模板稳定、数据流动

二、评估模型 + 4 档优先级

基于 4 问判据打分：Q1-Q4 各打 0/⚪/✅/✅✅，汇总后映射到 4 档。

档位含义判据数量
🔴 P0必做（高 ROI）已完成 1-2 项或视觉密度极高 + 高复用5
🟡 P1强烈推荐典型适合（分析/对比/排行类）12
🟢 P2可做不迫切查询列表类，改进锦上添花17
⚪ P3不适合单条 CRUD，一句话回执已足够14
📊 4 档的具体判据细则（点击展开）

P0 必做（视觉密度极高 + 高复用频率）：

  • 数据维度 ≥ 5 个 + 视觉密度 ≥ 2 个 ✅ + 复用频率 ≥ 2 次/周
  • 或：已完成 HTML 资产但未完全模板化（健身计划.html / review_template.html）

P1 强烈推荐（典型适合）：

  • 数据维度 ≥ 3 个 + 视觉密度 ≥ 1 个 ✅ + 含对比/排行/汇总
  • 或：用户每周调用 ≥ 1 次且需要扫读多次

P2 可做不迫切（锦上添花）：

  • 数据维度 2-3 个 + 用户每日查看
  • 或：流程编排类但用户已习惯文本输出

P3 不适合（保持 CLI 简洁）：

  • 单条 CRUD（add/update/delete）+ 一句话回执
  • 或：单次配置类操作（设目标/开启定时）

七、反模式提醒（绝对不要做）

反模式为什么不能做
❌ 单条 CRUD 加 HTML杀鸡用牛刀，破坏 CLI 简洁性（记吃了 / 记喝水 / 记运动）
❌ 简单状态查询加 HTML一句话能说清的事不必上页面（设体重目标 / 开启定时复盘）
❌ 5 个食物排行做成 5 个独立模板应该 1 个模板 + 5 个 category 参数
❌ HTML 里硬编码 HTML 字符串拼接复用度低、易出错（不要 html += f"<div>{name}</div>"）
❌ HTML 直连数据库违反《手册》第 4 节"五层职责映射"——必须 CLI 出 JSON → 注入
❌ 炫技动画 / 3D 图表用户要的是数据，不是特效
❌ 每次都重新生成全模板破坏"模板稳定"原则——模板改一次就行

九、第一性原理总结

卡路里 Skill 中适合"模板+注入"的唤醒词，本质都是"用户要反复扫读/对比/复盘/排行"。

判据口诀

  • ✅ 多字段 + 多条目 + 视觉密度 → 适合（详情页/报告页/排行页）
  • ✅ 多次复用 + 需要进度/对比/排行 → 强适合
  • ❌ 单条记录 + 一句话回执 → 不适合（CLI 文本已足够）

最有价值的 5 个改造

  1. 查健康报告（4 维 KPI + 联动建议）—— 一屏全显，复用频率最高
  2. 复盘（8 dim 卡片 + 飞书发送链路已通）—— 已有 70% 基础
  3. 5 个食物排行（1 模板 5 参数）—— ROI ×5
  4. 复盘训练（完成率热力图 + 漏做最多动作）—— 每次训练后必看
  5. 扫禁忌（已有结构化 hits）—— 改完 plan 必扫 1 次

核心原则

模板稳定，数据流动；样式预置，内容注入

不要只把 CLI 字段搬进 HTML，先盘点 CLI 全量字段 → 按用户场景思考 → 提超出原始请求的建议。
