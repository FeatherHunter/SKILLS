## 一、背景与目标

用户希望把卡路里技能的 `nutrition_products` 表从当前的 50 条,扩展到 **1000-10000 条**,目标是一个**个人级食物营养百科**,而非"个人饮食库"。

### 核心动机

1. **效率**:用户每天记录饮食时,90% 的食物应该能在库里秒命中,不需要 AI 临时换算
2. **覆盖**:即使是没吃过的食物,也要能在库里找到,避免每次"按克数估算"
3. **健身场景**:用户一天三次健身房,蛋白/碳水/脂肪的精确追踪是刚需
4. **长期主义**:1-3 年累积,这个库会成为用户饮食决策的"权威数据源"

### 用户饮食结构(从历史记录推断)

```
🍽️ 主食:米饭、面食、燕麦、全麦面包(粗细粮都有)
🥩 蛋白质:鸡蛋、牛奶、鸡胸、虾、酱鸭、酸奶
💪 健身补剂:可能有蛋白粉(待确认)
🍌 加餐水果:香蕉(今天 4 根)、苹果(?)
🥬 蔬菜:偏少(库里只有番茄沙司,缺新鲜蔬菜数据)
🧂 调料/酱:Member's Mark 系列、泰森、空刻、湾仔码头
🍜 半成品:空刻意面、湾仔码头水饺
```

## 二、第一性原则

### ❌ 错误理解

- "补全中国食物营养表"(百科任务,10000+ 条都补不完)
- "围绕用户的饮食建库"(库会越来越窄,最终失去价值)

### ✅ 正确理解

- **库的目标 = 用户能吃到的食物的 90%+ 覆盖**
- **数据源必须是权威**(中国食物成分表 / USDA / 包装标签)
- **入库只是手段,目的是让未来的"记 XX 克"零摩擦**
- **库的扩展必须可批量、可去重、可追溯**

## 三、现状分析

### 3.1 表结构(2026-06-30)

```sql
CREATE TABLE nutrition_products (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    product_name TEXT NOT NULL,
    brand TEXT,
    calories REAL NOT NULL,
    protein REAL NOT NULL,
    fat REAL NOT NULL,
    saturated_fat REAL,
    carbohydrates REAL NOT NULL,
    sugar REAL,
    dietary_fiber REAL,
    sodium REAL NOT NULL,
    note TEXT DEFAULT '',
    created_at TEXT DEFAULT CURRENT_TIMESTAMP,
    updated_at TEXT DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_product_name ON nutrition_products(product_name);
```

### 3.2 库数据(当前 50 条,有效 48 条)

- ID 11、47 已废弃(用 note 标记 "[已废弃]")
- 几乎全是**包装食品 + 半成品**(Member's Mark、空刻、泰森、湾仔码头)
- 缺少:**家常菜原料**(鸡胸/鱼/蔬菜) + **新鲜蔬果** + **健身常用食材**

### 3.3 核心问题

| 缺什么 | 为什么必需 |
|--------|-----------|
| `category`(分类) | 10k 条没法靠名字搜分类,需要枚举查询 |
| `source`(数据来源) | 中表 / USDA / 包装标签 不同权威等级,必须标注 |
| `edible_part`(可食部分) | 不同食物皮/骨占比差很大(香蕉 0.65,鸡蛋 0.88),影响 per-100g 口径 |
| `alias`(别名) | "苹果 / Apple / 红富士" 都要能搜到 |
| `is_deprecated`(废弃标记) | 不靠 note 字符串,需要布尔字段 |
| FTS5 全文索引 | `LIKE '%xx%'` 全表扫描慢,10k 条搜索会卡 |

## 四、改造方案(三阶段)

### 4.1 第一阶段:今天(2-3 小时)— 打地基 + 首批 100 条

**目标:可扩展的架构 + 第一批高频食物**

```
A. 扩展表结构(ALTER TABLE 加 5 个字段)
B. 写 batch_import.py(读 JSON 批量导入,带去重)
C. 命名/分类/数据源 三规范文档化
D. 录入 P0 第一批:健身蛋白 20 + 主食 12 + 水果 10 = 42 条
   + 用户之前常吃但库里没的:香蕉、酱鸭 等
E. 全库 lint:去重、查缺补漏
```

### 4.2 第二阶段:本月(每天 30 分钟)— 扩到 1000 条

**目标:覆盖日常饮食 90%**

```
A. 补完整中国食物成分表第 6 版第一册核心条目(~600 条)
B. 用户每吃一样新东西,顺手入库(养成习惯)
C. 每周 lint 一次,合并重复
D. 加 FTS5 全文索引(库 > 500 条时启用)
```

### 4.3 第三阶段:长期(1-3 年)— 扩到 10000 条

**目标:接近完整食物百科**

```
A. 加 USDA 兜底数据
B. 加食物图片 / 品牌库
C. 开放搜索 API / 与小工具联动
D. 数据导出/导入工具(便于跨设备同步)
```

## 五、命名规范

### 5.1 格式

```
[修饰词] + 基础名 + [状态/部位]
```

### 5.2 示例

```
✅ 鸡胸肉(去皮,生)
✅ 鸡胸肉(熟,无盐)
✅ Member's Mark 香煎鸡胸肉
✅ 鸡蛋(全蛋,生)
✅ 鸡蛋(蛋白,生)
✅ 米饭(白米,蒸熟)
✅ 燕麦片(快熟,Member's Mark)
✅ 香蕉(去皮,生)
✅ 苹果(红富士,生)

❌ "鸡胸"(太短,搜索歧义)
❌ "Member's Mark 鸡胸"(没说状态/部位)
❌ "鸡胸肉"(没说生熟/去皮)
```

### 5.3 命名规则细则

1. **永远 per-100g,可食部分**(已经扣过皮/骨/壳)
2. **状态/部位用括号注明**:生 / 熟 / 蒸 / 烤 / 油炸 / 干 / 鲜
3. **品牌放最前**(包装食品):`Member's Mark X`
4. **同物不同源加来源注**:`鸡蛋(全蛋,USDA)` / `鸡蛋(全蛋,中表)`
5. **搜索关键词至少 3 个**:`鸡胸肉`、`鸡`、`chicken breast`(英文用于国际搜索)

## 七、数据源策略(权威分级)

### 优先级(高 → 低)

| 等级 | 数据源 | 适用 | 标注 |
|------|--------|------|------|
| 🥇 | 中国食物成分表第 6 版 | 中国家常食物原料 | `source = "中表6"` |
| 🥈 | USDA FoodData Central | 国际通用 / 中表没有的 | `source = "USDA"` |
| 🥉 | 包装标签 | 包装食品实测 | `source = "包装标签"` |
| 4️⃣ | 权威食谱计算 | 复杂菜品估算 | `source = "食谱计算"` |
| 5️⃣ | AI 估算 | 实在找不到 | `source = "AI估算"`,**不推荐** |

### 校验规则

- 同食物有多源数据时,**优先高等级**
- 同一等级多个值时,**取中位数或加权平均**
- 用户实测 > 库数据(用户实测可触发 update)

## 八、批量录入工具设计(batch_import.py)

### 8.1 输入格式(JSON Lines)

```jsonl
{"name": "鸡胸肉(去皮,生)", "category": "肉蛋类", "brand": null, "calories": 133, "protein": 24.6, "fat": 5.0, "saturated_fat": 1.4, "carbohydrates": 0, "sugar": 0, "dietary_fiber": 0, "sodium": 45, "edible_part": 1.0, "alias": "鸡胸|chicken breast", "source": "中表6", "note": ""}
{"name": "鸡蛋(全蛋,生)", "category": "肉蛋类", "brand": null, "calories": 144, "protein": 12.6, "fat": 9.5, "saturated_fat": 3.1, "carbohydrates": 1.1, "sugar": 1.1, "dietary_fiber": 0, "sodium": 124, "edible_part": 0.88, "alias": "鸡蛋|egg", "source": "中表6", "note": ""}
```

### 8.2 功能要求

```python
batch_import.py
├─ 读 .jsonl 文件
├─ 校验每条数据(必填字段、范围合理)
├─ 去重(name + brand 相同视为同一条)
├─ 同名已有:询问"覆盖 / 跳过 / 合并"
├─ 批量 INSERT(用事务,失败回滚)
├─ 输出统计:成功 N 条 / 跳过 M 条 / 失败 K 条
└─ 失败原因明细(供 AI 修正)
```

### 8.3 CLI 设计

```bash
# 导入
python batch_import.py import data/p0_protein.jsonl --dry-run   # 预演
python batch_import.py import data/p0_protein.jsonl             # 实际写入

# 去重
python batch_import.py dedupe                                  # 全库去重

# 校验
python batch_import.py validate data/p0_protein.jsonl           # 不写入,只校验

# 导出
python batch_import.py export --category 肉蛋类 --output protein.jsonl
```

## 十一、风险与边界

### 11.1 风险

| 风险 | 缓解 |
|------|------|
| AI 估算数据不准 | `source = "AI估算"`,用户实测时触发 update 修正 |
| 同食物不同来源数据冲突 | 优先级规则:中表 > USDA > 包装标签 > 食谱 > AI |
| 命名不统一,搜不到 | 强制命名规范 + alias 字段 + FTS5 索引 |
| 批量导入失败回滚不彻底 | 用 SQLite 事务,失败整体回滚 |
| 1w 条搜索性能问题 | 阶段二加 FTS5,阶段三可考虑迁移到 PostgreSQL |

### 11.2 边界(不做什么)

- ❌ 不做实时 API(用户不需要,本地 CLI 足够)
- ❌ 不做图片识别入库(走现有"拍营养表"流程)
- ❌ 不做 AI 自动推断(入库前必须 web_search 查权威源)
- ❌ 不做跨设备同步(用户没提需求)
