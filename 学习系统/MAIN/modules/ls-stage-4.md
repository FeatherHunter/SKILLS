## 阶段4：实现理解

**目标**：能说清楚"怎么实现"，提取核心思想

**设计哲学**：
- AI时代不需要背源码，要提取核心思想
- 边读源码边产出，不是光看不练
- 面试呈现：类图+伪代码+核心思想

**达到Level**：Level 4（能说怎么实现）

**进度跟踪**：
```
- [ ] 步骤0：实施准备
      ├─ ❶ 判断知识类型 + 所需资源
      ├─ ❷ 检查本地是否已有
      ├─ ❸ 给出下载/获取命令（如需要）
      ├─ ❹ 等待用户确认"已就位"
      └─ ❺ 进入步骤1
- [ ] 步骤1：L4专项验证
- [ ] 步骤2：找入口
- [ ] 步骤3：源码追踪
- [ ] 步骤4：核心思想提取
- [ ] 步骤5：记忆锚点创造
- [ ] 步骤6：闪示回顾
- [ ] 步骤7：生成复习计划
- [ ] S4_GATE：阶段门控检查
```

**AI 强制约束**：
- 每完成一个步骤，在对话中更新对应 checkbox 为 `[x]`
- GATE 前必须汇报当前进度
- **源码片段入库**：各步骤追踪到的关键代码段，用户标注后通过 LLM-WIKI INGEST 写入 `wiki/raw/sources/`。格式：`# 来源：[仓库名] [文件路径]:[行号]` + 代码片段 + 用户标注
- **图表绘制**：各步骤中出现流程、对比、调用链等可视化需求时，使用 **baoyu-diagram** 生成 SVG，通过 LLM-WIKI 存入 `wiki/raw/assets/`，entity 页 frontmatter 引用

   - 选项A：继续基于现有理解学习，后续补录

3. 用户选择后，按用户选择执行
4. 记录用户的处理决策
```

## 步骤0：实施准备

**加载 `ls-source-prep.md`**，按其中「知识类型 → 权威实现映射」表和子步骤 ❶-❺ 执行。

简要流程：
- ❶ 查表判断需要哪些源码/文档
- ❷ 检查 `sources/` 下是否已有
- ❸ 给出精确可执行的下载命令（已有则跳过）
- ❹ 等待用户确认"已就位"
- ❺ 进入步骤1

**源码 vs Wiki 路径判断（步骤2/3 启动时）**：

| 源码 | Wiki 有实现分析（含具体文件路径+行号） | 步骤2-3 |
|------|--------------------------------------|--------|
| ✅ | ✅ | 源码 + Wiki 对照追踪（最完整） |
| ✅ | ❌ | 纯源码追踪，产出写入 Wiki |
| ❌ | ✅ | 基于 Wiki 分析（降级） |
| ❌ | ❌ | **无法执行**，提示用户二选一：(A) 下载源码 (B) 先生成实现分析进 Wiki |

**"Wiki 有实现分析"的判定**：实体页中必须有具体源码文件路径+行号。概要级描述（如"JVM 插入内存屏障"）不算。

## 步骤1：L4专项验证

**心理学技巧**：提取练习、验证式学习

**目标**：验证当前水平，找到起点

```
🔬 L4专项验证：你能说清楚"怎么实现"吗？

1. 这个知识点的核心类/方法是哪个？

## 步骤2：找入口

**心理学技巧**：情景编码

**目标**：知道从哪开始，不是从头读到尾

**核心设计理念**：你不需要读几十遍源码，你需要知道入口+关键路径

```
🚪 找入口：

任务：
1. AI通过 LLM-WIKI QUERY 告诉你：
   - 这个知识的源码大概在哪（如：android.os.Handler）
   - 核心入口类是哪个
   - 需要看哪几个关键类

示例输出：
- 入口类：Handler
- 关键类：Looper、MessageQueue、Message
- 调用链：Handler.sendMessage() → MessageQueue.enqueueMessage() → Looper.loop()
```

【循环验证】
```
1. 用户确认理解入口位置
2. AI确认用户能说出入口类名
3. 如果说不清 → 重新解释入口 → 循环
```

【精华沉淀】（步骤2结束时）
```
1. AI整理：
   - 入口类：____
   - 关键类：____
   - 调用链：____
2. 展示给用户
3. 用户确认
4. 执行 LLM-WIKI INGEST
```

## 步骤3：源码追踪

**心理学技巧**：产出驱动、验证式学习

**目标**：带着问题追踪源码，边读边产出

**核心设计理念**：ABC都要，按顺序来：入口→关键代码→问题追踪

### 执行流程

```
🔍 源码追踪：

阶段A：入口定位
任务：找到入口类，确认你能在源码中定位到这个类

阶段B：关键代码段
AI通过 LLM-WIKI QUERY 给你一段关键代码
任务：
1. 标注代码中的关键行
2. 说出这段代码在做什么

阶段C：问题追踪
AI提出一个问题（如："当调用sendMessage()时，消息是怎么到达MessageQueue的？"）
任务：沿着问题追踪源码，找到答案

⚠️ 每完成一个阶段，产出该阶段的笔记
```

【循环验证】
```
1. LLM-WIKI QUERY 获取该知识点的实现分析答案
2. 用户提交追踪结果+产出物
3. 对比：
   - 用户能说清调用链 → 掌握
   - 有重大遗漏 → 补强 → 重新追踪 → 循环
```

【精华沉淀】（步骤3结束时）
```
1. AI整理学习过程中的精华：
   - 关键代码段理解：____
   - 调用链描述：____
   - 问题追踪的答案：____
   - 用户的独特发现：____
2. 展示给用户看全部内容
3. 用户确认
4. 执行 LLM-WIKI INGEST
```

## 步骤4：核心思想提取

**心理学技巧**：精细加工、抽象化

**目标**：从源码层面提升到思想层面

**核心设计理念**：AI时代不需要背源码，要获取核心思想

```
🎯 核心思想提取：

任务：
1. 【伪代码】：用自己的话写出核心流程的伪代码
2. 【类图】：画出核心类及其关系
3. 【思想总结】：一句话说出这个知识的核心设计思想

示例产出：
- 伪代码：
  ```
  loop() {
    msg = queue.take()
    handler.dispatch(msg)
  }
  ```
- 类图：Handler ──→ Looper ──→ MessageQueue
- 思想：Handler将任务执行与任务提交解耦

⚠️ 这是面试的核心产出物，必须完整
```

【循环验证】
```
1. LLM-WIKI QUERY 获取该知识点的核心思想
2. 用户提交产出物
3. 对比：
   - 伪代码核心逻辑一致 + 类图正确 + 思想正确 → 掌握
   - 有遗漏 → 补充 → 重新产出 → 循环
```

【精华沉淀】（步骤4结束时）
```
1. AI整理学习过程中的精华：
   - 伪代码版本：____
   - 类图：____
   - 核心思想：____
   - 与标准答案的差异：____
2. 展示给用户看全部内容
3. 用户确认
4. 执行 LLM-WIKI INGEST
```

```
⚓ 创造你的记忆钩子：

1. 关联一个你**熟悉的事物**
2. 必须是你**自己**想到的，不是AI给的
3. 格式："XX就像XX，因为..."

示例：
- AI示例："Handler就像快递员，因为..."

不要复制示例，必须自己想一个！
```

【循环验证】
```
1. LLM-WIKI QUERY 获取该知识点的记忆技巧参考
2. 用户提交锚点
3. 对比：
   - 用户锚点有效且原创 → ✅ 进入下一步

```

## 阶段4 精华沉淀（汇总）

【可选：使用技能 baoyu-infographic 生成 实现全景图】

**目标**：将步骤1-7的精华串成完整的知识线

**流程**：

1. 【整合】AI将步骤1-7的精华整合：验证→入口→追踪→思想→锚点→回顾→计划
2. 【展示】将整合结果展示给用户
3. 【确认】问用户："需要沉淀吗？"用户确认
4. 【执行】如用户确认 → 调用 LLM-WIKI INGEST；如用户认为没必要 → 跳过
