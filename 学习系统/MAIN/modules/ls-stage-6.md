## 步骤1：极限测试

**数据来源**：LLM-WIKI QUERY 获取该知识的性能数据

```
📊 极限测试：

1. AI调用 LLM-WIKI QUERY 获取该知识的性能数据：
   - 性能极限（QPS、延迟）
   - 容量极限（数据量、并发数）
   - 资源极限（内存、CPU）
   - 实际测试环境

2. 用户学习这些数据

3. AI交互追问：
   "你觉得这个性能数据合理吗？"
   "和你之前了解的数据有差异吗？"

4. 讨论：数据差异可以用"设备/环境不同"来解释
```

```
   |------|------|------|----------|
   | XXX | ... | ... | ... |
   | 方案A | ... | ... | ... |
   | 方案B | ... | ... | ... |

   "为什么选XXX而不是YYY？"
```


### 反面案例卡产出

**这是整个阶段6的核心产出物**。面试二面高频问题："这个技术有什么问题？什么场景不该用？"。

```
【反面案例卡产出】

AI 将步骤 1-5 的边界探索成果整合为一张反面案例卡：

## 反面案例卡：[XX技术] 的失败场景

### 什么时候不该用
[1-2 句话说明核心不适用的场景]

### 典型失败案例
**场景**：[业务场景]
**后果**：[用了会怎样——具体症状，如 ANR、OOM、死锁]
**根因**：[为什么在这种场景下会失败]
**替代**：[这个场景应该用什么]

### 过度使用的信号
- [信号1]：[出现这个症状，说明你可能用错了]
- [信号2]：[另一种警示信号]

### 从失败到正确的迁移路径
如果已经在用但发现不适合 → A(评估)→B(对比)→C(迁移) 三步走

---
流程：
1. AI 整合步骤 1-5 的边界探索内容 → 填入反面案例卡模板
2. 展示给用户看完整卡片
3. 用户确认/修改
4. 执行 LLM-WIKI INGEST → concepts/failure-case-{知识名}.md
5. 执行 CLI：python3 learning.py progress update_interview_assets <id> failure_case_path '<path>'
```

## 步骤6：精华沉淀汇总

【可选：使用技能 baoyu-diagram 生成 边界全景图】

**目标**：将步骤1-5的精华串成完整的边界认知

```
1. 【整合】AI将步骤1-5的精华整合：
   - 性能数据（具体数字）
   - 坑点列表（3个以上）
   - 对比结论（方案优劣+适用场景）
2. 【展示】将整合结果展示给用户
3. 【确认】问用户："需要沉淀吗？"用户确认
4. 【执行】如用户确认 → 调用 LLM-WIKI INGEST；如用户认为没必要 → 跳过
```

## 阶段完成判定

- 能说出性能极限（具体数据）
- 能说出3个以上坑点
- 能说出替代方案优劣和适用场景
- 反面案例卡已产出并写入 Wiki

## 阶段完成后的 related_to 关联写入

阶段6完成时，AI 应将步骤4中对比过的替代方案自动建立 `related_to` 关联：

```
1. 提取步骤4中对比过的每个替代方案对应的知识点
2. 对每个，在当前 entity 页 frontmatter 写入：
   related_to:
     - knowledge_id: [替代方案知识点ID]
       relation: alternative
       established_at: [今天日期]
3. （如果步骤4对比表中已有类似关系描述，优先用已有字段）
4. 展示建议的关联给用户
5. 用户确认后 → 执行 LLM-WIKI INGEST（更新 entity 页 frontmatter）
```
