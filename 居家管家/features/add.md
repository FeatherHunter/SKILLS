# 录物品 / 拍物品

## 流程概述

1. 解析用户描述，提取基本信息
2. 位置智能推荐（同类物品位置，**强制 Step 2.5**）
   - 如用户说"和XX放一起" → 走 Step 2.6 参考锚定
3. 补全缺失字段
4. **检查相似物品（强制 Step 4，先调 search 查相似）**
5. 确认并写入数据库（已提供图片时通过 `--photo` 一次写入）
6. 展示结果并回读确认；未提供图片时可提示是否补图

---

## Step 1: 解析用户描述

AI 从用户自然语言中提取：
- 名称
- 分类
- 位置（**推荐 + 确认流程见 Step 2.5 / 2.6**）
- 数量
- 标签
- 备注
- 购买日期（可选）
- 过期日期（可选）

对照 `categories` 表(权威源)推断分类,不确定则询问用户。
执行 `python category_manager.py tree` 获取最新分类树;不能从 md 文档硬编码分类。

### 状态确认

对照 `references/statuses.md` 确认状态（默认"在家"）。

**AI自动推断规则（最小实现）**：
- 位置路径包含"快递" → 自动设置状态为"快递中"
- 其他情况默认"在家"

AI 仍需询问用户确认最终状态，但可以提示推断依据。

### 图片解析

AI 从图片中提取以下信息作为 tags：
- 颜色（白色、黑色、蓝色、红色等）
- 品牌/厂商（海天、可口可乐、优衣库等）
- 规格容量（500ml、2L、大瓶、小瓶等）
- 材质/面料（纯棉、羊毛、丝绸、塑料等）
- 口味/香型（橙味、原味、薄荷等）
- 适用场景（运动、职场、户外、居家等）
- 季节（春夏、秋冬、四季）
- 新旧程度（全新、九成新、八成新等）

**原则**：能识别多少就提取多少，不要遗漏图片中明显可见的信息。

## Step 2.5: 位置智能推荐（强制动作）

**触发条件**：用户开始录入新物品（`录物品`/`拍物品`），在确定 category 之后、问 location 之前，**必须执行本步**。

**执行方式 A（推荐）—— CLI 调用**：

```bash
# 基础版：只显示位置+数量
python home_manager.py suggest-locations --category-id 138  # 衣物顶级

# 增强版：附带显示每个位置的代表物品（解决"记忆模糊"痛点）
python home_manager.py suggest-locations --category-id 138 --with-examples  # 衣物顶级
```

**执行方式 B（备选）—— 直接 Python 调用**：

```python
from home_manager.location_ops import suggest_locations_with_examples

results = suggest_locations_with_examples(conn, category, limit=10, examples_per_loc=2)
# 返回: [(location, count, [示例1, 示例2]), ...]
```

**展示给用户**（AI 复制 CLI 输出后整理）：

```
📍 同类物品位置推荐（衣物）—— 共 3 个位置：

  1. 卧室/衣柜/上层  [2件同类]
     └ 代表：黑色连帽卫衣、白色长袖T恤

  2. 卧室/抽屉       [1件同类]
     └ 代表：深蓝牛仔裤

  3. 客厅/衣帽架     [1件同类]
     └ 代表：灰色运动外套

  4. 其他位置（请直接告诉我新位置路径）
```

**用户选择**：
- 输入编号 → 沿用该位置
- 描述"和XX放一起" → 跳到 Step 2.6（参考物品锚定）
- 直接给新路径 → 沿用

**为什么强制**：用户记忆模糊时，看到"代表物品名"能立刻指认"对对就是那件"，把多步走变成一步确认。

## Step 2.6: 参考物品锚定（可选子流程）

**触发条件**：用户在 Step 2.5 中说"和XX放一起"、"参考XX"、"跟XX同位置"、"和XX一样"、"同XX位置"等。

**执行流程**：

1. **接收参考物品名**：AI 从用户输入中提取物品名（如"和那件黑卫衣放一起" → 参考="黑卫衣"）

2. **调用 CLI 定位**：
   ```bash
   python home_manager.py find-location --reference "黑卫衣"
   ```

3. **CLI 输出示例**：
   ```
   🔍 参考「黑卫衣」找到 2 件候选：
   ------------------------------------------------------------
     1. 黑色连帽卫衣  [ID:42]  分类:衣物
        └ 📍 卧室/衣柜/上层 ×1[在家]
     2. 黑色针织卫衣  [ID:67]  分类:衣物
        └ 📍 客厅/衣帽架 ×1[在家]
   ```

4. **多候选处理**（找到 ≥2 件时 AI 必须追问）：
   ```
   找到 2 件"黑卫衣"，你要参考的是哪件？
     1. 黑色连帽卫衣 → 卧室/衣柜/上层
     2. 黑色针织卫衣 → 客厅/衣帽架
   回复编号即可
   ```

5. **单候选处理**（只找到 1 件时直接确认）：
   ```
   找到 1 件"黑色连帽卫衣" → 📍 卧室/衣柜/上层
   就放这里？还是换个位置？
   ```

6. **没找到**：AI 询问用户：
   - 是否换个名字重试？
   - 是否要录入全新的位置？

7. **用户确认后** → 回到 Step 2.5 的用户选择流程，把确认的位置填入 `location` 字段。

**与 Step 2.5 的关系**：Step 2.6 是 Step 2.5 的"子流程"，完成后回到 Step 2.5。两者可以组合 —— Step 2.5 先展示推荐 → 用户说"参考XX" → Step 2.6 精确定位 → 回到 Step 2.5 填入位置。

## Step 3: 补全缺失字段

AI 一句话问完所有缺失信息：

```
收到，先记个草稿 📝

{name} ｜ {category}

再补充几点就好：
· 它放在哪儿？（**必须至少两级路径，如`客厅/冰箱`、`厨房/灶台`**，不允许只写"客厅"）
· 谁的？（不说就默认「使用者」）
· 数量有几件/包？（不说就默认 1）
· 花了多少钱？（没记就跳过）
· 什么时候买的？（大概日期就行，跳过也可以）
· 会过期吗？（食品/化妆品类才需要，写个大概日期）
· 有没有图片要发过来？（有图必须在 add 时通过 `--photo` 一次写入；没有说"无"即可）

> tag 字段走独立流程，见 Step 3.5。

一口气告诉我就行 👇
```

**原则**：缺什么问什么，不问已经给过的

**必需字段**：name / category / location 必须全部确认

**location 路径规则**：必须至少包含两级（如 `客厅/冰箱`，`厨房/灶台`），不允许单级位置。

**硬约束（脚本层强制，无任何跳过通道）**：
- **tag 数量 ≥ 10**：不足时 add 命令直接报错，错误信息"需要 tag 最少十个，备注不能为空且要全面"
- **remark 非空**：为空时同样报错
- 拍物品流程必须**先看图**再 add 一次到位（详见"图片录入子流程"）
- **无任何应急通道**：AI 偷懒无退路，必须按规范录入
- **tag 录入流程**：见 Step 3.5（按模板 3 阶段录入）

## Step 3.5: tag 录入（按模板逐项问）

AI 按**全局 tag 体系**录入(分类与 tag 正交,不再按分类查 tag 模板)。`references/categories.md` 仅作分类参考,不含 tag 模板。**按 3 阶段逐项询问**(必填 → 推荐含条件性判定 → 凑数)。

### 阶段 1：必填维度

按模板"必填维度"列依次询问，每项一问：

```
按 {category} 模板，必填 tag 有 N 个：
  · {维度 1}？→ {用户答}
  · {维度 2}？→ {用户答}
  · {维度 3}？→ {用户答}
  · {维度 4}？→ {用户答}
```

### 阶段 2：推荐维度（含条件性判定）

对每个推荐维度：

- **普通维度** → 直接问（如"材质？纯棉/羊毛/塑料？"）
- **条件性维度（`*`）** → AI 按"双重判定"（能否回答 + 有无意义）
  - 至少一条通过 → 询问
  - 两条都不通过 → 跳过，记录到"已跳过"清单（阶段 3 后展示）

AI 应主动告知"我会跳过 X 维度（理由）"——保持透明。

**常见条件性维度判定参考**：见本文 Step 3.5 上下文(不再引用 categories.md,该章节已废止)。

### 阶段 3：凑数（不足 10 个时）

**判定**：已询问 tag 数 + 通过的条件性维度数 ≥ 10？否 → 触发凑数。

**凑数策略**：

1. AI 推荐 2-3 个高频补充维度（来自通用 5 方向）：
   ```
   现在已收集 N 个 tag，距 10 个还差 M 个。
   推荐补充：
     · 状态？（如新旧、是否在用）
     · 使用场景？（如办公/居家/户外）
     · 使用者？（如自己/家人）
   回复这几个 + 自填其他都行。
   ```
2. 用户回复后仍不足 → 一次性问"还有什么想标记的？"
3. 用户说"没了" → 跳过，进入 Step 3.6 验收

### 拍物品流程差异

图片已识别部分 tag，**先展示再问缺失**：

```
图片识别到 N 个 tag：
  · 颜色：白色
  · 品牌：优衣库
  · 款式：短袖
  · ...
确认/修改这些，再按 {category} 模板问缺失维度。
```

AI 必须等用户确认图片识别的 tag 后，再进入阶段 1-3。

### 录入前展示（阶段 3 后）

完整 tag 清单（含跳过维度说明）：

```
即将录入的 tag（共 N 个）：

必填：
  · 颜色：白色
  · 季节：夏季
  · 款式：短袖
  · 品牌：优衣库

推荐（已确认）：
  · 材质：纯棉
  · 版型：修身

补充：
  · 状态：九成新
  · 场景：休闲

跳过维度（条件性不适用）：
  · 是否有机（深度加工食品）

OK 吗？回复"对"或修改。
```

**用户确认前不得写入数据库**。

## Step 3.6: tag 验收

AI 在 tag 录入完成、进入 Step 4 前自动跑一次验收：

- [ ] 必填维度齐全(全局 tag 体系,见 Step 3.5 阶段 1)
- [ ] 总数 ≥ 10（不计入跳过的条件性维度）
- [ ] 无重复 / 高度相似 tag（白 vs 白色 / T恤 vs T桖）
- [ ] 无空 tag（"" / None）

**验收通过** → 进入 Step 4 检查重复。

**验收失败** → AI 列出问题项，用户修正后重验：

```
tag 验收未通过：
  · 缺必填维度：型号
  · tag 重复："白色" 和 "白"
  · tag 总数 8 < 10
请修正后重试。
```

## Step 4: 检查重复（强制先查相似）

AI **录物品前必须先调用 search 查相似物品**，复用 `search` 命令的模糊匹配能力，不允许跳过本步直接 add。

**调用规范**：
- `search --name "{name关键词}" --category-id {N}`（同分类+名称模糊，**默认走这条**）
- `search --name "{name关键词}"`（无 category 时使用，覆盖更广）

**判定规则**（找到相似时 AI 必须列出让用户选择）：
- 已有"已用完"状态的同名物品 → 询问"补充库存"还是"新建"
- 已有"已废弃"状态的同名物品 → 询问"恢复使用（补充）"还是"新建"
- 同分类 + 名称相似/含相同关键词 → 询问"补到现有（update --plus N）"还是"新建"
- 仅名称含相同关键词但分类不同 → 列出供参考，**不强制**用户选择

**tag 相似度辅助判定**（新增）：
- name 相似 + tag 集合 Jaccard > 0.5 → 视为同款，提示"tag 相似度 X.X，仅写法差异"
- name 相似 + tag 集合相似度低 → 仅提示，不强制用户选择
- Jaccard = |A∩B| / |A∪B|（相同 tag 数 / 总 tag 数）

示例：

```
找到相似物品：

  ID 100 | 白色优衣库短袖 | 衣物
    └ tag: 白色, 短袖, 优衣库, 夏季, 纯棉

  ID 105 | 白色优衣库短袖 | 衣物
    └ tag: 白, T恤, UNIQLO, 春夏, 全棉
    ⚠️ tag 相似度 0.7（仅写法差异）

这是同一件吗？选择：
  1. 补充到现有（update --plus 1）
  2. 新建记录
```

**未找到相似时**：直接进入 Step 5 写入。

用户选择"补充"后，AI 询问新物品的状态："现在是在家、备用、还是快递中？"

**用户确认前不得写入数据库**

## Step 5: 确认并写入

AI 展示完整记录，用户说"对"后执行。若用户已提供图片，必须先把图片保存到 `HOME_PHOTOS_DIR`，并在同一次 `add` 中传入 `--photo`，不得先 add 再 update。

```bash
python home_manager.py add \
  --name "薯片" \
  --category-id 156 \  # 零食(加工食品)
  --location "厨房/上柜/左侧" \
  --quantity 3 \
  --tags "零食,乐事,原味,袋装,原切,零食柜,家庭库存,待食用,休闲场景,常温保存" \
  --remark "袋装薯片，放厨房上柜左侧，作为家庭零食库存，常温保存，开封后尽快食用" \
  --photo "$HOME_PHOTOS_DIR/待录入_薯片.jpg"
```

**注意**：当前脚本不支持多位置同时录入。如需多位置，请分多次录入。

## Step 7: 展示结果并回读确认

物品录入成功后，AI 展示脚本输出，并用 `detail --id {物品ID}` 回读确认。若本次已传 `--photo`，确认 `photo` 字段是否按 `YYYYMMDD_{ID}_{名称}.jpg` 写入；若本次未提供图片，可询问是否需要后续补图。

### 图片录入子流程（拍物品）

**与文字录入的差异**：拍物品必须**先看图**再 add，**一次 add 到位**（带完整 tags + photo），不再用"先 add 再 update"的两步法。

**原因**：add 命令现在有硬约束（tag ≥ 10 + 备注非空）。如果先 add 不带 photo，就拿不到图片信息，无法满足约束。所以**必须重构**为先看图。

**新流程**：

1. **AI 接收用户图片**（先看图，**不要先 add**）
2. **AI 视觉模型分析图片**，提取**最全面详细而严谨的 tag**：
   - 颜色（白/黑/蓝/红等）
   - 品牌/厂商（必须）
   - 规格/容量/尺寸（500ml/2L/27英寸等）
   - 材质/面料（纯棉/羊毛/塑料/金属等）
   - 口味/香型（食品/化妆品必须有）
   - 适用场景（运动/职场/户外等）
   - 季节（春夏/秋冬/四季）
   - 新旧程度（全新/九成新/八成新等）
   - 包装（袋装/罐装/瓶装/盒装）
   - 产地/购买渠道
3. **AI 从图片推断**：
   - 物品名称（用图片中最显眼的标识）
   - 分类（如无法确定，询问用户）
4. **AI 提示用户确认**完整信息：
   ```
   即将录入（拍图识别）：
   - 名称：XXX
   - 分类：XXX
   - 位置：待确认（请告诉我放哪）
   - tags：tag1,tag2,...（共 N 个，≥ 10）
   - 备注：图片中观察到的关键信息（生产日期/规格/材质/使用场景等）
   - 数量：1
   对吗？回复"对"或修改
   ```
5. **先把原图保存到 $HOME_PHOTOS_DIR 下的临时文件名**（如 `待录入_{名称}.jpg`）；`add --photo` 内部拿到 item_id 后会复制为 `YYYYMMDD_{ID}_{名称}.jpg` 并写入 `item.photo`
6. **一次性调用 add 命令**（满足所有硬约束）：
   ```bash
python home_manager.py add \
      --name "XXX" \
      --category-id N \  # 从 categories 表查
      --location "XXX" \
      --tags "tag1,tag2,tag3,...,tag10" \
     --remark "从图片观察到的关键信息..." \
     --photo "$HOME_PHOTOS_DIR/待录入_{名称}.jpg"
   ```

**无应急通道**：如果图片信息实在无法凑出 10 tag（如严重模糊/黑屏），只能：
- 让用户重拍一张更清晰的图
- 或改用"录物品"文字录入
- **不存在**任何跳过硬约束的途径
