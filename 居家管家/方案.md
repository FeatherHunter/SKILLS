# 居家管家 - 家庭物品管理系统

## 概述

帮助用户管理家庭所有物品的技能，支持：
- 录入物品信息（名称、分类、位置、标签、图片等）
- 通过位置或名称快速查找物品
- 追踪物品状态变化（位置变动、借用、维修等）
- 根据天气自动推荐今日穿搭
- 按需触发物品盘点
- 记录物品访问频率，区分高频/低频物品

## 数据库结构

### 表1：items（物品表）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER | 主键自增 |
| name | TEXT | 物品名称 |
| category | TEXT | 大类（见附录） |
| location | TEXT | 存放位置（路径格式，当前实际位置） |
| owner | TEXT | 所有者（默认"使用者"） |
| status | TEXT | 状态（参考状态值完整列表） |
| quantity | INTEGER | 数量（默认1，如10双袜子） |
| purchase_price | REAL | 购买价格（可空） |
| remark | TEXT | 备注 |
| photo | TEXT | 图片路径 |
| access_count | INTEGER DEFAULT 0 | 被访问次数（查询/移动/更新状态等操作+1） |
| last_accessed_at | TIMESTAMP | 最后访问时间 |
| created_at | TIMESTAMP | 创建时间 |
| updated_at | TIMESTAMP | 更新时间 |

**说明：**
- location = 物品当前实际位置，不是"通常在哪"
- 物品长期放公司 → 直接更新 location 为公司地址
- access_count + last_accessed_at 用于区分高频/低频物品

### 表2：item_tags（标签表）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER | 主键自增 |
| item_id | INTEGER | 关联 items.id（UNIQUE约束） |
| tag | TEXT | 单个标签 |

**说明：**
- 多个物品可以有同一个tag（如：多件白色衣服）
- 同物品不能重复添加同一个tag（通过UNIQUE约束实现）
- 搜索用精确匹配

**合并同类项（方案C）：**
- 录入时如果发现相似tag，提示你："已有相似标签'白色'，要用'白色'还是继续用'白'？"
- 定期整理时，把所有"白"合并成"白色"（需你确认后执行）
- 判断相似：两个tag字面高度重叠（如"白"和"白色"、"短袖T"和"短袖T恤"），或你明确说两个tag是一回事

### 表3：locations（位置历史表）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER | 主键自增 |
| location_path | TEXT UNIQUE | 路径，如"客厅/东北角落/桌子上" |
| use_count | INTEGER DEFAULT 1 | 使用次数 |
| last_used | TIMESTAMP | 最后使用时间 |

**说明：**
- 录入新位置时自动写入此表
- 录入时我查此表给你智能提示你用过的位置
- 位置格式可变长，如：`客厅/东北角落/桌子上/从天花板到下第3个箱子`

## 状态值完整列表

| status | 含义 |
|--------|------|
| 在家 | 在固定位置 |
| 备用 | 备用库存 |
| 穿着中 | 正在身上用（如衣服穿身上） |
| 旅游中 | 带出去旅游了 |
| 洗护中 | 在洗衣机/晾衣架/烘干机 |
| 借用中 | 借给朋友了 |
| 维修中 | 坏了在修 |
| 已用完 | 消耗品用完了 |
| 待处理 | 需要清理或处理 |
| 已废弃 | 废弃了 |

**状态流转示例（衣物）：**
```
在家（衣柜）→ 穿着中/旅游中（出门）→ 洗护中（脏衣服）→ 在家（洗干净放回）
```

## 数据库初始化

首次使用时，需检查并创建数据库和表：

### 建表流程
1. 检查数据库文件是否存在
2. 不存在 → 创建数据库文件
3. 检查三张表是否存在（items / item_tags / locations）
4. 不存在 → 执行建表语句
5. 检查索引是否存在 → 不存在 → 创建索引
6. 初始化完成，确认后进入正常流程

### 建表 SQL

```sql
CREATE TABLE IF NOT EXISTS items (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,
    category TEXT NOT NULL,
    location TEXT NOT NULL,
    owner TEXT DEFAULT '使用者',
    status TEXT DEFAULT '在家',
    quantity INTEGER DEFAULT 1,
    purchase_price REAL,
    remark TEXT,
    photo TEXT,
    access_count INTEGER DEFAULT 0,
    last_accessed_at TIMESTAMP,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE IF NOT EXISTS item_tags (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    item_id INTEGER NOT NULL,
    tag TEXT NOT NULL,
    UNIQUE(item_id, tag)
);

CREATE TABLE IF NOT EXISTS locations (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    location_path TEXT UNIQUE NOT NULL,
    use_count INTEGER DEFAULT 1,
    last_used TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 管控原则

- 我记录你用过的所有位置（存入 locations 表）
- 路径可长可短，完全按你说得写

## 功能模块

### 1. 物品录入

**触发词：** "居家管家"

**流程：**
1. 你说"居家管家"启动技能
2. 用正常语言描述物品，我解析
3. 交互式补充信息（名称→位置→标签→图片）
4. 你确认后写入数据库
5. 同时更新 locations 表

**必需字段：** name + category + location

### 2. 物品查找

**触发词：** "找XX"、"XX在哪"、"有没有XX"

**查询方式：**
- 按名称精确/模糊搜索
- 按位置搜索（支持模糊路径）
- 按标签搜索（支持多标签组合）
- 按分类搜索
- 按状态搜索（如找所有"穿着中"的物品）
- 组合搜索

**返回：** 物品列表 + 位置 + 标签 + 状态

**搜索优先级：**
1. name 精确匹配
2. name 模糊匹配
3. tags 匹配
4. category 匹配

**多物品冲突时：** 必须用户确认，不默认选任何一个

### 3. 物品位置更新

**触发词：** "XX换位置了"、"把XX放到XX"

**操作：** 更新 location + 写入新位置到 locations 表

**注意：** 物品长期放公司/外地 → 直接更新 location，不保留旧位置

### 4. 物品状态更新

**核心原则：物品只增不删，可修改**
- 物品建立后不会物理删除，数量最小为0
- quantity=0 → 自动标记为"已用完"
- "已废弃"需用户明确表态（可能quantity>0但已废弃，如过期食品）
- 物理删除会导致数据不一致，禁止执行

**状态值：** 在家 / 备用 / 穿着中 / 旅游中 / 洗护中 / 借用中 / 维修中 / 已用完 / 待处理 / 已废弃

**更新范围：** 可同时更新多个字段（位置/状态/数量/tags/备注等）

**确认机制：** 任何操作前必须确认（见下方"交互确认原则"）

### 5. 今日穿搭推荐（子技能）

**触发词：** "今天穿什么"、"推荐穿搭"

**流程：**
1. 获取今日天气预报
2. 根据温度+天气筛选衣物（通过 tags 匹配）
3. 智能搭配并展示推荐
4. 用户自行决定是否穿着，手动修改状态（不自动改状态）

### 6. 物品盘点

**触发词：** "盘点衣柜"、"整理一下XX"

**流程：**
1. 你指定要盘点的位置
2. 我列出该位置所有物品
3. 你逐个确认物品状态是否正确
4. 你说"XX变了" → 更新物品信息
5. 盘点完成

### 7. 旅游归位流程（扩展功能）

**出门前：**
你说"居家管家 我要带XX出去旅游"
→ 我搜索"XX" → 你确认是哪些物品
→ 确认后这些物品 status 改为"旅游中"

**回家后：**
你说"居家管家 旅游回来了"
→ 我查所有 status="旅游中" 的物品
→ 逐个问你："这东西放回原位还是换位置？"
→ 确认后 status 改回"在家"，location 更新

## 边界情况与错误处理

所有边界情况通过**交互式采访用户**解决，不自行判断：

**搜索相关：**
- 搜不到物品 → 告知用户，询问是否换个关键词或新建物品
- 搜到多个同名物品 → 列出让用户选择，不默认选任何一个
- 用户输入模糊无法解析 → 追问确认

**状态相关：**
- quantity=0 → 自动标记为"已用完"，告知用户
- quantity>0 但用户说废弃 → 按"已废弃"处理，quantity保持原值
- 状态变更可能矛盾时 → 告知用户当前状态，请用户确认

**数据库相关：**
- 写入失败 → 告知用户失败原因，询问重试或跳过
- 数据库文件被锁 → 告知用户，等待后重试
- 表/字段不存在 → 触发数据库初始化流程

**盘点相关：**
- 盘点中途退出 → 已确认的修改已写入，未确认的保持原样，下次继续从退出处开始
- 盘点发现空位置 → 告知用户该位置没有物品

**穿搭推荐相关：**
- 筛选结果为空 → 告知用户无匹配衣物，建议扩展标签或录入新衣物
- 天气数据获取失败 → 告知用户无法获取天气，改用纯标签筛选展示

## 交互确认原则

**任何操作前必须确认，包括但不限于：**
- 是哪个物品（多物品时列出让你选）
- 数量多少（不是默认全量）
- 改成什么状态/位置
- 你说"对"我才执行，不对就修正

**确认流程示例：**
```
你说：我要把白T恤借给朋友
我：搜到2件白T恤，请确认是哪件？
你说：第2个
我：确认把"白T恤（优衣库/M码）"改为"借用中"吗？
你说：对
我：执行更新，已记录
```

## 访问频率记录

### 什么算"访问"
- 搜索/查询这个物品 ✅
- 移动了位置 ✅
- 更新了状态 ✅
- 更新了标签/备注 ✅
- 被穿搭推荐选中 ✅
- 刚录入（新增）❌ 不算

### 记录方式
- access_count += 1
- last_accessed_at = 当前时间

### 应用场景
- "找高频物品" → 按 access_count 排序
- "找长期没碰的东西" → 按 last_accessed_at 筛选

## 注意事项

1. **录入成本**：先录入"经常找不到的"物品，逐步完善，不必一次全部录入
2. **位置描述统一**：用 locations 表记录用过的位置，按使用频率提示统一
3. **数量问题**：quantity=10 表示10个同类物品
4. **隐私安全**：物品数据较私密，建议本地存储，不上传云端
5. **owner字段**：默认"使用者"，可区分"使用者/女朋友/公共"
6. **标签灵活**：tags 不预设，你说什么我存什么，支持任意多个标签
7. **标签合并**：发现相似标签时提示你合并，保持一致性
8. **盘点按需**：不定时，不自动提醒，换季时按需触发
9. **物品只增不删，可修改**：物品建立后只增和修改，不做物理删除，quantity最小为0
10. **location = 实际位置**：物品长期放哪就更新到哪
