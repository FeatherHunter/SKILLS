P0 7 个集成点散落,未抽 integrations/

现象
实测 7 处外部集成散落在 8 个脚本里,无统一抽象层。
根因
智剪工坊 v1.0 起没有 integrations/ 概念,所有"调外部"的代码直接写在业务脚本(cover.py / transcribe.py / separate.py / download_whisper_model.py 等)。
影响
① 外部 API 升级时所有脚本都要改;② 失败降级逻辑重复实现 7 次(且不一致);③ 测试时无法 mock 单一集成点;④ 单元测试无法独立跑业务逻辑。
修法
新建 lib/integrations/{matrix,huggingface,model_download}.py 三个模块,提供统一接口(call_matrix_with_retry() / call_hf_with_fallback() / download_with_mirror()),然后改 8 个脚本只 import 这些函数。
工作量
2-3h(抽 3 个集成模块)+ 1-2h(改 8 个脚本调用方)
7 个集成点完整清单

P0 CLI 输出未统一 JSON 三段式

现象
90+ 脚本中,只有 cover_compose/cli.py 输出 {status, data, message, warnings} JSON;其余 mix.py / trim.py / transcribe.py / beauty.py / batch.py 等都是 log_* 文本 + sys.exit(1)。
根因
v1.5/v1.6 设计 CLI 时规范尚未强制 JSON 输出;safe_run 装饰器统一了"出错退出码 + 错误信息",但没统一"成功 JSON 输出"。
影响
① AI(agent) 调用 CLI 后只能 parse stdout 文本,出错时无法 machine-readable 区分 ok / warn / error;② 难以做"CI 验证全部脚本通过/失败";③ 第三方工具(如 Python API 调用)拿不到结构化结果。
修法
抽 lib/json_output.py,提供 emit_ok(data, message) / emit_warn(data, warnings, message) / emit_error(message, code) 三个函数;所有 CLI 的 main() 末尾改调 emit_*,而不是 print/log。
工作量
3-4h(写 lib + 改 30+ 脚本 main 函数 + 保留 log 输出作为 stderr 可选)
JSON 三段式样例 
# 成功
{"status": "ok", "data": {"output": "out.mp4", "duration": 27.3}, "message": "BGM 混音完成"}

# 警告(部分降级)
{"status": "warn", "data": {"output": "out.mp4"}, "warnings": ["HF token 缺失,跳过 speaker-diarization"], "message": "转录完成,无说话人分离"}

# 失败
{"status": "error", "data": {}, "message": "ffmpeg 退出码 1: Stream map '' matches no streams", "code": "FFMPEG_STREAM_MAP_ERROR"}

P1 规则层 validators.py 不集中

现象
唯一集中的 validators.py 在 scripts/ai/cover_compose/validators.py(103 行,5 类硬规则);其余 30+ 脚本的硬规则散落在各脚本内部函数里。
根因
v1.5 写 CLI 时为快速落地,把"硬规则校验"放在业务函数内联实现;v1.7 cover_compose 才抽出独立的 validators.py(局部最佳实践,未全局推广)。
影响
① 改一条硬规则要扫所有脚本(改 N 处);② 软跳过通道不一致——有的用 match_mode='ask'(mix.py),有的用 --force(违规),有的默默 fallback;③ 错误信息格式不统一——有的含字段名+当前值+期望值+怎么修,有的只说"参数错"。
修法
① 抽 lib/validators.py 顶层,放通用校验器(路径存在 / 时间格式 / 数值范围 / 枚举白名单);② 把 cover_compose/validators.py 迁移/重写到 lib/validators_cover.py;③ 各脚本 import 顶层校验器,删内联校验。
工作量
4-6h(抽顶层 5-8 个通用校验器 + 改 30+ 脚本调用 + 加单测)

脚本硬规则位置当前实现

P1 对外写操作无幂等键 client_request_id

现象
规范要求"对外写操作必带 client_request_id,避免重试产生重复"。实测所有调 matrix / HuggingFace 的脚本(8 处)都没带幂等键。
根因
v1.0 设计阶段重点在"本地 ffmpeg 调用",对"远端写操作幂等性"的认知来自《优秀 Skill 指导手册》v1.7+,本 Skill 的对应代码没同步更新。
影响
① AI 重跑某一步(典型场景:网络抖动 + safe_run 重试)→ matrix 生成重复封面 → 用户白花 30s 等待 + 重复费用;② pyannote 重跑 → HF 计费翻倍;③ 无去重机制,数据库/对象存储里只能手动清理。
修法
在 lib/integrations/matrix.py 加 call_with_idempotency(tool, args, client_request_id) 函数;client_request_id 格式 {date}_{task_slug}_{step}_{uuid8}——和 SKILL.md §5.2 现有备份命名规则一致。
工作量
2-3h(改 5 个 matrix 调用脚本 + 加 client_request_id 透传)
幂等键设计样例 
# 现状(无幂等)
subprocess.run(["mavis", "mcp", "call", "matrix",
                "matrix_generate_image",
                "--file", req_file])

# 修后(带幂等)
from integrations.matrix import call_with_idempotency
import uuid
client_request_id = f"{date}_{task_slug}_{step}_{uuid.uuid4().hex[:8]}"
result = call_with_idempotency(
    "matrix_generate_image",
    payload=req,
    client_request_id=client_request_id,
    ttl=86400,  # 24h 内同 key 复用结果
)

P2 SKILL.md 头部缺版本锚点

现象
SKILL.md 头部 YAML frontmatter 只有 name / description / triggers / metadata: { openclaw: { emoji, requires } },没有 version 字段。当前真实版本是 v1.20.2(根据 .archive/CHANGELOG.md),但头部无法让 AI 立即识别。
根因
v1.0 设计时把 changelog 放在 .archive/CHANGELOG.md(开发者面向),SKILL.md 头部只对 AI 暴露"能做什么"(description + triggers),没暴露"我现在是哪一版"。
影响
① AI 加载 SKILL.md 时不知道"我读到的是 v1.20 还是有 v1.21+ 已合并";② 出 bug 时 AI 无法快速对账"这是不是已知 bug,有没有 fix 版本";③ 与 智剪工坊-意图编辑.html 的 UI 版本号脱钩,无法做兼容检测。
修法
SKILL.md frontmatter 加 3 行: 
version: "v1.20.2"
updated: "2026-07-20"
breaking_changes: ["v1.20 项目 title 字段必填"]
并在文件末尾加"当前版本说明"小节(类似 v1.20 新增了什么 / 修了什么 BUG)。
工作量
0.5h(改 SKILL.md frontmatter + 末尾加版本说明)

P2 intent.json 多版无 migration 工具

现象
intent.json 历经 v0.5 / v1.0 / v1.2 / v1.3 / v1.4 / v1.6 / v1.7 / v1.10 / v1.12 / v1.20 多版,没有 migration 工具。SKILL.md 也没说明"老 intent.json 是否兼容"。
根因
规范"改 schema → migration 脚本,不直接 ALTER 跑生产"——但视频领域 schema 不在数据库,而在 智剪工坊-意图编辑.html 的表单字段。HTML 字段新增是"用户重新填表"实现软迁移,无显式 migration 脚本。
影响
① 老 intent.json 缺失新必填字段(如 v1.20 的 project.title)→ AI 不知是缺失还是正常;② v1.10 新增 sequences[].title,老 intent 没该字段→ AI 必须"推断默认值";③ 用户切不同电脑加载老 intent 时行为不一致。
修法
写 tools/migrate_intent.py,支持 --from v1.0 --to v1.20 增量迁移: 
python tools/migrate_intent.py \
    --input intent_v1.json \
    --output intent_v120.json \
    --from v1.0 --to v1.20
迁移日志写入 intent_v1_migration_log.md(含"添加了哪些字段/默认值是什么")。
工作量
3-4h(写工具 + 列出 5 个关键版本差异 + 加 dry-run 模式)

P3 触发词表与 scripts/ 对账存在链路断裂

根因
SKILL.md v1.19 时确实未实现,后来补了 CLI,但 SKILL.md 头部 triggers 表未更新;AI 路由表 §AI 路由表-意图JSON字段枚举.md L72 也未更新。

修复顺序建议 · 从 P0 到 P3

按"价值/工作量"比排序,推荐 4 个 sprint 完成:

  1. Sprint 1(2-3h) — 抽 lib/integrations/{matrix,huggingface,model_download}.py(P0-1)。修后所有外部调用统一接口,失败降级逻辑统一。
  2. Sprint 2(3-4h) — 抽 lib/json_output.py(P0-2)。修后所有 CLI 输出 JSON,AI 可结构化解析。
  3. Sprint 3(4-6h) — 抽 lib/validators.py 顶层 + 迁移 cover_compose/validators.py(P1-3)。修后改硬规则只改 1 处。
  4. Sprint 4(2-3h) — 给 matrix/HF 调用加 client_request_id(P1-4),避免重试产生重复。
  5. Sprint 5(3-4h,可选) — 写 tools/migrate_intent.py(P2-6)+ SKILL.md 加 version 锚点(P2-5)+ 触发词对账(P3-7)。

最低合规线:Sprint 1 + 2(P0 全做)即可让智剪工坊达到"大型 5 层 Skill"的形式合规。Sprint 3-5 是锦上添花。
