# 案例 04 · Skill HTML 改造 · 架构师原则

> 不写 case study · 只写判据和规则
> 适用：任何想把 print-only CLI 改造为 HTML 模板的 Skill

---

## 原则 0 · 何时**不**该做

| 类型 | 决策 |
|---|---|
| 单条 CRUD（add/update/delete） | ❌ 不做 |
| 高频但无视觉（快速回执） | ❌ 不做 |
| 后台管理（需权限/事务） | ❌ 不适合 HTML |
| 数据采集（数据即产物） | ❌ HTML 没价值 |

**判据**：唤醒词里 **≥ 5 个**含"多字段 + 多条目 + 视觉密度"，才值得做。

---
## 原则 9 · 模板命名不要带版本号

```
❌ review_template_v2.html       ← 暗示有 v1 和 v3 共存
❌ dashboard_v3_apple.html       ← 暗示有 v1/v2/dark 共存
✅ review_report.html            ← 一个功能 = 一个文件
✅ health_dashboard.html         ← 一个功能 = 一个文件
```

**为什么**：
- 同一个功能只有一个文件可演进
- 版本号暗示"多个并存版本"，实际应"迭代就替换，不是迭代就并存"
- AI 引用时减少歧义（不用考虑该用哪个版本）

**例外**：如果真的有 N 份并行方案（实验性），改名前缀叫 `experiment_xxx.html` 而不是 `xxx_v2.html`。

## 原则 1 · CLI 阻塞（最高优先级）

**CLI 不返回结构化 JSON → 模板一定做不出来。**

```python
def xxx(start, end, as_dict=False):  # ← 关键参数
    result = ...
    if as_dict:
        return {"status": "ok", "data": result, "message": "..."}
    print(...)  # ← 默认行为不变（向后兼容）
```

**铁律**：CLI 重构**永远**保持默认行为不变，否则破坏现有调用。

---

## 原则 2 · 1 模板多参数（N 唤醒词共用）

N 个数据结构相同的唤醒词 → **1 模板 + N 参数**（不是 N 模板）。

满足 3 条件才用 1 模板：
- ✅ X 唤醒词数据 schema 一致
- ✅ X 唤醒词展示区域结构一致
- ✅ X ≤ 5（再多拆）

正例：`food_ranking.html` 服务 5 个榜单。

---

## 原则 3 · 模板四段式

```
首屏 → 标题 + 状态徽章 + 4-6 KPI 卡     (5 秒内定位现状)
主体 → 按维度分组的折叠块              (扫读而非滚动)
交互 → 轻 JS（折叠/勾选/筛选）         (不引框架)
尾部 → 复制回 AI + 回到顶部            (用户主动切回对话)
```

5 状态必有：正常 / 空 / 缺数据 / 错误 / 离线。

---

## 原则 4 · 占位符 + 注入（5 个必须）

```html
<!--INJECT-DATA-->    ← 1. 恰好 1 次（注入器会校验）
<script>
window.__DATA__ = { /* 注入 */ };
</script>
```

| # | 必须 | 原因 |
|---|---|---|
| 1 | 占位符唯一 | 注入器可校验 |
| 2 | `</` 转义 `'<\\/'` | 防提前闭合 script |
| 3 | `typeof window.__DATA__ === 'object'` 守卫 | 防未注入白屏 |
| 4 | 5 状态 fallback | 防数据缺失空指针 |
| 5 | `escapeHTML()` 函数 | 防 XSS |

---

## 原则 5 · 渲染器 4 步（统一模式）

```python
# 1. 拿 JSON（subprocess 列表传参，避开 PowerShell 乱码）
data = json.loads(subprocess.run(['python', cli] + args, capture_output=True, text=True).stdout)

# 2. 读模板 + assert 占位符唯一
template = read_text('templates/xxx.html')
assert template.count('<!--INJECT-DATA-->') == 1

# 3. 注入（转义 + 替换）
payload = json.dumps(data, ensure_ascii=False).replace('</', '<\\/')
output = template.replace('<!--INJECT-DATA-->', f'<script>window.__DATA__ = {payload};</script>', 1)

# 4. 写副本（原模板不动）
write_text(output_path, output)
```

4 不：不用 `os.system`、不省 assert、不改原模板、不字符串拼路径。

## 原则 6 · 改动前必答 3 问

1. **影响哪些文件**？ 代码 / 文档 / DB schema / HTML / changelog
2. **有没有数据迁移**？ 改 schema 写 migration 脚本，不直接 ALTER
3. **回滚方案**？ `.bak` + git commit + `revert`

**答不上来 = 改动还没准备好。**

## 原则 7 · Commit 与检索

| 规则 | 说明 |
|---|---|
| 每 Phase 一个 commit | 失败可独立 `git revert` |
| commit 前缀 `[技能名]` | 让 `git log --grep` 精确搜 |
| 内容含：文件清单 + 行为变化 + 向后兼容 | 便于未来 reviewer |
| 不进 git：测试 HTML / 临时方案 / 备份目录 | `.gitignore` 排除 |

---

## 原则 8 · 7 反模式（不要做）

| ❌ 反模式 | ✅ 该怎么做 |
|---|---|
| `--force` 跳过校验 | 设计成"无跳过通道" |
| HTML 直接查数据库 | CLI 出 JSON → 注入 |
| `except: pass` 静默失败 | stderr + 非 0 退出码 |
| 模板硬拼 `<div>{name}</div>` | 占位符 + 注入器 |
| 每 Skill 自己维护 CSS | 共享 `_shared.css` |
| 每模板自己 escapeHTML | 共享 `_injector.py` |
| 炫技动画（3D / 粒子） | 克制 — 用户要数据 |

## 决策矩阵：CLI 输出 → 模板类型

| 输出类型 | 唤醒词示例 | 模板类型 |
|---|---|---|
| 单条记录 | 记吃了 / 记运动 | ❌ 不做 |
| 当日列表 | 查今天吃 | ✅ 卡 + 进度 |
| 多天趋势 | 查热量历史 | ✅ 折线 + 数据点 |
| TOP N 榜 | 5 食物排行 | ✅ 1 模板 + N 参数 |
| 完整性检查 | 扫禁忌 | ✅ 状态徽章 |
| 多维综合 | 健康报告 | ✅ 8 dim 卡片 |
| 计划 vs 实际 | 复盘训练 | ✅ 热力图 |

## 决策树：1 模板 vs N 模板

```
N ≥ 2 唤醒词共用同一份 HTML？
├─ YES → schema 一致？
│   ├─ YES → 1 模板 + N 参数 ✅
│   └─ NO  → N 模板 + 公共 CSS
└─ NO  → 单模板即可
```

## 原则 10 · 过程型 HTML 的 AI 协同模式（⭐ 最关键）

### 第一性原理

```
Layer 0 事实:
  ① LLM = HTTP API(远程服务)
  ② HTML = 浏览器渲染的文件(无服务端能力)
  ③ 当前架构 = 没有服务端中转 + 只有 IDE 进程内 agent

推导链:
  ① + ② + ③
    ↓
  HTML 与 LLM 之间没有通路
    ↓
  唯一路径: 用户手动承上启下
              HTML → 复制 → 粘贴 → AI 执行
    ↓
  设计约束: 所有过程型 HTML 都必须设计为"复制模板"
```

**结论**：HTML 是**单工设备**，永远是单向渲染。它不能反向触发 LLM，也没有任何"内部按钮"代替用户把数据送回 AI。

---

### 设计推论

| 推论 | 应用 |
|---|---|
| 任何"过程型"HTML 必须提供**复制按钮** | 用户的选择/操作必须有"出口"回到 AI |
| 复制内容含 **4 部分** | 让 AI 有上下文(场景+数据+动作+来源) |
| "采纳"按钮也走复制路径（短确认） | 保持一致性,不让用户困惑 |
| 永远不期望"用户不去操作就生效" | 现实约束,设计必须围绕这个 |

---

### 判定（什么时候必须有"复制 prompt"）

```
IF HTML 包含让用户做出选择 OR 操作结果的展示
   THEN 必须设计"复制 prompt"按钮 + "采纳"按钮
   ELSE 不需要(纯结果型即可)
```

---

### 4 部分 prompt 模板

```
① 场景: <用户在 HTML 中做了什么>
② 数据: <用户看到的最终数据(修改后)>
③ 期望: <AI 应该执行什么 CLI 操作>
④ 来源: <HTML 数据来自哪个 CLI / 时间>
```

**示例**（扫禁忌修改计划）：

> ① 我扫描了禁忌,需要修改健身计划
> ② 原动作: 杠铃硬拉 → 替换为: 支撑式悍马划船（理由: 腰椎禁忌）
> ③ 请执行: `python scripts/plan_generator.py update-action --week 1 --day 1 --old "杠铃硬拉" --new "支撑式悍马划船"`
> ④ 数据来源: 禁忌扫描 2026-07-23 / plan v14.4

---

### 反模式（HTML 自己做事）

| ❌ 反模式 | 失败原因 |
|---|---|
| "一键保存" / "一键替换" / "一键执行" | 用户点了 → HTML 调不到任何后端 → 不响应/报错 |
| HTML 自己 `fetch()` AI API | 暴露 API key + 跨域 + 静态托管不可写 |
| HTML 调本地服务 localhost | 不存在的服务,用户机器配置不可控 |
| 隐藏"复制"按钮 | 用户不知道有这一步,流程断在 HTML |

---

### 设计 checklist（动手前 6/8 通过）

- [ ] 该 HTML 是否包含"用户操作"?
- [ ] 如果"否",单纯展示就行(结果型)
- [ ] 如果"是",必有 2 个按钮:"采纳" + "复制修改指令"
- [ ] 复制内容含 4 部分(场景/数据/期望/来源)?
- [ ] "采纳"按钮的复制文本是否够清晰让 AI 知道下一步?
- [ ] 错误态:无修改时复制按钮仍存在吗?
- [ ] 复制 prompt 是否含完整 CLI 命令(让 AI 不需要再问路径)?
- [ ] 用户测试:模拟"复制+粘贴"链路,AI 能正确执行吗?
