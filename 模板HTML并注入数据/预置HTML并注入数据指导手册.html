Universal Pattern

预置 HTML + 注入数据指导手册

一套适用于任何 Skill 的轻量视觉增强方法：把稳定的交互和视觉预先写进 HTML 模板，把每次变化的数据用 JSON 注入进去，让简单功能快速拥有漂亮、可交互、可复用的页面。

适用：查看详情页、操作向导、做事模式、清单页、报告页、复盘页。原则：模板稳定，数据流动；样式预置，内容注入。

1. 为什么要这样做

很多 Skill 的核心能力已经有 CLI 或脚本，但纯文字输出不适合展示复杂信息：用户要反复扫读、对比、点击、计时、勾选或复盘。预置 HTML + 注入数据的本质，是把“交互体验”从每次回答里抽出来，变成稳定资产。

不再每次临时拼页面
视觉、布局、JS 交互都沉淀在模板里。
数据仍然走 Skill 规则
数据来自 CLI / JSON，不让 HTML 直连数据库。
修改成本低
改视觉只改模板；改数据只改 JSON 契约。
适合快速增强
不用做完整前端工程，也能有可靠页面。

2. 第一性原理

HTML 页面不是数据源，也不是业务规则层。它只是用户和数据之间的“可视化操作界面”。
问题答案
谁负责真实数据？Skill 的数据层 / CLI / JSON 输出。
谁负责业务规则？CLI、validators、数据库约束；HTML 只做展示层辅助校验。
谁负责体验？预置 HTML 模板：布局、样式、交互、空状态、错误态。
什么应该变化？注入的 JSON 数据。
什么应该稳定？模板结构、CSS、JS、字段契约、错误处理。

所以正确方向是：先定义数据契约再写稳定模板最后注入 JSON。

3. 什么时候适合使用

适合不适合
详情页：食谱、物品、任务、书籍、训练记录需要多人实时协作的复杂 Web App
向导页：做菜模式、安装流程、检查流程需要后端鉴权、实时数据库写入的页面
清单页：采购清单、待办清单、行李清单高频数据编辑且需要强一致性的后台管理系统
报告页：统计、复盘、审计结果大量数据分页检索、权限管理复杂的系统

5. 三种实现模式

模式 A：Jinja2 渲染器

适合详情页、报告页、采购清单这类“生成后基本静态”的页面。

CLI JSON → render.py → Jinja2 template.render(data=data) → output.html

模式 B：静态 HTML + window 注入

适合做菜模式、步骤向导、离线交互页。模板不依赖 Jinja2，直接读取 window.__DATA__。

<script>
window.__DATA__ = {...完整 JSON...};
window.__ENTITY_ID__ = "...";
</script>

模式 C：HTML 内嵌 JSON Script

适合数据很大、想避免全局变量污染的页面。

<script id="payload" type="application/json">
{"status":"success","data":{}}
</script>
<script>
const DATA = JSON.parse(document.getElementById('payload').textContent);
</script>

6. 数据契约

数据契约是整个模式的核心。模板只认识契约，不应该猜字段。

{
  "status": "success",
  "data": {
    "entity": { "id": "...", "name": "..." },
    "sections": [],
    "summary": {},
    "items": []
  },
  "message": "已生成"
}

契约规则

  • 顶层建议统一为 {status, data, message}。
  • 实体主键必须稳定：id 不要用自然语言当 key。
  • 数组字段即使为空也返回 []，不要省略。
  • 可空字段显式 null，不要让模板靠 undefined 猜。
  • 时间、金额、单位字段要拆开，不要只给一段自然语言。

7. 模板设计规范

首屏
标题、摘要、关键状态、主要操作按钮。
主体
卡片化 section，每个数据块有明确标题。
交互
折叠、勾选、计时、筛选、份数调整等轻 JS。
尾部
复制回执、下一步动作、保存提示。

必须有的状态

  • 正常态：数据完整。
  • 空态：列表为空时给解释。
  • 缺数据态：关键字段缺失时清晰提示。
  • 错误态：没有注入数据、JSON 解析失败。
  • 离线态：尽量不依赖外部网络。

8. 注入方式

推荐做法：生成临时副本，不改原模板

template = read("templates/view.html")
payload = json.dumps(data, ensure_ascii=False).replace("</", "<\\/")
inject = "<script>window.__DATA__ = " + payload + ";</script>"
html = template.replace("<body>", "<body>\n" + inject, 1)
write("output/view_xxx.html", html)

模板端读取

const PAYLOAD = window.__DATA__;
if (!PAYLOAD) {
  showError('数据未注入');
} else {
  init(PAYLOAD.data || PAYLOAD);
}
为什么不建议直接改原模板？

原模板应该是稳定资产。如果每次把数据写进原模板，下一次生成会污染模板，也容易把用户隐私数据提交进仓库。正确做法是：原模板只放结构和逻辑，输出文件才包含注入数据。

9. 标准工作流

  1. 识别用户意图：这是查看、清单、向导还是报告。
  2. 调用 Skill CLI 获取 JSON，禁止 HTML 直连数据库。
  3. 校验 JSON 是否满足模板契约。
  4. 读取预置 HTML 模板。
  5. 把 JSON 注入模板，生成输出 HTML 文件。
  6. 用浏览器或静态检查验证关键字段出现。
  7. 用 <media src="..." type="file" /> 交付。
  8. 如果页面会让用户复制结果回来，定义复制文本格式和保存 CLI。

10. 推荐文件结构

某个Skill/
  SKILL.md
  templates/
    xxx_view.html          # 预置模板
  scripts/
    xxx_manager.py         # 业务 CLI / JSON 输出
    xxx_render.py          # 可选：Jinja2 渲染器
  references/
    xxx_html_mode.md       # 调用规范、注入方式、错误处理
  CHANGELOG.md

如果功能很小，可以不写 render 脚本，直接采用“静态模板 + AI 注入”。如果会频繁复用，就应该沉淀为 scripts/*_render.py。

11. 安全与容错

风险处理方式
XSSJinja2 开 autoescape；JS 注入时用 JSON 序列化，不拼用户原文；注入到 script 时额外转义 </script> 断标签风险。
模板污染只生成副本，不把注入数据写回模板。
字段缺失模板提供错误页；CLI 提前校验契约。
外部资源失效优先单文件、系统字体、本地运行。
隐私泄露输出文件放临时目录或明确输出目录，不自动提交。
中文乱码全链路 UTF-8；Windows PowerShell 中文参数用 Python subprocess 列表传参。

12. 对抗式审查清单

  • 模板是否能在没有数据时显示明确错误，而不是白屏？
  • 数据是否来自 CLI / JSON，而不是 HTML 自己查库？
  • 是否污染了原始模板文件？
  • 字段名是否稳定，有没有今天叫 total_time 明天叫 totalTime？
  • 数组为空时页面是否还能正常显示？
  • 用户输入是否通过 JSON 序列化或 autoescape 进入页面？
  • 输出路径是否明确，是否尊重环境变量？
  • 是否有复制回 AI 的回执格式？
  • SKILL.md、HTML 镜像、references、CHANGELOG 是否同步？
  • 手机、平板、桌面是否都能读、能点、不卡？

13. AI 头脑风暴扩展

不要只把用户说的字段搬进 HTML。AI 必须先看 CLI 能拿到的全部数据，再从用户真实使用场景出发，提出能让页面更舒服、更强大的功能建议。

核心要求

  • 先盘点 CLI 可拿到的全量字段：主实体、关联表、历史记录、标签、统计、状态、时间、备注、关系链。
  • 再按用户场景思考：用户打开这个页面时，是要看、选、做、买、复盘、决策，还是继续写入？
  • 必须提出超出用户原始请求的功能建议，但不能违背 Skill 边界。
  • 建议要分级：立刻做、可选增强、未来版本；不要一次把页面堆满。
  • 每个建议都要说明使用哪类数据、改善什么体验、是否需要新增 CLI 字段。

头脑风暴问题清单

问题可能产出的 HTML 功能
哪些字段用户最常看？首屏关键指标、摘要卡、状态徽章。
哪些字段用户会对比？对比表、排序、筛选、分组视图。
哪些动作用户会重复做？一键复制、勾选记忆、快捷按钮、批量操作提示。
哪些数据能形成提醒？过期提示、超时提示、缺字段提示、异常高亮。
哪些数据能帮助决策？推荐排序、风险提示、历史评分、成本/时间估算。
哪些内容适合折叠？背景知识、历史记录、调试信息、低频字段。
用户完成后要回写什么？复制回执、反馈表单、评分、复盘问题。

建议输出格式

基于 CLI 全量数据，我建议这个 HTML 增加：
1. 首屏摘要卡：用 A/B/C 字段，让用户 3 秒知道当前状态。
2. 分组/筛选：用 tags/status/category 字段，减少扫读成本。
3. 一键复制回执：用 id/date/feedback 字段，方便回写到 CLI。
4. 异常提醒：用 due_at/count/required 字段，提前暴露风险。

本次先做：1、2。
暂不做：3、4，因为需要新增 CLI 字段或会让页面过重。

边界

  • 可以主动建议视觉和交互，不可以擅自新增数据库写入路径。
  • 可以基于已有 JSON 派生展示字段，不可以伪造数据。
  • 可以提出“需要 CLI 增加字段”，但必须标为未来增强。
  • 页面功能增强必须服务用户任务，不做炫技动画。

14. 最小可复制骨架

模板

<!doctype html>
<html lang="zh-CN">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width,initial-scale=1">
  <title>Skill View</title>
</head>
<body>
  <main id="app"></main>
  <script>
    const PAYLOAD = window.__DATA__;
    const app = document.getElementById('app');
    if (!PAYLOAD) {
      app.innerHTML = '<h1>数据未注入</h1>';
    } else {
      const data = PAYLOAD.data || PAYLOAD;
      app.innerHTML = `<h1>${escapeHTML(data.name || '未命名')}</h1>`;
    }
    function escapeHTML(s){return String(s ?? '').replace(/[&<>"']/g,c=>({'&':'&amp;','<':'&lt;','>':'&gt;','"':'&quot;',"'":'&#39;'}[c]));}
  </script>
</body>
</html>

注入器伪代码

data = run_cli_json([...])
template = read_text('templates/view.html')
payload = json.dumps(data, ensure_ascii=False).replace('</', '<\\/')
injected = template.replace('<body>', '<body>\n<script>window.__DATA__ = ' + payload + ';</script>', 1)
write_text('output/view.html', injected)
