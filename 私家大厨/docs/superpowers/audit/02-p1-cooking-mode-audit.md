# P1做菜模式走查报告

> 审计日期：2026-05-23
> 审计范围：`features/view.md` 功能二（做菜模式），行110-335
> 测试路径：用户输入"开始做宫保虾球"

## 流程推演

### 测试路径1：直接进入做菜模式（无前置查看）

用户输入："开始做宫保虾球"

**推演流程**：

1. **路由匹配**（SKILL.md）：触发词"开始做"命中P1，提取菜名"宫保虾球"，路由到 `view.md` → 功能二：做菜模式。路由本身正确。

2. **加载view.md**：AI读取view.md，看到工作流指示：
   ```
   用户说"做菜模式"或"开始做XX"
       → 【复用】复用查看食谱已查询的数据
       → 【设计】调用 Taste Skill + UI/UX Pro Max
       → 【生成】生成 HTML
       → 【发送】保存到媒体目录，通过 QQBot 发送文件
   ```

3. **数据获取（故障点）**：工作流写的是"复用查看食谱已查询的数据"，但用户直接说"开始做宫保虾球"，从未触发过P2查看食谱流程，**没有任何已查询的数据可复用**。AI面临困境：
   - view.md第28-29行说"直接进入做菜模式（复用已查数据，不再重复查询）"
   - 但没有已查数据，AI该怎么做？文档未说明。
   - 命令参考（第282-323行）列出了所有查询命令，但工作流没有指示AI在此场景下执行查询。

4. **外部技能加载（故障点）**：AI需要加载Taste Skill和UI/UX Pro Max，但路径指向不存在的位置（详见故障F3）。

5. **HTML生成（故障点）**：即使数据获取成功，等待面板设计规范存在重复文本（详见故障F4）。

6. **文件存储（故障点）**：存储路径为Linux路径，与实际Windows环境不符（详见故障F5）。

**结论**：直接进入做菜模式的路径在数据获取环节断裂，AI无法确定是否应该自行查询数据。

## 发现的故障

### [P0] F1：直接进入做菜模式时数据来源缺失

- **位置**：`features/view.md` 第76-85行（工作流）、第28-29行（内部分流逻辑）
- **现象**：AI执行者在用户直接输入"开始做宫保虾球"时，工作流指示"复用查看食谱已查询的数据"，但此时没有任何已查询的数据。文档既没有说明AI应该自行执行查询命令，也没有提供降级方案。AI可能：(a) 跳过数据查询，生成空HTML；(b) 错误地认为不需要数据；(c) 自行决定查询但违反"不再重复查询"的指示。
- **复现路径**：用户输入"开始做宫保虾球" → 期望：AI查询数据并生成做菜模式HTML → 实际：AI困惑于"复用已查数据"但无数据可复用
- **修复建议**：在view.md工作流中增加分支逻辑：
  ```
  用户说"做菜模式"或"开始做XX"
      ↓
  【判断】是否有已查数据？
      → 有：复用已查数据
      → 无：按命令参考执行查询，获取全部数据
      ↓
  【设计】调用 Taste Skill + UI/UX Pro Max
      ↓
  【生成】生成 HTML
  ```

### [P1] F2：命令参考未明确标注做菜模式所需的最小数据集

- **位置**：`features/view.md` 第282-323行（命令参考）
- **现象**：命令参考列出了所有查询命令，但没有标注做菜模式具体需要哪些数据。做菜模式需要：菜名、总时间、份量、食材清单、步骤列表（含duration/heat_level）、步骤×食材关联、技法、小贴士。但文档没有明确说"做菜模式至少需要执行以下命令"，AI可能遗漏关键查询（如step_ingredient或technique）。
- **复现路径**：用户输入"开始做宫保虾球" → AI执行部分查询 → 生成的HTML缺少步骤关联食材或技法信息
- **修复建议**：在做菜模式工作流下方增加"做菜模式最小数据集"清单：
  ```
  做菜模式必须查询：
  1. python scripts/recipe_manager.py show <菜名>  # 基础信息
  2. python scripts/ingredient_manager.py list <recipe_id>  # 食材清单
  3. python scripts/step_manager.py list <recipe_id>  # 步骤列表
  4. python scripts/step_ingredient_manager.py list-by-step <step_id>  # 每步食材
  5. python scripts/technique_manager.py list-by-step <step_id>  # 每步技法
  6. python scripts/tip_manager.py list <recipe_id>  # 小贴士
  ```

### [P1] F3：外部技能路径全部错误，AI无法加载

- **位置**：`features/view.md` 第43-49行（技能引用）、第332-335行（参考路径）
- **现象**：view.md中存在两套路径，且两套都是错误的：

  **技能引用区（第43-49行）**：
  - Taste Skill: `~/.openclaw/skills/taste-skill/SKILL.md` -- 不存在
  - UI/UX Pro Max: `~/.openclaw/skills/ui-ux-pro-max/SKILL.md` -- 不存在

  **参考区（第332-335行）**：
  - Taste Skill: `/mnt/d/2Study/StudyNotes/SKILLS/taste-skill/skills/taste-skill/SKILL.md` -- WSL路径格式
  - UI/UX Pro Max: `/mnt/d/2Study/StudyNotes/SKILLS/ui-ux-pro-max-skill/.claude/skills/ui-ux-pro-max/SKILL.md` -- WSL路径格式

  **实际文件位置**（Windows环境）：
  - Taste Skill: `D:/2Study/StudyNotes/SKILLS/taste-skill/skills/taste-skill/SKILL.md` -- 存在
  - UI/UX Pro Max: `D:/2Study/StudyNotes/SKILLS/ui-ux-pro-max-skill/.claude/skills/ui-ux-pro-max/SKILL.md` -- 存在

  AI执行者在Windows环境下使用这些路径，`~/.openclaw/...` 路径无法解析，`/mnt/d/...` 是WSL路径格式在Windows bash中可能不被识别。
- **复现路径**：AI尝试加载外部技能 → read工具报错"文件不存在" → 无法获取设计规范 → HTML生成违反强制要求
- **修复建议**：统一使用Windows路径，并在技能引用区和参考区保持一致：
  ```

  - 目录：`D:/2Study/StudyNotes/SKILLS/taste-skill/skills/taste-skill/SKILL.md`

  ### 2. UI/UX Pro Max
  - 目录：`D:/2Study/StudyNotes/SKILLS/ui-ux-pro-max-skill/.claude/skills/ui-ux-pro-max/SKILL.md`
  ```
  参考区同步更新。

### [P3] F6：计时器恢复算法缺少暂停场景的处理说明

- **位置**：`features/view.md` 第236-241行（计时器恢复算法）
- **现象**：计时器恢复算法只给出了一个公式：
  ```
  remaining = duration - (currentTimestamp - startTimestamp)
  ```
  但Bug1约束（第163-177行）提到了暂停状态（paused）的处理：`paused 状态保存 remaining 值，重开时用 startTime = Date.now() - (duration - remaining)`。恢复算法部分没有体现暂停场景的公式，AI可能在实现时忽略暂停状态的特殊处理。
- **复现路径**：用户暂停计时器 → 刷新页面 → AI使用基础公式计算remaining → 暂停时间被计入已用时间 → 倒计时错误
- **修复建议**：在计时器恢复算法中增加暂停场景：
  ```
  刷新页面后，按以下公式计算剩余时间：
  - 正常状态：remaining = duration - (currentTimestamp - startTimestamp)
  - 暂停状态：remaining = 保存的remaining值（不重新计算）

  若 remaining <= 0，该步骤到期，触发高亮提醒。
  ```

### [P3] F7：到期提醒规范过于模糊

- **位置**：`features/view.md` 第259-261行
- **现象**：到期提醒规范只写了"计时器到期时，页面出现提示信息告知用户，具体样式由AI根据上下文自行决定"。但核心约束第6条（第143行）要求"左上角出现提示条"。两处描述不一致——一个说"左上角提示条"，一个说"AI自行决定"。AI可能生成不符合预期的提醒样式。
- **复现路径**：计时器到期 → AI参考到期提醒规范 → 生成浮动弹窗而非左上角提示条 → 与核心约束不一致
- **修复建议**：将到期提醒规范与核心约束对齐：
  ```
  计时器到期时：
  - 左上角出现提示条（与核心约束#6一致）
  - 顶部计时器区域对应计时器高亮
  - 用户点击提示条或确认按钮后消失
  ```

---

### [P3] F8：依赖检查算法未覆盖等待型步骤的特殊逻辑

- **位置**：`features/view.md` 第225-230行
- **现象**：依赖检查算法 `canUnlock(step N)` 只检查了"上一步是否完成"，但核心约束第1条（第137行）说"等待型步骤只启动计时器，不阻止后续步骤解锁"。算法没有体现这个特殊逻辑——如果step N-1是等待型，即使计时器未到期，step N也应该解锁。当前算法依赖"step N-1 is completed"的定义，但没有说明等待型步骤的"完成"是指"启动计时器"还是"计时器到期"。
- **复现路径**：步骤2是等待型（腌制10分钟）→ 用户完成步骤2（启动计时器）→ 步骤3是否立即解锁？→ 算法未明确
- **修复建议**：补充依赖检查算法：
  ```
  canUnlock(step N):
      if step N-1 is not completed:
          return false
      // 等待型步骤：启动计时器即视为"完成"，不锁后续
      // 普通型步骤：用户点击"完成步骤"即视为"完成"
      return true
  ```
