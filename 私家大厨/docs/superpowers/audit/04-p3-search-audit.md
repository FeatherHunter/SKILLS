# P3 搜索筛选 走查报告

> 审计日期：2026-05-23
> 审计方法：AI执行者视角，模拟3条搜索路径的完整流程
> 审计文件：`features/search.md`、`SKILL.md`（路由表）、`references/commands.md`

---

## 搜索功能概览

search.md 定义了6种搜索方式：

| # | 搜索方式 | CLI命令 | 适用场景 |
|---|---------|---------|---------|
| 1 | 按菜名搜索 | `python scripts/recipe_manager.py search <关键词>` | "搜一下排骨" |
| 2 | 按分类筛选 | `python scripts/category_manager.py search <菜系>` 等 | "川菜有哪些" |
| 3 | 按难度筛选 | `python scripts/recipe_manager.py list --difficulty <难度>` | "来个简单的" |
| 4 | 按状态筛选 | `python scripts/recipe_manager.py list --status <状态>` | "找已做的菜" |
| 5 | 按食材搜索 | `python scripts/ingredient_manager.py search <食材名>` | "哪些菜里有虾" |
| 6 | 组合搜索 | 多条命令 + 结果过滤 | "找个辣的川菜，30分钟能搞定的" |

**搜索结果格式**（标准）：序号|菜名|难度|总时间|状态|评分 表格 + 后续引导
**空结果格式**：提示无结果 + 3条建议（放宽条件/换说法/直接说菜名）

## 路径1：用户输入"找川菜"

### 路由匹配

- 用户输入：`找川菜`
- P3触发词命中：`找`
- 菜名提取：触发词后面的文字 = `川菜`（2字符，有效）
- 路由结果：`search.md`，搜索词 = `川菜`

### 执行流程

AI执行者读取search.md，匹配到"按分类筛选"场景：

```
用户：川菜有哪些
→ python scripts/category_manager.py search 川菜
```

实际执行命令：`python scripts/category_manager.py search 川菜`

### 发现的故障

#### [P2-1] 分类搜索的展示字段未定义

- **位置**：`features/search.md` 第46-54行 vs 第126-139行
- **现象**：search.md 按分类筛选的示例（第46-54行）只展示了命令，没有定义返回结果的展示格式。而标准结果格式（第126-139行）要求展示 `序号|菜名|难度|总时间|状态|评分`，但 `category_manager.py search` 返回的数据是否包含这些字段（难度、总时间、状态、评分）未说明。
- **复现路径**："找川菜" → 执行 `category_manager.py search 川菜` → 返回结果 → AI不知道该用什么格式展示
- **影响范围**：所有按分类筛选的搜索（菜系、季节、烹饪方式、口味、用餐类型）
- **修复建议**：在search.md的"按分类筛选"部分补充标准输出格式示例，或说明分类搜索返回的字段列表。

#### [P2-2] 不同搜索方式的示例格式不统一

- **位置**：`features/search.md` 第33-44行 vs 第126-139行
- **现象**：按菜名搜索的示例使用叙述式展示（"1. 酱大骨（川菜，中等难度，45分钟）"），而标准结果格式使用表格展示（`序号|菜名|难度|总时间|状态|评分`）。两种格式不一致，AI执行者不知道该用哪种。
- **复现路径**：AI读取search.md → 看到两种格式 → 不确定该用哪种
- **影响范围**：所有搜索方式的展示
- **修复建议**：统一为一种格式。建议保留标准表格格式，将按菜名搜索的示例也改为表格形式。

## 路径2：用户输入"哪些菜里有虾"

### 路由匹配

- 用户输入：`哪些菜里有虾`
- P3触发词命中：`哪些菜`
- 菜名提取：触发词后面的文字 = `里有虾`（3字符）
- 问题：按规则R4，`里有虾` 会被提取为菜名（3字符 >= 2字符），但实际并不是菜名
- SKILL.md 第186行的示例："哪些菜里有虾" → 触发词"哪些菜" → 无（菜名<2字符）

**矛盾**：SKILL.md说提取结果是"无（菜名<2字符）"，但"里有虾"是3字符，按规则应 >= 2字符。这里SKILL.md的示例与规则矛盾。

### 执行流程

即使路由匹配成功，AI执行者面临关键问题：

1. 路由将"哪些菜里有虾"分流到search.md
2. search.md的"按食材搜索"示例展示：`python scripts/ingredient_manager.py search 虾`
3. 但AI如何从"哪些菜里有虾"中提取出搜索词"虾"？

### 发现的故障

#### [P1-1] 食材搜索的搜索词提取逻辑缺失

- **位置**：`features/search.md` 第88-101行 + `SKILL.md` 路由规则
- **现象**：路由规则只定义了"菜名提取"（触发词后面的文字），但食材搜索需要的是"食材名提取"。对于"哪些菜里有虾"，触发词"哪些菜"后面是"里有虾"，AI需要进一步从"里有虾"中提取出"虾"。这个二次提取逻辑在文档中完全没有定义。
- **复现路径**："哪些菜里有虾" → P3命中"哪些菜" → 菜名提取得"里有虾" → AI需要执行 `ingredient_manager.py search` → 搜索词应该是"虾"而不是"里有虾" → AI如何知道要提取"虾"？
- **影响范围**：所有"哪些菜里有XX"、"什么菜用到XX"句式的搜索
- **修复建议**：方案A：在search.md中补充食材搜索词的提取规则（去掉"里有"、"用到"等连接词）。方案B：在路由表中为"哪些菜"触发词定义搜索词提取规则（不同于菜名提取规则）。

#### [P1-2] SKILL.md路由示例与规则矛盾

- **位置**：`SKILL.md` 第186行 vs 第43行（R4规则）
- **现象**：路由示例声称"哪些菜里有虾"的菜名提取结果是"无（菜名<2字符）"，但触发词"哪些菜"后面的文字是"里有虾"（3字符），按R4规则应 >= 2字符，不应被过滤为"无"。
- **复现路径**：AI按R4规则提取 → "里有虾"（3字符）→ 应视为有效菜名 → 但示例说是"无"
- **影响范围**：AI执行者对路由规则的理解
- **修复建议**：修正示例，明确"哪些菜"类触发词的菜名提取逻辑。建议：对于"哪些菜"、"什么菜"等搜索类触发词，菜名提取后还需要二次处理提取搜索词。

## 路径3：用户输入"找个辣的川菜，30分钟能搞定的"

### 路由匹配

- 用户输入：`找个辣的川菜，30分钟能搞定的`
- P3触发词命中：`找`
- 菜名提取：触发词后面的文字 = `个辣的川菜，30分钟能搞定的`（不是有效菜名）
- 路由结果：`search.md`，走组合搜索

### 执行流程

AI读取search.md的"组合搜索"部分（第103-122行）：

```
筛选条件：
    - 菜系：川菜
    - 口味：辣
    - 总时间 ≤ 30分钟

执行步骤：
    1. python scripts/category_manager.py search 川菜
    2. 在结果中过滤时间和口味
```

### 发现的故障

#### [P1-3] 组合搜索缺少"过滤"步骤的实现方式

- **位置**：`features/search.md` 第112-115行
- **现象**：组合搜索的第2步是"在结果中过滤时间和口味"，但 `category_manager.py search` 的返回结果中是否包含时间和口味字段未说明。即使包含，AI如何执行"过滤"？是用CLI命令过滤，还是AI自行在结果中筛选？
- **复现路径**："找个辣的川菜，30分钟能搞定的" → 执行 `category_manager.py search 川菜` → 拿到川菜列表 → 需要过滤时间和口味 → `category_manager.py search` 返回的结果有时间和口味字段吗？→ 如果没有，AI需要对每道菜执行额外查询 → 流程不清晰
- **影响范围**：所有组合搜索场景
- **修复建议**：方案A：为 `category_manager.py search` 补充 `--max-time` 和 `--flavor` 过滤参数。方案B：在search.md中明确组合搜索的完整步骤，包括如何获取时间和口味数据。

#### [P2-3] 组合搜索的条件解析规则未定义

- **位置**：`features/search.md` 第103-122行
- **现象**：search.md只展示了一个组合搜索的示例，但没有定义AI如何从自然语言中解析出多个筛选条件。例如：
  - "找个辣的川菜，30分钟能搞定的" → 菜系=川菜，口味=辣，时间<=30
  - "来个简单的家常菜" → 难度=简单，菜系=家常？
  - "有什么快手的海鲜" → 难度=快手菜，食材分类=海鲜
- **复现路径**：用户各种组合输入 → AI需要自行判断哪些词映射到哪些筛选条件 → 无明确规则
- **影响范围**：组合搜索的准确率
- **修复建议**：补充组合搜索的条件解析规则，定义常见修饰词与筛选字段的映射关系。

#### [P2-4] 组合搜索可能需要多次CLI调用

- **位置**：`features/search.md` 第112-115行
- **现象**：如果 `category_manager.py search` 不返回时间和口味数据，AI需要对每道菜执行额外查询来获取这些字段。假设川菜有10道，AI需要执行10次以上的额外查询，效率极低。
- **复现路径**：执行 `category_manager.py search 川菜` → 返回10道菜 → 对每道菜执行 `recipe_manager.py show` 获取时间 → 对每道菜执行 `flavor_manager.py list` 获取口味 → 过滤 → 效率问题
- **影响范围**：组合搜索的性能和用户体验
- **修复建议**：在manager的search命令中支持返回更多字段，或提供专用的组合搜索CLI命令。

## 搜索结果后续引导审计

### 引导语分析

search.md 第138行定义的后续引导：
```
想看哪道的做法？说"看看第X道"或直接说菜名。
```

### 发现的故障

#### [P2-5] "看看第X道"的路由处理需要上下文记忆

- **位置**：`features/search.md` 第138行 + `SKILL.md` P2路由
- **现象**：用户搜索后说"看看第1道"，"看看"是P2触发词，会路由到 `view.md`。但"第1道"不是菜名，AI需要从上一次搜索结果中查找第1道菜的菜名。这个"序号→菜名"的转换逻辑在文档中没有定义。
- **复现路径**："找川菜" → 返回结果列表 → "看看第1道" → P2命中"看看" → 菜名提取得"第1道" → "第1道"不是有效菜名（或者会被当作菜名去查询）→ 查询失败
- **影响范围**：所有搜索结果的后续查看操作
- **修复建议**：在search.md中补充说明：当用户说"看看第X道"时，AI应从最近一次搜索结果中查找对应序号的菜名，然后用该菜名执行查看操作。或在SKILL.md中为"看看第X道"定义特殊处理规则。

#### [P3-1] 直接说菜名的兜底路由可能冲突

- **位置**：`SKILL.md` 第49行（R5兜底规则）
- **现象**：引导语说"或直接说菜名"，例如用户搜索川菜后说"宫保虾球"。按R5兜底规则，无触发词命中时走search.md。但用户的实际意图是"查看宫保虾球"（P2），而非"搜索宫保虾球"（P3兜底）。
- **复现路径**："找川菜" → 返回结果 → "宫保虾球" → 无触发词命中 → 走兜底search.md → 执行 `recipe_manager.py search 宫保虾球` → 而非直接展示食谱详情
- **影响范围**：搜索后的查看操作体验
- **修复建议**：在search.md中说明，当用户在搜索结果上下文中直接说菜名时，应视为P2查看食谱意图，而非P3兜底搜索。或在引导语中改为"说'看看宫保虾球'"而非"直接说菜名"。
