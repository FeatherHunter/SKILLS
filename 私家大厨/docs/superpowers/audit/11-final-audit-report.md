## 执行摘要

- **总发现：60个故障**（去重后）
- **P0-阻塞：8个** — 核心功能完全不可用或流程断裂
- **P1-严重：11个** — 功能可用但存在重大缺陷
- **P2-中等：28个** — 影响输出质量或用户体验
- **P3-轻微：13个** — 文档细节或规范不一致

**核心结论**：系统存在3个结构性问题影响全局：
1. **菜名提取规则（R4）存在根本性缺陷**，影响路由层及所有下游功能
2. **路径格式全部不可达**（外部技能路径 + 文件存储路径），阻断HTML生成和发送
3. **多处枚举值不一致**，跨文件、跨模块的数据定义存在矛盾

## P0-阻塞（必须修复）

### [P0-1] R4菜名提取规则与实际行为存在根本性矛盾

- **位置**：SKILL.md:41-42, 59
- **现象**：R4规则描述"触发词后面紧跟的文字视为菜名"，但多个示例中菜名实际取自触发词**前面**的文字。例如"宫保虾球菜谱"中触发词"菜谱"在末尾，菜名"宫保虾球"在前面；"宫保虾球做过几次了"中触发词"做过"在中间，菜名在前面。
- **影响范围**：所有触发词不在输入开头位置的场景，影响P2/P5/P6共3个优先级
- **发现来源**：01-routing-audit（故障#1）
- **修复建议**：重写R4规则，区分触发词在开头/末尾/中间的提取方向：开头取后面文字，末尾取前面文字，中间优先取前面文字。

### [P0-2] 触发词在中间时菜名提取方向系统性错误

- **位置**：SKILL.md:41-42（匹配流程第2步）
- **现象**：当触发词出现在输入中间时，R4规则一律取后面文字，导致提取到补充说明而非菜名。例如"把宫保虾球第2步改成小火"提取"小火"而非"宫保虾球"；"宫保虾球做过几次了"提取"几次了"而非"宫保虾球"。
- **影响范围**：P4修改食谱、P5烹饪历史的全部操作
- **发现来源**：01-routing-audit（故障#2）、05-p4-update-audit、06-p5-history-audit
- **修复建议**：在匹配流程第2步增加方向判断逻辑：先检查触发词前面是否有≥2字符的文字，如有则优先作为菜名。

### [P0-3] R4规则的≥2字符门槛无法过滤语义无效的菜名

- **位置**：SKILL.md:42
- **现象**：R4规则仅以字符数（≥2）判断菜名有效性，无法区分具体菜名和泛指短语。"哪些菜里有虾"提取"里有虾"（3字符但非菜名），"录入一个新菜"提取"一个新菜"（4字符但非菜名）。
- **影响范围**：P3搜索筛选（哪些菜/什么菜/哪个）、P7录入食谱（录入/新建/添加）
- **发现来源**：01-routing-audit（故障#3）、04-p3-search-audit
- **修复建议**：增加R7规则——触发词后面紧跟"一个/一道/个/些/里有/用到"等量词/介词时，跳过这些词再提取菜名；或为P3/P7设置特殊提取逻辑。

### [P0-4] 外部技能路径全部不可达，AI无法加载设计规范

- **位置**：view.md:43-49（技能引用）、view.md:332-335（参考路径）
- **现象**：Taste Skill和UI/UX Pro Max的4个引用路径全部无法访问。`~/.openclaw/skills/...`在Windows下不存在，`/mnt/d/...`是WSL路径格式在Git Bash中不可用。view.md声明"违反强制要求，输出视为不合格"，但技能文件根本无法加载。
- **影响范围**：查看食谱和做菜模式的全部HTML输出
- **发现来源**：02-p1-cooking-mode-audit（F3）、03-p2-view-recipe-audit（F03/F04）、09-dependency-audit
- **修复建议**：统一使用Windows路径：Taste Skill → `D:/2Study/StudyNotes/SKILLS/taste-skill/skills/taste-skill/SKILL.md`；UI/UX Pro Max → `D:/2Study/StudyNotes/SKILLS/ui-ux-pro-max-skill/.claude/skills/ui-ux-pro-max/SKILL.md`。

### [P0-5] 直接进入做菜模式时数据来源缺失

- **位置**：view.md:76-85（工作流）、view.md:28-29（内部分流逻辑）
- **现象**：用户直接输入"开始做宫保虾球"时，工作流指示"复用查看食谱已查询的数据"，但此时没有任何已查询的数据。文档既未说明AI应自行执行查询命令，也未提供降级方案。AI面临三难：跳过数据查询生成空HTML、错误认为不需要数据、或自行查询但违反"不再重复查询"的指示。
- **影响范围**：做菜模式核心流程
- **发现来源**：02-p1-cooking-mode-audit（F1）
- **修复建议**：在view.md工作流中增加分支逻辑——判断是否有已查数据，有则复用，无则按命令参考执行查询获取全部数据。

### [P0-6] history_manager.py不支持菜名查询

- **位置**：scripts/history_manager.py:25, 75
- **现象**：`history_manager.py list`和`stats`命令只接受recipe_id（UUID），不支持菜名。但history.md示例写的是`history_manager.py list 宫保虾球`，与代码实现直接矛盾。AI执行者按文档示例操作会直接失败。
- **影响范围**：P5烹饪历史全部操作（add/list/stats/update）
- **发现来源**：06-p5-history-audit、10-cross-reference-audit
- **修复建议**：方案A（推荐）：在history_manager.py的list和stats中增加名称查询支持，参照recipe_manager.py show的`WHERE id = ? OR name LIKE ?`逻辑。方案B：在history.md中明确说明AI必须先用recipe_manager.py show获取ID。

### [P0-7] 文件存储路径不可达，阻断QQBot文件发送

- **位置**：view.md:106, 267, 269；shopping.md:135
- **现象**：`/home/feather/.openclaw/media/qqbot/`和`/home/feather/.openclaw/media/qqbot/shopping/`在Windows环境下不存在。这两个路径是HTML文件的唯一存储位置，路径不可达 = HTML无法保存 = QQBot无法发送文件给用户。
- **影响范围**：查看食谱、做菜模式、采购清单三个核心功能的输出环节
- **发现来源**：02-p1-cooking-mode-audit（F5）、07-p6-shopping-audit、09-dependency-audit
- **修复建议**：将路径改为Windows兼容格式，如`D:/2Study/StudyNotes/SKILLS/私家大厨/output/qqbot/`，或使用环境变量统一管理。

### [P0-8] "不想要"不在P4触发词列表中，废弃食谱功能无法触发

- **位置**：SKILL.md:117-129 vs update.md:147
- **现象**：update.md示例"用户：不想要宫保虾球了"→ 废弃食谱，但SKILL.md P4触发词列表中没有"不想要"。用户说"不想要宫保虾球了"不会触发P4，会走兜底search.md，废弃意图完全丢失。
- **影响范围**：废弃食谱功能完全不可用
- **发现来源**：05-p4-update-audit（P0-1）
- **修复建议**：在P4触发词列表中增加：`不想要`、`废弃`、`删掉`。

## P1-严重（应该修复）

### [P1-1] 命令参考未标注做菜模式所需的最小数据集

### [P1-2] 查询命令清单缺少执行顺序和依赖说明

- **位置**：view.md:284-323
- **现象**：18条查询命令未说明执行顺序。step_ingredient_manager.list-by-ingredient依赖ingredient_manager.list返回的ingredient_id，technique_manager.list-by-step依赖step_manager.list返回的step_id。AI按文档顺序从上到下执行会在没有ID的情况下调用依赖命令。
- **影响范围**：查看食谱和做菜模式共用此命令清单，两个功能均受影响

### [P1-3] 修改食谱的前置查询流程未在update.md中说明

- **位置**：update.md:69-87（修改步骤示例）
- **现象**：示例使用`<step_id>`作为占位符，但未说明AI如何从用户输入的"第2步"（sequence number）获取step_id。同样，所有命令都需要recipe_id，但update.md未说明如何从菜名获取recipe_id。
- **影响范围**：所有涉及实体ID的修改操作
- **发现来源**：05-p4-update-audit（P1-1, P1-2）
- **修复建议**：在update.md开头增加通用前置步骤说明：recipe_manager.py show获取recipe_id → step_manager.py list获取step_id。

### [P1-4] 多步操作中中间ID获取流程不明确

- **位置**：update.md:89-109（添加食材含步骤关联）
- **现象**：示例标注"假设返回 ingredient_id = xxx"，但未说明AI如何从CLI输出中解析ingredient_id、如果目标步骤不存在怎么办、添加失败是否需要回滚。
- **影响范围**：所有需要中间ID的多步操作
- **发现来源**：05-p4-update-audit（P1-3）
- **修复建议**：说明CLI命令的输出格式、增加错误处理说明、增加步骤不存在时的处理逻辑。

### [P1-5] 采购清单工作流缺少"菜名→ID"查询步骤

- **位置**：shopping.md 工作流 + 命令参考
- **现象**：用户输入是菜名（如"宫保虾球"），但shopping_manager.py generate只接受recipe_id（UUID）。shopping.md的工作流缺少关键步骤：AI必须先通过recipe_manager.py show获取recipe_id。
- **影响范围**：采购清单功能的核心流程
- **发现来源**：07-p6-shopping-audit（BUG-01）
- **修复建议**：工作流改为：用户请求 → AI用菜名查recipe_id → AI调用脚本获取JSON → AI生成HTML → 发送HTML给用户。

### [P1-6] 同名食材合并逻辑未定义

- **位置**：shopping.md HTML生成要求第11项
- **现象**：文档要求"多食谱同名食材合并显示总用量+来源"，但未定义合并规则：单位不同时如何合并？quantity为null时如何处理？合并后is_optional如何处理？
- **影响范围**：多食谱采购清单的正确性
- **发现来源**：07-p6-shopping-audit（BUG-05）
- **修复建议**：在shopping.md中添加合并规则：按name匹配、相同单位直接相加、不同单位用"+"连接、is_optional取必需优先。

### [P1-7] 食材搜索的搜索词提取逻辑缺失

- **位置**：search.md:88-101 + SKILL.md路由规则
- **现象**：路由规则只定义了"菜名提取"，但食材搜索需要"食材名提取"。对于"哪些菜里有虾"，触发词"哪些菜"后面是"里有虾"，AI需要进一步从"里有虾"中提取出"虾"。这个二次提取逻辑完全缺失。
- **影响范围**：所有"哪些菜里有XX"、"什么菜用到XX"句式的搜索
- **发现来源**：04-p3-search-audit（P1-1）
- **修复建议**：在search.md中补充食材搜索词提取规则，去掉"里有"、"用到"等连接词后提取食材名。

### [P1-8] 组合搜索缺少"过滤"步骤的实现方式

- **现象**：组合搜索的第2步是"在结果中过滤时间和口味"，但category_manager.py search的返回结果中是否包含时间和口味字段未说明，AI如何执行"过滤"也未定义。

- **现象**：文档要求输出"最高评分"和"最低评分"（含日期），代码不输出。文档"总次数"，代码"做过次数"。文档avg_rating显示2位小数（4.23），代码显示1位小数（4.1）。

## 根因分析

经过对60个故障的交叉比对，识别出以下**5个根本原因**：

| 根因 | 影响的故障数 | 说明 |
|------|-------------|------|
| R4菜名提取规则设计缺陷 | 8个（P0-1/2/3, P2-1/2, P1-7, P3-7, P0-6间接） | 规则仅覆盖触发词在开头的场景，未考虑末尾和中间位置 |
| 路径格式未适配Windows环境 | 6个（P0-4/7, P2-4/9/26, P1-1间接） | 文档编写时使用Linux/WSL路径，未适配实际运行环境 |
| 枚举值定义分散且不同步 | 8个（P1-10/11, P2-22/23/24/25/28, P3-11） | 同一字段的枚举值在多个文件中维护，缺乏单一权威源 |
| 前置查询流程未文档化 | 5个（P0-5, P1-2/3/4/5） | 工作流假设AI已知实体ID，未说明从菜名/序号到ID的转换步骤 |
| 缺少错误处理和降级方案 | 4个（P2-8/9, P1-6, P3-5） | 文档只描述正常路径，未覆盖异常情况 |

**建议**：修复时应从根因入手，而非逐个处理故障。修复R4规则将一次性解决8个故障，统一路径格式将解决6个故障。
